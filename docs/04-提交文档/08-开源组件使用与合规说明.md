## 合规结论与使用边界

AXIOM Space 的产品定义、业务流程、多智能体协作、学习画像、卡片语义、评估闭环、知识图谱交互及参赛内容由团队自行设计和实现；项目同时复用了 Web 框架、UI 基础件、数据库 ORM、任务队列、图谱渲染、文档渲染、测试工具和 RAG 基础设施等第三方成果。**使用开源组件不等于把第三方项目冒充为自研，也不等于项目自身已经开源。**

本次审计结论：

| 核验项 | 当前事实 | 结论 |
|---|---|---|
| NPM 直接运行依赖 | 60 个；52 MIT、3 Apache-2.0、2 BSD-3-Clause、1 BSD-2-Clause、1 ISC、1 GSAP 专用许可 | 主体为宽松许可证，但 GSAP 不能写成“MIT 开源组件” |
| NPM 直接开发依赖 | 20 个；16 MIT、3 Apache-2.0、1 BSD-3-Clause | 可用于开发测试；发布包仍需按实际打包内容核验 |
| 完整依赖树 | 本机扫描到 MIT、Apache-2.0、BSD、ISC、MPL-2.0、CC-BY-4.0 等；1 个包元数据标为 Unknown | `khroma@2.1.0` 随包 `license` 文件实际为 MIT，仍应在自动清单中修正记录 |
| 容器服务 | PostgreSQL、Redis、Qdrant、Ollama、LightRAG | 版本、许可证分界与固定策略已逐项登记 |
| 模型权重 | 默认本地嵌入模型为 `bge-m3:latest` | 模型许可证独立于 Ollama 软件，需单独保留模型卡与版本 |
| 项目自身许可证 | 根目录未声明公共开源许可证 | 自研代码版权由团队保留；参赛提交不自动授予公众复制、修改与再分发权 |
| 第三方集中披露 | 本说明书提供直接依赖、容器、模型、用途与许可证记录 | 参赛阶段集中披露已形成；公开发行包按实际产物生成随包声明 |

## 1. 文档目的与职责边界

本文档只回答一个问题：**AXIOM Space 复用了哪些第三方软件、模型和基础设施，分别承担什么作用，团队应履行哪些许可证与披露责任。**

本文档负责：

- 列出直接运行依赖和开发依赖的实际安装版本；
- 说明各类开源组件在项目中的用途，而不是只复制包名；
- 区分开源许可证、专用免费许可、模型许可和云 API 服务条款；
- 标记浮动版本、未知许可证、声明文件缺失等发布风险；
- 定义从引入依赖到最终分发的合规检查流程；
- 向评审呈现团队在哪些基础设施上进行了二次设计和系统集成。

本文档不负责：

- 不解释 AI 工具怎样参与开发，相关内容见《09-AI 工具使用与 AI 辅助开发流程说明》；
- 不解释系统架构和业务实现，相关内容见《03-系统设计与开发说明书》；
- 不把 API 付费服务称为开源组件；
- 不对许可证作超出官方文本的法律承诺；
- 不替团队自动决定 AXIOM Space 自研代码最终采用何种许可证。

## 2. 审计范围、方法与证据

### 2.1 审计范围

本次清单覆盖：

1. `package.json` 中 60 个 `dependencies`；
2. `package.json` 中 20 个 `devDependencies`；
3. `pnpm-lock.yaml` 锁定的间接依赖树；
4. `docker-compose.yml` 中 5 个容器服务；
5. `.env.example` 和启动脚本中声明的本地嵌入模型；
6. `public/` 中随应用分发的静态资产；
7. 当前仓库是否存在项目 `LICENSE`、第三方声明或上游 `NOTICE` 汇总。

不在本次自动扫描范围内、但发布前必须另行核验：

- 现场填写的商业 AI API 密钥及其服务条款；
- 用户自行上传的课程资料版权；
- AI 生成资源中可能出现的第三方文本、图片、代码或引用；
- Docker 基础镜像中操作系统层的全部软件包；
- 比赛演示视频、PPT、配乐、截图中额外使用的素材；
- 未来新增而尚未进入锁文件的依赖。

### 2.2 审计方法

| 证据 | 用途 | 局限 |
|---|---|---|
| `package.json` | 确定直接依赖及声明版本范围 | 不是最终安装版本 |
| `node_modules/<pkg>/package.json` | 获取当前已安装版本和许可证元数据 | 元数据可能缺失或写错 |
| 随包 `LICENSE` / `license` / `NOTICE` | 复核实际条款和版权声明 | 需逐包保留，不能只记 SPDX 名称 |
| `pnpm licenses list --json` | 扫描完整安装依赖树 | 可能把缺少 `license` 字段的包标为 Unknown |
| `pnpm-lock.yaml` | 锁定可复现版本 | 不替代许可证正文 |
| 官方项目许可证页 | 复核容器与模型 | `latest` 标签仍会漂移 |
| 源码搜索 | 确认组件实际用途 | 动态加载或构建期行为仍需人工复核 |

### 2.3 版本口径

下表同时区分：

- **声明版本：** `package.json` 中团队允许安装的范围；
- **实际版本：** 本次审计机器 `node_modules` 中安装的版本；
- **发行版本：** 最终提交或发布时必须由锁文件、镜像摘要和构建产物再次确认。

本文清单不能永久替代自动扫描；任何依赖更新都可能改变许可证或 NOTICE 义务。

## 3. 自研成果与第三方成果边界

### 3.1 团队自研与集成成果

根据项目文档和源码，团队主要负责：

- 面向大学生的学习需求分析与产品定义；
- “输出—证据审核—改变下一步”的学习闭环；
- 文献卡、灵感卡、永久卡的业务语义；
- 多智能体角色划分、协作协议、工具边界和状态写回；
- 动态学习画像、证据时间线、知识缺口和教学策略；
- 学习路径、掌握评估、补救动作与主动资源建议；
- 知识图谱十种阅读布局及其交互；
- 各第三方框架之间的分层、适配、组合、测试和部署；
- 参赛案例、教学材料、提交文档及演示流程。

这些是项目在通用组件之上形成的领域设计和工程实现。

### 3.2 第三方成果

第三方成果包括：

- 框架、库、CLI、类型声明、浏览器和数据库客户端；
- 容器镜像及其包含的软件；
- 开源 RAG、向量数据库和本地模型运行器；
- 模型权重与模型卡；
- 图标、动画引擎和文档渲染工具。

文档中应使用“基于”“集成”“调用”“适配”“使用”等准确词语，不应把 Next.js、Hono、Prisma、Qdrant、LightRAG、Ollama、GSAP 等项目表述为团队自研。

### 3.3 项目代码权属与发行策略

当前仓库根目录未声明公共开源许可证。这表示团队保留 AXIOM Space 自研代码的版权与授权决定权，不表示项目缺少权属，也不影响按照赛事规则提交评审材料。

- 第三方依赖采用 MIT 等许可证，不会自动决定 AXIOM Space 自研代码的许可证；
- 赛事提交按主办方规则授权评审，不自动等同于向公众授予复制、修改和再分发权；
- 后续如选择公开源码，由项目负责人确定适用许可证并生成对应授权文件；
- 公开发行前，再按实际源码、构建产物、容器、模型和素材完成兼容性复核。

## 9. 主要许可证义务

以下是工程执行摘要；真正分发时以随包许可证原文为准。

### 9.1 MIT、ISC、BSD

通常允许使用、复制、修改和分发，但应：

- 在软件副本或实质部分中保留版权声明和许可文本；
- 保留免责声明；
- BSD-3-Clause 不得在未经许可时用原作者或贡献者名称为衍生产品背书；
- 不因“宽松”而删除来源和版权信息。

### 9.2 Apache License 2.0

分发时重点检查：

- 提供 Apache-2.0 许可证副本；
- 修改过上游文件时作显著说明；
- 保留适用的版权、专利、商标和归属声明；
- 上游若提供 `NOTICE`，在发行物中以符合条款的方式包含相关 NOTICE；
- 不把许可描述为授予上游商标使用权。

### 9.3 MPL-2.0

MPL-2.0 是文件级弱 Copyleft。若只是未修改地使用依赖，一般保留许可证和源码获取信息；若修改了受 MPL 覆盖的文件并分发，需按许可证要求提供这些文件的源代码。最终发布前应确认打包方式和是否修改过上游文件。

### 9.4 CC-BY-4.0

对 `caniuse-lite` 等数据类依赖，需要保留适当署名、许可证链接和修改说明。它不是普通代码 MIT 许可，第三方声明生成器不能忽略数据许可证。

### 9.5 多许可证表达式

- `OR`：通常可以选择一个许可路径，但必须记录选择；
- `AND`：需要同时满足列出的许可条件；
- 不得在没有记录的情况下默认选择“看起来最宽松”的一项。

## 10. 特殊风险与处置决策

### 10.1 GSAP 不是普通开源许可证

当前 `gsap@3.15.0` 的包元数据声明为 Standard “No Charge” License。官方条款允许在网站、Web 应用或数字界面中使用，但禁止在特定条件下用于与 Webflow 视觉动画构建能力竞争的产品，并要求不得移除专有声明或品牌。

AXIOM Space 当前用 GSAP 控制知识图谱节点、相机和布局过渡，不是无代码视觉动画构建器；从工程用途看属于官方列出的常规 Web 应用方向。但最终发布仍应：

1. 把 GSAP 单列为“第三方专用许可”，不放进“全部 MIT 开源”宣传；
2. 保留其专有声明和当前适用条款快照；
3. 若产品未来增加可视化动画编辑器或生成器，重新评估禁止用途；
4. 升级 GSAP 前重新阅读当时条款。

### 10.2 Redis 7 浮动标签

官方许可页明确区分：

- Redis 7.2.x 及更早：BSD-3-Clause；
- Redis Community Edition 7.4.x～7.8.x：RSALv2 或 SSPLv1；
- Redis 8 及以后：RSALv2、SSPLv1 或 AGPLv3 三选一。

`redis:7-alpine` 只固定主版本，无法从仓库声明判断最终拉取的是 7.2 还是 7.4/7.8。因此在最终发行前必须：

- 明确选择具体 Redis 版本；
- 阅读并记录该版本许可证；
- 若希望保持 BSD-3-Clause 路径，固定经过安全评估的 7.2.x 具体补丁和镜像摘要；
- 若使用 7.4+，由负责人确认 RSALv2/SSPLv1 对预期部署和分发方式的影响；
- 安全性、维护期与许可证不能只选一个维度，必须共同评估。

### 10.3 `latest` 标签与不可复现性

`ollama/ollama:latest`、`ghcr.io/hkuds/lightrag:latest` 和 `bge-m3:latest` 会随时间指向不同内容。风险包括：

- 许可证或依赖树变化；
- 接口不兼容；
- 模型输出变化；
- 演示环境无法复现；
- 漏洞扫描结果失效。

最终交付应固定语义版本和不可变 digest/hash，并把它们写入构建证据。

### 10.4 静态媒体来源

`public/videos/test.mp4` 会被 Web 应用静态分发，但仓库内没有与该文件对应的来源、作者、许可或生成说明。即使它只是测试视频，也必须在发布前完成以下之一：

- 证明为团队自制并补充来源记录；
- 补齐第三方授权和署名；
- 若不再使用，从发行物中移除。

`public/favicon.svg` 同样应记录为团队原创、合法生成或具有明确授权。系统字体当前主要依赖浏览器/操作系统字体栈；若未来打包 WOFF/TTF 文件，应单独审计字体许可证。

### 10.5 用户资料与 AI 生成内容

开源软件许可不解决内容版权：

- 用户上传课程 PDF 不因进入知识库而变成可公开再分发内容；
- RAG 引用应保留来源，不应大段复制受保护内容；
- AI 生成图片、文本、代码和视频需按所用模型与服务条款审查；
- 比赛演示中使用的截图、配乐、头像和案例资料应另建素材授权清单。

## 11. 第三方声明与发布物组织

### 11.1 建议的发行文件

最终发行包至少应包含：

```text
LICENSE                         # AXIOM Space 自研代码的授权决定
THIRD_PARTY_NOTICES.md          # 第三方名称、版本、用途、版权和许可证索引
licenses/
  npm/<package>/LICENSE         # 实际分发依赖的许可证原文
  containers/<component>/...    # 容器软件许可证或官方链接快照
  models/<model>/MODEL_CARD.md  # 模型来源、版本/hash、许可证和用途
SBOM.cdx.json                   # 建议：CycloneDX 或等价 SBOM
```

若比赛提交格式不允许改变根目录，可将上述材料放在提交包约定的 `licenses/` 目录，但不能只在本说明书中口头说“遵守开源协议”。

### 11.2 `THIRD_PARTY_NOTICES.md` 最小字段

每一项至少包含：

- 名称；
- 精确版本；
- 上游主页或仓库；
- 在项目中的用途；
- 许可证标识；
- 版权声明；
- 许可证全文位置；
- 是否修改上游代码；
- 是否包含 NOTICE；
- 发行物中实际包含还是仅开发期使用。

### 11.3 署名不能替代许可义务

README 中写“感谢所有开源项目”有礼貌，但不等于合规。合规必须能从每个分发组件追溯到具体版本、许可证文本和必要声明。

## 12. 开发、提交与发布流程

### 12.1 新增依赖前

1. 明确为何不能用现有能力或少量自研代码解决；
2. 检查维护状态、安全记录、包体积和许可证；
3. 避免 Unknown、无许可证、不可再分发或与项目目标冲突的依赖；
4. 对专用许可、Copyleft 或模型许可请求负责人复核；
5. 记录用途和替代方案。

### 12.2 合入代码时

1. 锁定版本；
2. 将依赖加入对应 `dependencies` 或 `devDependencies`；
3. 更新第三方清单；
4. 保存许可证和 NOTICE；
5. 运行许可证扫描和安全扫描；
6. 验证生产构建是否实际包含该组件；
7. 在设计文档中准确说明“集成”，不冒充自研。

### 12.3 每次发布前

1. 从干净环境按锁文件安装；
2. 生成完整依赖树和 SBOM；
3. 扫描 Unknown、专用许可、Copyleft 和新许可证；
4. 对比上一个版本的依赖变化；
5. 固定容器摘要和模型 hash；
6. 汇总许可证与 NOTICE；
7. 清理无来源测试素材；
8. 由开发负责人和合规负责人双人确认；
9. 保存扫描输出、构建号和确认记录。

## 13. 公开发行合规基线

公开发行源码、镜像或安装包时，按以下基线逐项生成发行证据；参赛阶段的第三方使用披露以前述完整清单为准：
