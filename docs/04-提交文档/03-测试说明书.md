# AXIOM Space 测试说明书

> 文档定位：说明项目测试了什么、如何判断通过、已经取得哪些结果，以及当前仍有哪些验证边界。  
> 结果口径：历史运行记录与本次文档编制时的环境复核分开列示。

## 1. 测试目标

AXIOM Space 的测试不以“页面能打开”或“接口返回 200”为唯一标准。作为包含大模型、长期学习数据和多智能体工具调用的系统，测试重点回答以下问题：

1. 学习对象是否属于正确的用户和 Vault；
2. 路径、卡片、画像、评估和资源是否具有可检查的业务结果；
3. AI 输出是否能够指回来源、证据和目标对象；
4. 错误输入是否被明确拒绝，失败路径是否误写数据；
5. Agent 是否遵守工具契约、风险确认和审计边界；
6. RAG、推荐、UI ReadModel 等派生能力是否反向污染事实源；
7. 多类型资源是否具有正确结构并能形成实际文件；
8. 主流程在浏览器和真实基础设施中是否可以运行。

## 2. 测试依据

测试体系由“需求—对象—验收—用例—代码”逐层对应：

| 层次 | 测试依据 | 作用 |
|---|---|---|
| 产品需求 | 用户场景和功能范围 | 确定系统应该完成什么 |
| 对象契约 | 领域对象、关系和状态边界 | 确定数据和行为怎样才算正确 |
| 验收标准 | 可判断的输入、输出、错误和证据 | 确定怎样才算通过 |
| 测试计划 | 分层范围、公共数据和执行顺序 | 确定如何实施验证 |
| 可执行测试 | `test/acceptance/`、`test/e2e/` | 执行规格、API、数据库、Agent 和 UI 验证 |
| 状态记录 | `test/TEST-RUN-STATUS.md` | 保存最近一次完整门禁结论与已知缺口 |

## 3. 测试分层

### 3.1 SDD 规格验收

把领域对象和场景注册为可执行规格，检查必要字段、状态、来源、错误和边界。该层覆盖范围广，用于防止需求在实现过程中被遗漏。

命令：

```bash
pnpm test:acceptance
```

### 3.2 数据库测试

在真实 Prisma/PostgreSQL 环境中检查：

- User/Vault 所有权；
- 唯一性与外键约束；
- Card、Path、Session 等对象级联；
- 跨 Vault 访问隔离；
- RAG 索引幂等状态；
- 删除和失败路径的数据一致性。

命令：

```bash
pnpm test:acceptance:db
```

### 3.3 API/RPC 测试

直接验证 Hono 路由、Zod 校验、鉴权和业务输出，覆盖 Vault、Card、Galaxy、Learning、Event、RAG 等主要场景。

```bash
pnpm test:acceptance:api
```

### 3.4 Agent 契约测试

检查 Agent 运行时的关键安全与稳定边界：

- ToolContract 输入输出；
- 风险等级与确认；
- 审计记录；
- 凭据脱敏；
- Shell allowlist；
- 子 Agent 运行记录；
- 资源推送触发；
- LLM 用量边界。

```bash
pnpm test:acceptance:agent
```

### 3.5 Sidecar 与资源测试

验证 RAG、资源任务、manifest、PushRecord、通知副作用和多格式资源质量。真实模型模式额外检查不同资源格式是否满足结构要求。

```bash
pnpm test:acceptance:sidecar
RUN_REAL_LIVE_AI=1 pnpm test:acceptance:sidecar
```

### 3.6 UI 状态测试

检查模式切换、选中节点、布局、Learn 状态和持久化，确保界面状态不会改变权限和业务事实。

```bash
pnpm test:acceptance:ui-state
```

### 3.7 浏览器 E2E

Playwright 在真实浏览器中检查入口、模式切换、弹窗、快捷键和未登录状态等用户可见行为。

```bash
pnpm test:e2e:sdd
```

### 3.8 构建与静态检查

```bash
pnpm lint
pnpm build
```

生产构建同时检查 Prisma Client 生成、Next.js 构建和 TypeScript 类型。

## 4. 核心功能测试

### 4.1 六维画像

| 检查项 | 通过标准 | 证据 |
|---|---|---|
| 六维输出 | 六个维度均有结构化结果 | EducationProfile API/组件 |
| 证据来源 | 画像变化能指向消息、评估、卡片或能力记录 | Profile context |
| 置信度 | 无充分证据时不写成高置信结论 | Profile protocol |
| 随学更新 | 新观察形成历史并标注变化维度 | Profile history UI |
| 数据纯度 | Cognition 查询不修改 Card/Path/Profile 源数据 | API 契约测试 |

当前结论：画像 API、上下文装配、历史和页面展示已有运行时覆盖；画像判断质量仍需更多人工标注样本。

### 4.2 多智能体

| 检查项 | 通过标准 |
|---|---|
| 角色契约 | 资源角色输出符合对应 Schema |
| 工具边界 | 工具调用通过输入、输出和风险校验 |
| 高风险操作 | 未确认时不得执行 |
| 审计 | 成功和失败调用均记录 toolName、risk、status |
| 子 Agent | 保存角色、状态和运行记录 |
| 失败处理 | 超时、中断或格式失败进入 failed/降级状态 |

历史记录显示 Agent 合约 9 个场景和五角色真实模型 Schema smoke 已通过。

### 4.3 学习路径

检查主题生成路径、资料生成路径、步骤顺序唯一性、前置条件、状态流转、进度计算和路径调整来源。API 运行记录已覆盖路径创建、Step 执行和真实 AI 路径调整。

仍需加强：完整浏览器主链路中，从新建主题一直运行到真实评估和路径重算的自动化覆盖。

### 4.4 多类型资源

资源不是只检查“有文本”，而是按类型设置检查规则：

| 类型 | 主要检查 |
|---|---|
| document | 内容可读、标题和结构完整 |
| mindmap | 包含合法 mindmap 结构 |
| quiz | JSON 数组、题目数量、选项、答案和解释 |
| code | 代码块数量、说明和可辨识语言 |
| diagram | 合法 Mermaid 图类型和结构 |
| video | 包含 video/canvas/scene/animation 等可执行标记 |
| SVG | SVG 结构正确；降级时必须标记问题 |

真实模型测试记录表明 mindmap、quiz、code、diagram、video 等主要格式可稳定通过；SVG 对模型格式错误提供确定性 fallback，结果通过 `qualityIssues` 暴露降级状态。

### 4.5 学习评估

检查 AssessmentAttempt 是否保留用户原文、重复作答是否形成新记录、结果是否包含 score/passed/feedback/evidence，以及没有证据的结果是否禁止更新 mastery。

当前契约和部分 API 路径已覆盖；AssessmentResult 仍缺少足够规模的教师标注黄金集，不能仅凭自动化结构测试证明教学判断完全准确。

### 4.6 RAG 与知识图谱

检查内容哈希、重复索引跳过、索引失败状态、Vault workspace 隔离、引用业务对象、WikiLink 解析、跨 Vault 关系拒绝和隐含关系确认。

历史 API 记录包含跨 Vault RAG 隔离场景，Sidecar 记录包含 RAG 成功/失败状态和幂等验证。

### 4.7 文件渲染

HyperFrames smoke test 真实生成 DOCX、PDF、PPTX，并检查文件头和文件大小。最近保存的一次产物摘要为：

| 文件 | 大小 | 文件头 | 判断 |

| PDF | 24,982 bytes | `25504446` | 有效 PDF 文件头 |
| PPTX | 31,529 bytes | `504b0304` | 有效 ZIP/Office 容器 |

## 7. 已知测试缺口

1. Web MVP 完整“主题→路径→Forge→评估→Galaxy→画像反馈”尚未全部用真实数据自动化串联。
2. 资料导入的 API 与真实模型抽取已有记录，UI 全链路覆盖仍可加强。
3. 学习评估缺少足够规模的教师标注黄金集。
4. CardQualityScore、PromotionCriteria、AIContributionRatio 等部分细对象有规格注册但缺少独立运行时测试。
5. 真实模型资源会受模型和网络波动影响，需要在提交环境重新跑质量矩阵。
6. 性能方面已有超时和进度机制，但尚缺统一环境下的 P50/P95 基准报告。
7. 讯飞星火兼容接口已经实现，正式提交前仍应在有效赛事凭据下保存一次成功调用记录。

## 9. 验收结论

从仓库代码、历史门禁记录和已保存产物看，AXIOM Space 已建立覆盖领域契约、数据库、API、Agent、Sidecar、UI 状态和浏览器交互的多层测试体系，并对多智能体、RAG、资源文件和数据边界进行了针对性验证。
