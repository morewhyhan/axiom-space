# AXIOM Space 项目说明书

> 赛题：基于大模型的个性化资源生成与学习多智能体系统开发  
> 作品名称：AXIOM Space  
> 文档定位：说明项目为什么做、解决什么问题、已经实现什么，以及与赛题要求的对应关系。

## 1. 项目概述

AXIOM Space 是一套面向高校学习场景的个性化掌握学习系统。它关注的不是“再生成一份答案”，而是帮助学习者完成一条可以持续积累的学习闭环：系统先通过对话理解学生，再结合课程资料规划学习路径；学习过程中提供针对性辅导和多类型资源，引导学生用自己的语言形成知识卡片；最后依据学习证据进行评估，并把结果反馈给画像、路径和资源推荐。

系统当前以 Web 工作台形态交付，围绕五个相互衔接的工作空间展开：

| 工作空间 | 主要职责 | 形成的学习成果 |
|---|---|---|
| Dashboard | 汇总当前学习状态和近期活动 | 可追踪的学习概览 |
| Learn | 创建课程任务、规划路径、推进步骤 | 有顺序、有状态的学习路径 |
| Forge | 开展 AI 辅导、编辑卡片、生成资源 | 学习对话、知识卡片和资源产物 |
| Galaxy | 展示卡片、关系和学习结构 | 可浏览的个人知识图谱 |
| Cognition | 呈现画像、观察、知识缺口和建议 | 随学习更新的认知反馈 |

AXIOM Space 的核心主张是：**AI 不替学生完成学习，而是把诊断、解释、追问、输出、评估和沉淀组织成一套连续过程。**

## 2. 问题与需求

### 2.1 我们观察到的问题

高校学习者已经能够方便地获得课程、论文、视频和大模型回答，但“资源丰富”并不自然等于“真正掌握”。实际学习中仍存在四类断点：

1. 学习资料数量多、结构散，学生难以判断先学什么、后学什么。
2. 通用 AI 往往直接给出答案，但不了解学生已有基础、理解偏好和真实薄弱点。
3. 学习过程停留在阅读和收藏，缺少主动输出与可检查的掌握证据。
4. 一次对话结束后，问题、理解、资源和评估结果没有进入长期知识结构。

项目的公开资料研究验证了 AI 辅助学习已经成为高频行为，也支持主动回忆、自我解释、概念关联和智能导师等学习机制。我们同时保留一个审慎结论：现阶段研究以公开资料为主，产品的长期使用意愿和付费价值仍需要更大规模的真实用户验证。

### 2.2 目标用户

当前版本优先服务有明确学习目标、愿意投入思考和输出的高校学生与深度自学者，重点适配以下场景：

- 学习一门结构复杂的专业课程；
- 从一份讲义、教材或课程资料开始学习；
- 理解一个困难概念并验证是否真正掌握；
- 把零散理解沉淀为可复用的个人知识；
- 根据薄弱点持续调整后续学习计划。

### 2.3 需求如何进入产品

我们没有把需求简单转化为更多按钮，而是转化为一组相互约束的学习对象：

| 用户需求 | 产品机制 | 可见结果 |
|---|---|---|
| 不知道从哪里开始 | 画像诊断与路径规划 | Learn 中的章节、步骤与先后顺序 |
| 解释不适合自己 | 画像上下文进入 Agent | 不同基础、偏好和目标下的辅导策略 |
| 学过但没有掌握 | 卡片输出与费曼式评估 | 用户原文、质量反馈与掌握结果 |
| 资料无法长期复用 | 卡片、RAG 与知识图谱 | 可检索、可关联、可继续生长的知识库 |
| 后续学习缺少方向 | 缺口分析与资源推送 | 下一步任务、相关卡片和资源建议 |

## 3. 已实现的核心能力

### 3.1 对话式动态学习画像

系统已实现以对话证据为基础的六维学习画像，而不是要求学生填写一次性长表单。六个主维度分别回答：

1. **学什么**：学习目标及使用场景；
2. **会什么**：当前基础及知识边界；
3. **怎么讲最容易懂**：有效解释路径与资源偏好；
4. **哪里会卡住**：误解模式与学习阻抗；
5. **一次讲多少**：节奏与认知负荷；
6. **怎么算学会**：掌握标准与验证方式。

画像并非静态标签。前台教学对话、用户卡片、评估结果、路径进度和后台观察都可以形成画像证据。系统保存画像历史和变化维度，在 Cognition 中展示当前状态、观察记录和历史变化，并将压缩后的画像上下文重新注入教学、路径和推荐环节。

为降低“模型凭一次对话过度判断”的风险，画像更新保留来源、置信度和观察状态。当前代码已经实现画像上下文装配、问题生成、历史记录和页面展示；更大规模的画像准确性标注集仍是后续验证工作。

### 3.2 多智能体协同学习

系统实现的不是多个彼此割裂的聊天窗口，而是一套按职责分工的 Agent Harness：

- **前台教学 Agent**负责理解当前学习任务、解释概念、调用工具、主动追问并维持对话连续性；
- **后台分析 Agent**在不打断用户的情况下整理学习证据、识别薄弱点并推动画像更新；
- **资源角色 Agent**按文档、导图、题目、代码、图示和视频等资源类型隔离任务与输出规范；
- **编排层**负责意图路由、上下文装配、工具契约、并行执行、状态记录、结果预算和失败处理。

对用户而言，多智能体协作最终体现为可追踪的页面变化：资源生成进度、AI 观察记录、画像变化、学习步骤状态、RAG 引用和推荐理由，而不是仅展示一张架构图。

### 3.3 个性化学习路径

Learn 支持从学习主题或导入资料生成学习路径。路径由章节和步骤构成，每个步骤具备顺序、状态、概念、前置关系和关联卡片等信息。学生可以从具体步骤进入 Forge，使后续对话围绕明确学习对象进行。

系统已经实现：

- 从主题生成路径；
- 从课程资料抽取概念并生成路径；
- 步骤状态流转与进度计算；
- 当前步骤与 Forge 会话绑定；
- 依据学习证据生成路径调整建议；
- 将知识缺口、画像偏好和路径状态用于资源建议。

当前路径调整 API 和真实模型调用已经有测试记录，但“评估质量是否足以长期替代教师判断”仍需要专门的人工黄金集持续校准。因此系统把调整建议保留为有原因、有来源的记录，而不是把模型判断包装成绝对结论。

### 3.4 多类型学习资源生成

系统围绕当前课程、路径步骤或知识卡片生成资源，避免生成与学习现场脱节的通用材料。目前资源体系支持：

| 资源类型 | 主要用途 | 当前承载形式 |
|---|---|---|
| 讲解文档 | 系统解释概念、边界和应用 | Markdown，可进一步渲染为 DOCX/PDF |
| 知识导图 | 展示概念层次 | Mermaid mindmap |
| 练习题 | 检查理解与迁移 | 结构化 JSON 与交互预览 |
| 代码案例 | 连接概念与实践 | Markdown 代码资源 |
| 关系图示 | 展示流程、结构或依赖 | Mermaid diagram |
| 交互式视频/动画 | 以场景和动画辅助解释 | 可预览 HTML；也支持视频 URL 播放 |
| 演示文稿 | 形成可交付课件 | PPTX |

资源生成包含类型状态、生成进度、质量信号、问题列表和文件清单。项目已经完成 DOCX、PDF、PPTX 的真实文件渲染 smoke 测试，并保存了可检查产物。真实模型资源矩阵测试表明 mindmap、quiz、code、diagram、video 等主要格式可稳定生成；SVG 在模型输出失败时采用确定性降级，并通过质量字段明确标记，不把降级结果伪装成高质量产物。

### 3.5 个性化资源推送

系统实现资源与任务建议盒，根据当前卡片、路径、知识关系、画像偏好和学习事件寻找候选项。建议项包含目标、理由、置信度和建议格式，用户可以理解“为什么现在推荐它”。

推送机制遵守两个边界：一是建议必须指向真实学习对象或证据；二是推荐不直接替用户修改卡片、路径和知识关系，需要经过业务服务或用户操作确认。

### 3.6 智能辅导与主动追问

Forge 已实现围绕卡片和路径步骤的对话式辅导，包括：

- 流式回复与 Markdown 渲染；
- 数学公式、代码块和结构化内容展示；
- 当前 Vault、路径、卡片、画像和 RAG 上下文装配；
- 主动提出理解检查问题；
- 工具调用过程和高风险操作确认；
- 对话持久化、恢复与压缩；
- 个人知识引用和相关卡片建议。

系统的教学原则是通过解释、反例、追问和输出推动理解，而不是仅用更长的回答制造“学会了”的错觉。

### 3.7 学习效果评估

系统当前形成三层评估思路并已有相应实现入口：

1. **理解层**：通过费曼式对话和迁移问题检查用户能否用自己的语言解释；
2. **沉淀层**：通过卡片质量规则检查定义、边界、例子、证据、关联和应用；
3. **系统层**：综合评估、卡片、路径和行为证据，更新掌握状态、画像和后续建议。

卡片升级不是简单修改类型。系统保存升级尝试、拒绝原因和修订记录；路径完成也不是“看过即完成”，需要结合用户输出和评估证据。当前评估与路径调整契约已经实现，评估结果质量仍缺少足够规模的人工黄金集，这一限制已纳入测试说明。

### 3.8 个人知识图谱与长期沉淀

系统以 Card、Edge、Cluster 为核心构建个人知识结构：

- 卡片区分灵感、文献和永久知识；
- WikiLink 可以在当前 Vault 内解析为显式关系；
- RAG 可以发现候选隐含关系，但不会未经确认直接污染源数据；
- Galaxy 提供星系、平面、同心、任务流和证据等观察视角；
- Cognition 从图谱、路径、画像和评估中展示知识缺口和下一步。

Galaxy 的目标不是装饰，而是让“我已经形成了哪些知识、它们如何关联、下一步缺什么”变得可见。

## 4. 一次完整学习如何发生

以“反向传播为什么需要链式法则”为例，系统中的主流程如下：

1. 学生进入课程 Vault，系统在对话中了解基础、目标和理解偏好。
2. 画像形成初始六维判断，并保留证据和置信度。
3. 学生输入主题或导入课程资料，系统生成有顺序的学习路径。
4. 学生从当前步骤进入 Forge，Agent 获得课程、路径、卡片、画像和个人知识上下文。
5. 前台 Agent 进行解释、举例和追问；后台分析记录学习证据。
6. 学生用自己的语言补充卡片，系统保存修订并同步个人知识索引。
7. 资源 Agent 围绕同一知识点生成文档、导图、题目、代码和动画等学习材料。
8. 系统根据用户回答和卡片内容进行评估；不充分时给出明确缺口。
9. 通过评估后更新步骤进度、能力记录和画像，并形成后续路径或资源建议。
10. 学习成果进入 Galaxy 和 Cognition，成为下一轮学习可复用的上下文。

## 5. 项目创新点

### 5.1 从“答案生成”转向“掌握闭环”

AXIOM Space 将大模型能力放入一个有状态、有证据、有后续动作的学习过程。一次 AI 回答不会被自动视为学习成果，只有用户输出、评估证据和知识沉淀共同发生，学习状态才向前推进。

### 5.2 用 Agent Harness 组织协作，而不是展示多个机器人

系统把前台教学、后台分析和资源生成拆成不同职责，并通过统一上下文、工具契约、状态机和审计记录协同。这样既保留不同角色的专长，又避免多个 Agent 各自输出、无法形成业务结果的问题。

### 5.3 画像不是展示标签，而是参与决策

六维画像进入教学提示、路径调整和资源推荐；新的学习证据又反向更新画像。画像因此成为学习系统的运行变量，而不是注册完成后不再变化的静态档案。

### 5.4 个人知识是事实源，RAG 是可重建增强层

卡片和关系保存在业务数据库中，LightRAG 负责检索增强。即使索引服务失败，用户知识仍可读取；索引状态、内容哈希和失败原因独立记录。这种设计兼顾了 AI 检索能力与学习数据的可靠性。

### 5.5 资源生成具备质量和交付意识

资源不仅在聊天框中显示，还具备类型状态、进度、质量问题、清单和文件渲染能力。系统已经能够输出真实 DOCX、PDF、PPTX 文件，并对格式失败或降级结果显式标记。

## 7. 完成度与边界

本项目已经形成可运行 Web 应用、真实数据模型、RPC API、多智能体运行时、课程数据、资源渲染和自动化验收体系，不是仅由静态页面和演示脚本组成的概念作品。

同时，我们不回避以下边界：

- 完整“主题到评估再到画像更新”的真实数据浏览器闭环尚未全部纳入自动化 E2E；
- 学习评估的契约和路径调整已实现，但仍需扩大教师标注黄金集；
- 不同大模型生成复杂资源时存在格式波动，系统通过校验和降级标记处理，而非声称完全消除；
- 讯飞星火已完成适配与兼容检查入口，但当前系统保留多模型配置能力；
- 公开资料研究较充分，真实学生长期使用研究仍需继续开展。

这些边界并不削弱项目已经完成的工程成果。相反，它们体现了团队对 AI 教育系统可靠性、可验证性和真实应用条件的审慎判断。

## 8. 项目价值

对学生而言，AXIOM Space 将一次次孤立的 AI 问答变成可继续学习的路径、卡片、关系和反馈；对教师而言，它提供了观察学习证据和识别薄弱点的新方式；对高校课程建设而言，它展示了如何把课程资料、多智能体生成、个性化教学和学习评估连接为一套可扩展系统。

AXIOM Space 当前完成的最重要工作，不是让 AI 生成更多内容，而是为“内容如何真正成为学生自己的知识”建立了一套可运行、可追踪、可继续验证的产品与技术框架。
