# AXIOM Space 使用说明书

> 文档定位：说明如何安装、启动和体验 AXIOM Space，并给出评委最短验收路径。  
> 默认形态：Web 应用；完整本地环境使用 PostgreSQL、Redis、Ollama 和 LightRAG。

## 1. 运行环境

### 1.1 推荐环境

| 项目 | 推荐要求 |
|---|---|
| 操作系统 | Windows 11 + WSL2、Linux 或 macOS |
| Node.js | 20.x |
| 包管理器 | pnpm，与锁文件兼容的当前版本 |
| Docker | Docker Desktop/Engine，支持 Docker Compose |
| 浏览器 | 最新版 Chrome、Edge 或 Chromium |
| 内存 | 建议 16 GB 及以上，完整 AI/RAG 环境更为稳定 |
| 磁盘 | 建议预留 10 GB 以上用于镜像、模型和测试产物 |

完整开发启动脚本会使用以下服务：

- PostgreSQL 16：业务数据库；
- Redis 7：后台任务基础设施；
- Ollama：LightRAG embedding 模型；
- LightRAG：个人知识检索增强；
- Next.js：Web 应用；
- Jobs Worker：后台任务进程。

### 1.2 最小与完整模式

| 模式 | 能力 | 启动方式 |
|---|---|---|
| 前端/应用模式 | 启动 Next.js，适合已有外部数据库和服务时使用 | `pnpm dev:web` |
| 完整本地模式 | 自动启动数据库、Redis、Ollama、LightRAG、Worker 和 Web | `pnpm dev` |

仅运行前端命令不代表全部 AI、RAG 和异步能力可用。评委验收建议使用完整本地模式或团队提供的已配置演示环境。

## 2. 安装步骤

### 2.1 安装依赖

在项目根目录执行：

```bash
pnpm install --frozen-lockfile
```

如本机尚未安装 pnpm，可通过 Corepack 启用：

```bash
corepack enable
corepack prepare pnpm@latest --activate
```

正式验收应使用 `pnpm-lock.yaml`，避免依赖版本漂移。

### 2.2 准备环境变量

项目包含 `.env.example`。首次执行 `pnpm dev`、`pnpm build` 或安装后脚本时，`scripts/bootstrap-dev.mjs` 可以在缺少 `.env` 时复制模板。

也可以手工执行：

```bash
cp .env.example .env
```

不要把真实密钥提交到源码或截图中。

### 2.3 核心配置

```dotenv
DATABASE_URL=postgresql://axiom:axiom_dev_password@127.0.0.1:5432/axiom?schema=public
NEXT_PUBLIC_APP_URL=http://localhost:3000
BETTER_AUTH_URL=http://localhost:3000
NEXT_PUBLIC_BETTER_AUTH_URL=http://localhost:3000
BETTER_AUTH_SECRET=<至少32字节随机密钥>

AI_PROVIDER=<模型提供商>
AI_MODEL=<模型名称>
AI_API_KEY=<模型密钥>

REDIS_URL=redis://127.0.0.1:6379
LIGHTRAG_BASE_URL=http://127.0.0.1:9621
OLLAMA_EMBEDDING_MODEL=bge-m3:latest
```

完整 LightRAG 模型与 embedding 配置见 `.env.example`。业务聊天模型和 LightRAG 的模型配置相互独立，便于分别选择和排错。

### 2.4 配置讯飞星火

讯飞适配器从以下环境变量读取赛事账号配置：

```dotenv
XUNFEI_APP_ID=<讯飞应用ID>
XUNFEI_API_KEY=<讯飞API Key>
XUNFEI_API_SECRET=<讯飞API Secret>
XUNFEI_SPARK_VERSION=v3.5
```

当前 `.env.example` 尚未列出这四项，正式提交前应补入模板但不填写真实密钥。

系统提供兼容性检查接口：

```text
GET /api/agent/xunfei/compat
```

该接口需要登录态；配置缺失时会明确返回所需变量。

## 3. 启动系统

### 3.1 完整启动

确认 Docker 已启动，然后执行：

```bash
pnpm dev
```

脚本将依次：

1. 检查 Docker 和 Compose；
2. 启动 PostgreSQL、Redis 和 Ollama；
3. 检查并下载 embedding 模型；
4. 启动 LightRAG；
5. 执行 Prisma `db push`；
6. 启动 Jobs Worker；
7. 从 3000 开始寻找可用端口并启动 Next.js。

终端会显示实际访问地址，通常为：

```text
http://localhost:3000
```

如果 3000 被占用，脚本会在后续端口中选择可用端口。

### 3.2 生产构建

```bash
pnpm build
pnpm start
```

生产环境必须配置真实 `BETTER_AUTH_SECRET`、公开 URL、数据库和模型凭据，不应启用 `DEV_MODE`。

### 3.3 健康检查

```text
GET /api/health
```

正常响应示例：

```json
{
  "status": "ok",
  "timestamp": 0
}
```

`timestamp` 为实际服务器时间戳。

## 4. 初始化演示数据

### 4.1 CS408 课程数据

```bash
pnpm db:seed:cs408
pnpm db:verify:cs408
```

CS408 演示课程覆盖：

- 数据结构；
- 计算机组成原理；
- 操作系统；
- 计算机网络。

课程材料分别覆盖数据结构、计算机组成原理、操作系统和计算机网络，并随提交作品包一同提供。

`scripts/seed-cs408.ts` 会创建演示用户、课程 Vault、领域分组、主题节点、知识卡片和关系边。种子脚本中的演示账号为：

```text
邮箱：demo@axiom.space
密码：demo123456
```

该账号只用于本地或隔离演示环境，公开部署时必须修改或移除默认密码。

### 4.2 A3 黄金演示数据

```bash
pnpm db:seed:a3-golden
pnpm check:a3-golden
```

这套数据用于稳定复现赛题演示主线。最终提交时应以团队确认的演示账号和脚本为准。

## 5. 评委最短体验路径

### 5.1 登录并选择课程

1. 打开系统地址。
2. 使用演示账号登录。
3. 进入 CS408 或 A3 演示 Vault。
4. 在 Dashboard 查看学习统计、知识节点和近期活动。

验收重点：系统具有用户和课程空间边界，不是无状态的公开聊天页面。

### 5.2 查看对话式画像

1. 进入 Forge 或初始画像对话。
2. 回答自己的学习目标、基础和理解偏好。
3. 进入 Cognition。
4. 查看“学什么、会什么、怎么讲最容易懂、哪里会卡住、一次讲多少、怎么算学会”六个维度。
5. 查看 AI 观察记录和画像历史。

验收重点：画像来自对话和学习证据，并能在后续学习后发生变化。

### 5.3 创建学习路径

1. 进入 Learn。
2. 点击创建新任务。
3. 选择从主题生成，输入一个具体概念。
4. 选择基础、进阶或高级难度。
5. 提交后查看章节、步骤、顺序、状态和进度。

推荐演示主题：

```text
反向传播为什么需要链式法则
```

也可以选择导入资料，从本地课程文档抽取概念并生成路径。

### 5.4 进入 AI 学习工作台

1. 在当前步骤点击进入 AI 工作台。
2. 确认 Forge 顶部显示当前任务、步骤和卡片。
3. 输入：

```text
先解释这个概念，再问我一个检查理解的问题。不要直接替我写最终卡片。
```

4. 观察流式回复、Markdown、公式、代码或引用。
5. 在右侧卡片编辑器用自己的语言补充定义、例子、边界、关联和应用。
6. 保存卡片，查看 RAG 状态和相关卡片建议。

验收重点：对话绑定学习对象，用户输出进入长期卡片，而不是停留在聊天记录。

### 5.5 生成多类型资源

在 Forge 发起资源生成，选择或请求：

```text
讲解文档、知识导图、练习题、代码案例、关系图示、交互式动画
```

页面将显示不同资源的等待、生成、校验、完成或失败状态。完成后可打开对应预览；文件资源支持 DOCX、PDF 或 PPTX 等格式。

验收重点：资源围绕当前卡片/路径和画像生成，具备实际结构、进度与质量状态。

### 5.6 完成评估

1. 回到 Learn 或打开评估面板。
2. 回答理解或迁移问题。
3. 查看反馈、证据和薄弱点。
4. 如未通过，根据反馈补充理解和卡片。
5. 再次评估后观察步骤状态与路径建议变化。

验收重点：系统不会把“看过 AI 回答”直接视为掌握。

### 5.7 查看知识沉淀

1. 进入 Galaxy，查看新增卡片和关系。
2. 切换星系、平面、同心、任务流或证据视角。
3. 点击节点查看关联卡片。
4. 进入 Cognition，查看画像变化、知识缺口和下一步建议。

验收重点：学习结果同时进入知识结构、认知反馈和后续任务。

## 7. 常见问题

### 7.1 `pnpm dev` 提示 Docker 不可用

确认 Docker Desktop/Engine 已启动，并检查：

```bash
docker version
docker compose version
```

Windows 用户需要开启 WSL integration。

### 7.2 数据库连接失败

检查 PostgreSQL 容器和 `DATABASE_URL`：

```bash
docker compose ps
pnpm db:push
```

默认端口为 5432；如端口冲突，修改 `POSTGRES_PORT` 和 `DATABASE_URL`。

### 7.3 LightRAG 不可用

系统仍以 PostgreSQL 为事实源，卡片不会因 LightRAG 失败而丢失。检查：

- `LIGHTRAG_BASE_URL`；
- LightRAG 容器健康状态；
- embedding 模型是否已下载；
- LightRAG 的 LLM/embedding 密钥。

RAG 状态会显示 `disabled`、`indexing`、`indexed` 或 `failed`。

### 7.4 AI 对话没有响应

检查 `AI_PROVIDER`、`AI_MODEL`、`AI_API_KEY` 和网络。模型凭据错误会影响 Forge、路径生成和资源生成，但不应影响已有卡片和本地页面读取。

### 7.5 讯飞兼容检查失败

确认四个 `XUNFEI_*` 变量，服务器时间是否准确，以及运行时支持 WebSocket。讯飞适配器默认超时为 60 秒。

### 7.6 测试提示缺少 `esbuild`

当前依赖目录可能是不完整安装。删除或覆盖依赖前应先确认没有其他本地工作，然后使用锁文件重新安装：

```bash
pnpm install --frozen-lockfile
```

再运行测试。不要把依赖加载失败误判为业务测试失败或通过。

### 7.7 PDF/PPTX 渲染失败

PDF 依赖 Puppeteer/Chromium；PPTX 渲染还依赖 `scripts/mckinsey-pptx/` 的 Python 环境。检查相应运行时和脚本说明。页面中的 Markdown、Mermaid 等基础资源不依赖 PPTX 渲染器。

## 8. 数据与安全注意事项

1. `.env`、真实 API Key、Session Cookie 不得进入提交仓库、测试产物或演示截图。
2. `DEV_MODE` 只能用于本地开发，生产环境必须关闭。
3. 默认演示密码不得用于公开部署。
4. AI 生成内容应结合引用、课程资料和人工判断使用。
5. 高风险 Agent 操作需要用户确认，不应绕过确认接口直接修改数据。
6. 测试产物提交前应再次检查 Authorization、token 和用户隐私信息。
7. 第三方组件许可证以锁文件对应版本为准，正式提交包应附完整清单。

## 10. 验收建议

评委现场体验建议先使用已初始化的演示环境，按照第 5 章完成一条 6～7 分钟的主流程；随后根据需要查看开发说明书中的代码索引和测试说明书中的运行证据。

如果需要在全新机器部署，应先完成依赖安装、环境变量、Docker 基础设施、数据库初始化和课程种子，再执行提交前测试门禁。这样可以将“产品体验问题”和“环境配置问题”清楚分开。
