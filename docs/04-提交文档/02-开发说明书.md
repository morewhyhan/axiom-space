# AXIOM Space 开发说明书

> 文档定位：说明系统如何设计、实现和集成，并明确自主开发、第三方能力及当前技术边界。

## 1. 开发目标

AXIOM Space 的开发目标不是搭建一个调用大模型的聊天页面，而是把个性化学习需要的长期状态和业务规则落实为可运行软件：

- 学习画像需要有维度、证据、置信度和历史；
- 学习路径需要有步骤、顺序、状态和调整记录；
- AI 对话需要绑定课程、路径和卡片上下文；
- 资源生成需要有角色分工、进度、质量和实际产物；
- 学习结果需要进入卡片、图谱、评估和后续推荐；
- Agent 调用工具需要遵守权限、风险确认和审计边界。

为此，项目采用 Clean Architecture 与领域驱动相结合的组织方式，让业务规则尽量独立于具体 UI、数据库和模型服务。

## 2. 总体架构

```text
表现层
Next.js App Router + React + Tailwind + shadcn/ui
        |
状态与数据访问层
Zustand + React Query + typed Hono RPC hooks
        |
API 网关层
Hono routes + Zod validation + unified errors + auth
        |
核心领域层
Agent / Learning / Profile / Path / Push / RAG contracts
        |
基础设施层
Prisma + PostgreSQL + Redis/BullMQ + LightRAG + File renderers + LLM providers
```

依赖方向从外向内：页面通过 Hook 调用 Hono RPC；API 负责身份、校验和转发；核心层承载业务决策；数据库、RAG、队列和模型适配位于基础设施边界。

## 3. 前端实现

### 3.1 页面结构

项目使用单一 Web 工作台承载连续学习流程，而不是将功能拆成彼此无关的演示页面。

- `components/dashboard/`：学习统计和近期活动；
- `components/learn/`：路径创建、章节步骤、评估和推送建议；
- `components/forge/`：聊天、卡片编辑、资源进度、RAG 引用和 Agent 编排状态；
- `components/galaxy/` 与 `components/three/`：知识图谱数据和 3D 可视化；
- `components/cognition/`：六维画像、观察记录和历史；
- `components/resources/`：多类型资源预览与下载。

页面使用稳定的模式导航串联主流程。全局模式、选中节点、当前 Vault 和面板状态由 Zustand 管理；服务端数据通过 React Query Hooks 获取，避免组件直接拼接 HTTP 请求。

### 3.2 类型化 RPC

所有业务 API 统一挂载在 `/api` 下，包含 Agent、Learning、Vault、Dashboard、Galaxy、Cognition、Events 和 RAG 等路由组。前端使用 Hono RPC 客户端推导请求和响应类型，主要调用封装在：

- `lib/api-client.ts`
- `hooks/use-agent.ts`
- `hooks/use-learning.ts`
- `hooks/use-galaxy.ts`
- `hooks/use-cognition.ts`
- `hooks/use-dashboard.ts`

API 层统一处理认证、Zod 校验和结构化错误，减少页面与后端契约漂移。

## 4. 数据与领域模型

Prisma 数据库承载用户知识和学习过程的事实源。核心对象包括：

| 领域 | 核心对象 | 作用 |
|---|---|---|
| 身份 | User、Account、Session、Vault | 用户、登录态和课程空间边界 |
| 知识 | Card、Cluster、Edge、CardRevision | 卡片内容、分组、关系和修订 |
| 学习 | LearningPath、LearningPathStep | 路径、步骤、进度和状态 |
| 会话 | LearningSession、LearningMessage | 绑定学习对象的对话历史 |
| 画像 | VaultMemory、VaultCapability、VaultSkill、EducationProfileHistory | 长期记忆、能力、技能和画像历史 |
| 评估 | AssessmentResult、PromotionAttempt | 掌握证据和卡片升级尝试 |
| Agent | AgentSession、AgentAuditLog、AgentConfirmationToken | 运行状态、审计和确认 |
| RAG | RagDocumentIndex | 索引状态、内容哈希和错误 |
| 资源 | ResourceGenerationJob、PushRecord | 资源任务、清单和推送记录 |
| 事件 | DomainEvent、NotificationReceipt | 领域变化和通知消费 |

所有学习对象以 User/Vault 为主要边界。数据库约束和 API 校验共同防止跨 Vault 读取、关系错连和派生层反向污染源数据。

## 5. 多智能体系统

### 5.1 分层设计

项目在 PI Agent 基础循环之外实现了面向学习业务的 Harness。它由三层构成：

1. **前台教学层**：处理学生当前问题，保持流式交互；
2. **后台认知层**：分析证据、画像信号和知识缺口；
3. **专项资源层**：按资源类型执行受约束的生成任务。

系统不是为了追求 Agent 数量，而是把不同决策放到适合的职责边界中。资源角色拥有独立输出规范，后台分析不打断前台对话，高风险工具操作需要确认。

### 5.2 一轮对话的运行链路

```text
用户消息
  -> 身份与 Vault 校验
  -> 绑定 Path / Step / Card / Session
  -> 装配画像、记忆、RAG 和近期对话
  -> 意图路由与教学策略选择
  -> Agent 推理和工具调用
  -> 流式返回页面
  -> 保存消息、审计和领域事件
  -> 后台提取画像信号与学习证据
  -> 更新 Cognition / Learn 可见状态
```

关键实现位于 `server/core/agent/`：

- `ChatOrchestrator.ts`：会话编排和消息流；
- `orchestration-engine.ts`：编排任务和角色协作；
- `IntentRouter.ts`：意图识别；
- `ProjectContextLoader.ts`、`agent-context.ts`：上下文装配；
- `ToolContracts.ts`：工具输入输出、风险和副作用；
- `ParallelToolExecution.ts`：可并行工具执行；
- `BackgroundAnalyzer.ts`：后台证据分析；
- `ResourceGenerationOrchestrator.ts`：资源角色编排；
- `AgentStateMachine.ts`：运行状态；
- `OperationConfirmation.ts`：高风险操作确认；
- `LLMUsageTracker.ts`：模型用量边界；
- `MemorySummarizer.ts`：长对话压缩。

### 5.3 工具治理

每个 Agent 工具具有明确名称、输入、输出、风险等级和副作用说明。工具调用经过 Schema 校验；需要改变用户数据或执行高风险动作时进入确认流程；成功和失败都可以形成审计记录。日志脱敏规则避免保存 Authorization、API key 和 token 原文。

这套机制使大模型处于受约束的业务运行时中，而不是获得不透明的数据库或系统访问权。

### 5.4 Agent 消息协议与工作流状态

多智能体协作使用显式消息协议，而不是让多个模型通过自然语言自由接力。每条 `AgentMessage` 至少包含：

```typescript
interface AgentMessage {
  messageId: string
  from: AgentRole
  to: AgentRole | 'broadcast'
  type: 'task' | 'result' | 'feedback' | 'status'
  payload: Record<string, unknown>
  timestamp: number
  priority: 'high' | 'normal' | 'low'
}
```

其中 `messageId` 用于去重，`from/to` 用于定向路由，`type` 区分任务与结果，`payload` 传递结构化上下文。消息总线保留有界历史，避免运行时间增长导致内存无限累积。

编排实例使用独立 `orchestrationId`，其状态包含：

- `steps`：每个角色的任务、输入、输出、状态和错误；
- `currentStepIndex`：当前步骤；
- `progress`：0～100 的可视化进度；
- `results`：前序步骤向后续角色传递的结果；
- `logs`：带角色、时间和级别的运行记录；
- `startedAt/completedAt`：运行时间边界。

资源生成工作流采用：

```text
Profile -> Planner -> Generator -> Reviewer -> Pusher
```

评估反馈工作流采用：

```text
Profile -> Planner -> Generator -> Pusher
```

每一步只能从 `pending` 进入 `running`，然后进入 `completed` 或 `failed`。编排器等待与当前 `stepId` 匹配的结果消息，默认设置 30 秒步骤超时；失败会写入步骤错误和编排日志，不继续伪造后续成功结果。

需要区分两套互补机制：`orchestration-engine.ts` 提供面向业务工作流的显式角色协议；`ChatOrchestrator.ts` 与 PI Agent Core 承载真实对话循环、工具调用和流式返回。前者解决跨角色业务协作，后者解决单轮交互执行，两者不是重复实现。

### 5.5 工具风险模型

`ToolContracts.ts` 为生产工具维护独立于提示词的元数据：

```typescript
type ToolRisk =
  | 'read'
  | 'write'
  | 'destructive'
  | 'network'
  | 'llm'
  | 'interactive'
  | 'orchestration'

interface ToolContract {
  name: string
  risk: ToolRisk[]
  requiresVault: boolean
  requiresConfirmation?: boolean
  idempotent: boolean
  sideEffects: string[]
  successDetails?: string[]
}
```

该设计解决三个问题：

1. **可执行边界**：`read`、`write`、`delete_card`、`push_resource` 等能力不再只是模型能看到的文字描述，而是可供运行时、评测和审计读取的契约。
2. **破坏性操作前置确认**：删除卡片、批量抽取、合并重复卡片、导入和清理链接等操作显式标记 `requiresConfirmation`。
3. **幂等语义**：读取和部分关系写入可重试；文件覆盖、资源生成和删除等非幂等行为不能被无条件重复执行。

例如，`add_graph_edge` 被设计为 Vault 内的幂等写入；`delete_card` 同时标记 destructive、需要确认，并明确副作用包含删除关联边。这样审计系统能够判断“模型做了什么”，而不必重新解释一段聊天文本。

## 6. 六维画像实现

画像核心实现分为四个阶段：

1. **证据采集**：从用户消息、卡片、评估、路径和能力记录获取信号；
2. **信号判断**：识别是否出现值得更新的画像变化，减少每轮无差别重算；
3. **画像归纳**：映射到六个主维度及动态子维度，保存证据、置信度和状态；
4. **上下文压缩**：只把影响当前教学的必要信息注入 Agent。

主要实现位于：

- `server/core/learning/education-profile.ts`
- `server/core/learning/profile-protocol.ts`
- `server/core/learning/profile-context.ts`
- `server/core/agent/profile-questioner.ts`
- `components/cognition/`

系统保留画像历史，页面可以展示变化维度。画像更新采用“证据支持的渐进判断”，不把单次模型推断当作永久事实。

### 6.1 画像协议不是六个分数

每个画像维度在 `PROFILE_DIMENSION_PROTOCOL` 中都定义六类约束：

| 协议字段 | 作用 |
|---|---|
| `purpose` | 该维度解决什么教学判断 |
| `initialScope` | 首次画像最多判断到什么程度 |
| `extractionTargets` | 可以提取哪些子信息 |
| `triggerSources` | 哪些学习事件可以触发更新 |
| `doNotExtract` | 哪些表面信号不得写入画像 |
| `teachingImpact` | 该维度如何改变下一轮教学 |

例如，“会什么”维度允许从用户自己的解释、测验、卡片和追问回答中取证，但明确禁止把“AI 刚讲过”或“用户礼貌地说懂了”当作稳定掌握。“哪里会卡”只有在重复错误、概念混淆或迁移失败时才形成稳定判断，第一次提问不能直接被贴成薄弱标签。

这种“允许提取 + 禁止提取”双向协议比单纯让模型打六个分更保守，也更适合教育场景。

### 6.2 画像更新采用两条通道

系统将画像更新分为低成本信号检查和深度归纳：

```text
Channel A：当前对话/事件 -> 信号检查 -> 是否值得更新
Channel B：证据聚合 -> 六维归纳 -> 冲突处理 -> 历史快照
```

Channel A 用于发现明确目标、基础暴露、解释偏好、重复卡点、负荷反馈和掌握证据，避免每轮对话都启动一次全量画像重算。只有出现显著信号、达到周期阈值或会话结束等条件时，才进入 Channel B。

Channel B 读取多源证据，形成观察项并处理三种状态：

- `active`：当前有效；
- `superseded`：被更近、更强证据替代；
- `refuted`：被反证否定。

画像因此能够表达“系统过去这样判断，但后来证据发生变化”，而不是覆盖掉历史。

### 6.3 画像到教学策略的压缩

完整画像不直接全部塞入每一次模型上下文。`profile-context.ts` 将其压缩为当前任务需要的教学策略，例如：

- 是否需要补前置概念；
- 先例子、图解、代码还是定义；
- 一轮解释的信息块大小；
- 是否优先安排练习或迁移任务；
- 需要避免的已知混淆；
- 本轮采用什么掌握标准。

这种压缩一方面减少 token 消耗，另一方面避免大量历史标签压过当前问题。注入内容保留来源和置信度，低置信观察只能作为提示，不能成为确定性教学结论。

### 6.4 画像如何反向影响系统

画像不是 Cognition 页面专用 ReadModel，它进入三个决策点：

1. **Agent 教学**：改变解释入口、追问方式和节奏；
2. **路径调整**：决定补前置、增加练习、拆小步骤或推进迁移；
3. **资源推送**：决定更适合题目、代码、导图还是总结材料。

新的用户输出、评估和资源反馈又回流为证据，形成闭环：

```text
画像 -> 教学/路径/资源 -> 用户行为 -> 评估证据 -> 画像历史
```

## 7. 学习路径、评估与推荐

### 7.1 路径生成

路径可以从主题或导入资料产生。资料导入服务执行内容哈希、语义切分、概念抽取、关系抽取、卡片写入和路径生成，并返回创建、跳过和错误统计。较长文档采用带重叠的语义分块，抽取结果标明来自原文、混合补充或 AI 生成。

主要实现：

- `server/core/learning/document-import-service.ts`
- `server/api/routes/learning.ts`
- `components/learn/workspace/`

#### 文档导入管线

导入服务不是将全文一次性提交给模型。其实际处理链路为：

```text
输入校验
 -> SHA-256 内容哈希
 -> 长度判断
 -> 语义分块（主片段 + overlap + headingPath）
 -> 低温度概念/灵感/关系抽取
 -> 跨分块去重
 -> 必要条件结构规划
 -> 卡片与关系写入
 -> 学习路径生成
 -> 结果统计与领域事件
```

长文分块保留 `index/total`、标题路径、前向重叠和全局摘要。后续分块不仅看到局部正文，还获得持续更新的 `globalDigest`，用于降低跨章节概念割裂。抽取温度设为较低水平，以优先保证结构稳定。

导入结果将内容覆盖分为：

- `documented`：有原文证据；
- `mixed`：原文与 AI 结构补充混合；
- `ai_generated`：为结构完整性生成，但没有直接文献证据。

页面和后续逻辑可以据此区分“材料里确实写了什么”和“系统为了学习结构补了什么”。

#### 必要条件建模

项目没有简单按关键词把文档平铺成卡片，而是为父概念规划多个必要条件：

- `description`：这个子条件是什么；
- `whyNecessary`：删除它会缺少哪一块解释；
- `sufficiencyRole`：它与其他条件怎样共同覆盖父概念；
- `coverage`：证据覆盖类型；
- `evidenceTitles`：可以追溯到哪些来源卡片。

模型规划失败时，系统使用保守 fallback，从已有概念及“定义与边界、组成要素、因果机制、应用场景、掌握标准”等结构生成候选，并明确将 `coverageCheck.sufficient` 设为 `false`，不把降级结构伪装成已充分论证。

### 7.2 评估和调整

学习步骤完成时，系统可以结合近期用户原文、卡片内容和任务概念生成评估结果。结果包含分数、通过状态、反馈和证据；路径调整记录包含原因、来源和建议变更。

`path-adjustment-engine.ts` 负责将评估和画像信号转化为路径建议。系统不会用一个无法追溯的分数直接改变全部学习计划。

### 7.3 推送建议

`push-suggestion-engine.ts` 从卡片关系、路径、画像、能力、缺口和学习事件中生成候选，再进行规则与 AI 审核。建议包含目标对象、原因、置信度和推荐格式，最终在 Learn 中呈现。

### 7.4 评估结果的证据约束

评估对象不是一个孤立分数。有效结果需要同时具备：

```text
target + assessmentType + rubric + userAnswer
       -> score + passed + feedback + evidence[]
```

其中 `evidence[]` 是更新 mastery 的必要条件。用户可以多次作答，每次生成新的 Attempt，不覆盖历史。系统由此可以区分“第一次不会、经过干预后掌握”和“数据库里只剩最终答案”。

路径调整只接受带 `reason` 和 `sourceObjectId` 的建议。调整历史保存修改前后的状态和触发来源，使路径变化可以解释、回放和测试。

卡片升级同样采用门槛而不是自由改类型。质量检查关注内容是否具备实质解释、定义、边界、例子、证据、关联、应用和必要性；未通过时保存 PromotionAttempt 和拒绝原因，用户补充后可以再次审核。

## 8. 资源生成与多模态实现

### 8.1 资源编排

资源生成器根据主题、难度、当前卡片和画像上下文，将不同类型分派给专项角色。状态机记录等待、生成、校验、保存、完成或失败，并生成 manifest。

支持的核心资源类型包括：

- `document`：讲解文档；
- `mindmap`：知识导图；
- `quiz`：结构化练习；
- `code`：代码案例；
- `diagram`：关系或流程图；
- `svg`：矢量图形；
- `video`：交互式 HTML 动画或视频资源。

前端分别使用 Markdown、Mermaid、结构化题目组件、视频组件等进行预览，并显示生成进度和问题状态。

### 8.2 文件交付

项目实现了 HyperFrames 资源渲染链路：

- HTML/Markdown 经 `html-to-docx` 输出 DOCX；
- HTML 经 Puppeteer 输出 PDF；
- 结构化页面描述经 PPTX 渲染桥输出 PPTX。

测试产物保存于 `test/artifacts/hyperframes-renderer/`，文件头和大小均经过自动检查。该能力使资源可以从页面预览进一步变成可提交、可授课、可分享的真实文件。

### 8.3 质量控制

资源测试不以“返回了字符串”为通过标准。不同类型设置结构检查，例如导图关键字、Mermaid 图类型、题目数量与字段、代码块数量、视频/动画标记等。真实模型结果记录 `qualityScore` 和 `qualityIssues`；SVG 等格式可在失败时使用确定性 fallback，并保留降级信息。

### 8.4 资源生成状态机

资源生成按单个类型维护状态，而不是使用一个全局 loading：

```text
pending -> generating -> completed
                      \-> failed
```

每种资源独立保存内容、文件名、MIME、错误和质量信息。这样即使视频生成失败，文档、题目和代码仍可以完成并进入 manifest，不需要整批回滚为“全部失败”。

资源 manifest 对实际产物进行最终对账：每个 item 至少包含类型、路径或内容引用、状态；清单数量必须与已生成产物一致。前端根据类型选择专门预览器，而不是把所有内容当 Markdown：

- Mermaid 资源进入隔离渲染；
- Quiz 按结构化题目展示；
- Video 支持视频 URL 或 HTML 动画；
- PDF 使用内嵌预览；
- DOCX/PPTX 提供文件交付；
- 代码和文档使用 Markdown/高亮渲染。

### 8.5 并行与失败隔离

不同资源类型之间没有数据依赖时可以并行生成；同一类型内部仍遵守规划、生成、校验和保存顺序。并行执行减少总等待时间，但每个角色拥有独立结果和错误边界。

失败处理遵循四个原则：

1. 单类型失败不抹掉其他已完成资源；
2. 格式不合法不能仅凭“有内容”标记成功；
3. 可确定性修复的格式允许降级，但必须留下 `qualityIssues`；
4. 文件引用不存在时状态必须是 failed，不能以空链接冒充完成。

## 9. RAG 与知识图谱

### 9.1 数据职责

PostgreSQL 中的 Card 和 Edge 是用户知识事实源，LightRAG 是可重建的检索增强层。卡片保存后生成稳定文档身份和内容哈希；内容未变化时跳过重复索引；索引失败时记录状态和错误，但不影响卡片本身读取。

每个 Vault 使用独立 LightRAG workspace，API 和测试同时验证跨 Vault 隔离。RAG 返回引用时映射回 cardId、title、path 等业务对象，避免页面只展示无法定位的向量片段。

#### 索引身份与一致性

每张卡片的 RAG 身份由业务边界确定：

```text
workspace  = <prefix>_<vaultId>
documentId = axiom:<vaultId>:card:<cardId>
contentHash = SHA-256(formattedCardContent)
```

同步前先读取 `RagDocumentIndex`：当 `contentHash` 相同且状态为 `indexed` 时直接跳过；内容变化时先删除旧索引，再写入新文档并轮询处理状态。数据库记录：

- provider；
- vaultId/cardId；
- workspace/documentId；
- contentHash；
- trackId；
- status；
- lastError；
- indexedAt/lastSyncedAt。

其状态语义为：

```text
pending | indexing | indexed | failed | disabled
```

如果未配置 LightRAG，系统写入 `disabled` 和原因；如果远端处理失败，写入 `failed` 和 `lastError`。无论哪种情况，Card 仍保存在 PostgreSQL 中。这就是“事实源与派生索引分离”的工程含义。

#### 删除与忙状态处理

LightRAG 删除旧文档可能返回 `deletion_started`、`not_allowed` 或 `busy`。客户端不会立即假设删除成功，而是检查 pipeline health，等待 active、scanning、destructive busy 和 pending enqueue 全部归零；超过超时则显式失败。这避免在索引仍忙时覆盖同一业务文档产生竞态。

### 9.2 图谱关系

显式关系来自用户链接、课程结构和已确认业务操作；AI 或 RAG 发现的隐含关系先作为候选呈现，用户确认后再进入 Edge。Galaxy 根据 Card、Cluster、Edge 和路径信息生成不同观察视角。

## 10. 安全、可信与稳定性

### 10.1 防幻觉

项目通过组合机制降低幻觉风险：

- 课程资料与个人知识 RAG；
- 回答引用映射到真实业务对象；
- 事实、公式、数字、代码和链接的检查入口；
- 资源类型结构校验和质量问题；
- 画像、推荐和评估保留证据与置信度；
- AI 候选关系不直接写入事实图谱。

这些机制用于降低风险和暴露不确定性，项目不作“生成内容绝对无事实错误”的不现实承诺。

#### 断言级检查

`FactualCheckGuardrail` 从生成内容中提取高风险断言，包括公式、具体数字/日期、代码和 URL，并返回：

```text
verified | unverified | suspicious
```

当前实现对少量已知公式提供基础校验；代码和 URL 等无法在本地可靠证明的内容标记为 `unverified`，要求添加来源或人工复核。这里的设计重点不是用有限规则假装完成通用事实核查，而是将“不确定”变成结构化质量信号。

#### 内容安全与提示注入

`ContentSafetyGuardrail` 使用两段式策略：先做敏感模式快速筛选，再结合句子上下文排除合法学术讨论与明确否定表达。输入清理同时识别常见的忽略上文、覆盖系统指令等提示注入模式。

当前安全词库和启发式规则属于基础防线，不能替代成熟内容安全服务。正式部署应接入更完整的语义审核并持续维护教育场景白名单。

### 10.2 数据边界

- Better Auth 管理身份和会话；
- API 按 User/Vault 校验所有权；
- 数据库约束保护唯一性和关系完整性；
- 高风险 Agent 操作需要确认；
- Agent 工具调用产生审计记录；
- 测试产物写入前进行凭据脱敏。

### 10.3 稳定性

- 对话采用流式呈现；
- 资源生成展示阶段状态和错误；
- RAG 索引具有内容哈希幂等；
- Redis/BullMQ 提供后台任务基础；
- Agent 具有状态机、预算、超时和失败记录；
- 页面具有错误边界和空状态。

当前 `document.import` 的通用 BullMQ worker 分支尚未接通，文档导入主流程由已实现的 Learning 服务/API 承载；此处不表述为已完成的统一异步导入队列。

### 10.4 派生层不得反向修改事实源

项目把 UI ReadModel、RAG、推荐、画像展示和资源清单视为派生层，并建立以下不变量：

- 打开 Dashboard/Cognition 不得新增或修改 Card、Path；
- RAG 索引失败不得导致 Card 保存失败或删除；
- Recommendation 只能返回建议，不能直接创建 Card/Edge/Path；
- 隐含关系只有确认后才能进入 Edge；
- UI 模式切换和布局变化只能修改 Zustand 状态，不能改变权限与内容；
- 派生统计必须能够指回源对象和时间。

这些不变量在 acceptance、API、Sidecar 和 UI-state 测试中分别检查。其目的在于防止 AI 系统常见的“展示计算反过来污染业务事实”。

## 15. 核心设计方案深解

这一章集中说明 AXIOM Space 最具辨识度的设计。这里介绍的不是某个框架本身，而是团队如何把通用大模型、Agent、RAG 和知识图谱组织成一套掌握学习系统。

### 15.1 AXIOM Agent Harness：把生成能力牵引到学习结果

`Harness` 的核心含义不是“多放几个 Agent”，而是把大模型原本发散的生成能力牵引到确定的学习对象、工具边界和结果落点。

AXIOM Harness 在七个环节对模型行为进行牵引：

| 牵引环节 | 具体设计 | 用户可见证据 |
|---|---|---|
| 入口牵引 | 用户从 Learn 步骤、Forge 卡片或 Galaxy 节点进入，不从无上下文空白对话开始 | Current Focus 显示任务、步骤和卡片 |
| 上下文牵引 | 装配 Vault、路径、卡片、会话、画像、RAG 与相关节点 | 回答围绕当前学习对象 |
| 角色牵引 | 前台 Agent 教学，后台 Agent 取证，专项角色处理资源、路径和评估 | 用户只面对一个导师，后台变化进入 Cognition |
| 工具牵引 | 模型必须通过卡片、路径、资源、评估和图谱工具改变系统状态 | 保存、生成、建立链接、完成步骤都有状态 |
| 证据牵引 | 对话被转化为摘要、薄弱点、掌握证据和画像观察 | AI 观察记录与知识缺口 |
| 反馈牵引 | 画像和评估结果反向影响下一轮解释、路径和推荐 | Learn 和 Forge 行为随证据变化 |
| 知识牵引 | 用户确认的卡片进入 RAG 与图谱，成为长期上下文 | RAG 状态、引用、节点与关系边 |

完整链路是：

```text
用户意图 / Learn 当前步骤 / Forge 当前卡片
    -> Harness 装配任务上下文、学习证据和知识召回
    -> Agent1 前台解释、追问、调用工具
    -> 用户输出回答、卡片或资源需求
    -> Agent2 后台提取证据、更新画像、识别薄弱点
    -> 状态写入 Cognition / Dashboard / Learn / Galaxy
    -> Guide、Assess、Forge 读取新状态
    -> 下一轮教学、路径和资源被重新约束
```

这条链路形成一个重要区别：普通聊天系统的结果是“一段回复”，AXIOM Harness 的结果可以是 Message、Card、Edge、AssessmentResult、ProfileObservation、PathAdjustment 或 ResourceManifest。这些对象可持久化、可追踪，也可以成为下一轮学习输入。

#### 双 Agent 不等于两个聊天框

Agent1 负责实时教学和工具协调，必须优先保证交互延迟与连续性。Agent2 在回复完成后分析用户真实表达，提取画像、技能、卡片和观察建议。后台分析不会向用户再开一个对话窗口，而是把结果写入认知观察和业务状态。

这样设计是因为教学对话和长期画像存在不同时间尺度：前者要求立即回应，后者要求综合证据、保留历史并谨慎更新。把它们塞进同一个提示词，会使实时回复过重，也容易让画像被单轮上下文污染。

#### 子 Agent 按需隔离

系统定义 Oracle、Profile、Forge、Guide、Assess 五类教育角色：

| 角色 | 职责 | 主要边界 |
|---|---|---|
| Oracle | 教学协调、苏格拉底式对话和结果汇总 | 读取知识、解释、追问和评估理解 |
| Profile | 画像构建和更新 | 读取证据、更新画像和技能 |
| Forge | 多类型资源与卡片加工 | 资源、卡片和文件工具 |
| Guide | 路径规划和资源推荐 | 路径、图谱和推荐工具 |
| Assess | 掌握评估和薄弱点分析 | 费曼测试、题目和质量检查 |

主循环不会为了展示“多 Agent”而每轮强制启动全部角色。只有需要隔离上下文、并行执行或使用特定工具集时，才通过子 Agent 生命周期启动对应角色。子 Agent 默认禁止 Shell、删除、合并、重建索引等高风险工具。

### 15.2 清晰、准确、必要：知识进入主结构的准入规则

AXIOM Space 不把模型生成的所有节点和关系都写入知识库。团队为知识对象建立了三个连续判断：

```text
清晰：知道它在当前知识库里怎么用。
准确：它的说法和依据一致。
必要：删掉它会损害理解。
```

判断顺序不能颠倒：

```text
它说的是什么？
    -> 它说得对吗？
        -> 它必须存在吗？
```

不清晰就无法判断准确，不准确就没有保留价值，不必要就不应进入主结构。

#### 清晰：对象、用法、边界和位置

一个知识对象必须回答：它指什么、不指什么、如何使用、与相邻概念的边界是什么。名称短并不等于清晰。例如“性能”可能指 CPU、网络、数据库或学习表现；“CPU 流水线吞吐率”才具有稳定用法。

清晰还要求区分归属与连接：

```text
归属说明它是什么。
连接说明它影响什么。
```

例如 TLB 主要归属内存管理，同时可以连接 Cache 局部性，但连接不能取代归属。系统因此同时维护 Cluster 归属和 Edge 关系，而不是用一张无类型网络表达一切。

#### 准确：结论必须有依据

节点定义、所在位置和关系类型必须由课程资料、领域定义、用户表达或既有卡片支持。没有依据的 AI 判断只能成为候选，不能作为事实写入。

系统在数据上保留来源卡、文档分块、用户消息、评估证据、关系来源和置信度，使“准确”不只是一次模型自评。

#### 必要：执行删除测试

对候选节点或关系提出一个简单问题：

```text
如果删掉它，会丢掉什么？
```

如果答案只是“可能有点相关”，它不应进入主结构；只有会丢掉概念边界、前置条件、证据链或学习步骤，才具备必要性。

这套标准落实到卡片升级、文档必要条件建模、图谱关系确认和推送审核。用户正文保留定义、例子、理解和疑问；系统元数据保存归属理由、证据、置信度和确认状态，既保证可审计，又不让技术判断干扰日常阅读。

### 15.3 Galaxy：同一知识图谱的多种认知投影

Galaxy 不是装饰性 3D 背景，而是一套面向学习问题的知识观察系统。底层仍是同一组 Card、Edge、Cluster 和 Path 数据，不同布局通过投影规则回答不同问题：

| 视角 | 投影规则 | 回答的问题 |
|---|---|---|
| 星系 | 按星团、类型和整体空间展示 | 我整体学了什么 |
| 平面 | 按真实语义边展开，强连接更靠近 | 哪些知识点真的有关联 |
| 同心 | 以选中节点为中心展开一跳、二跳邻域 | 这个概念周围有什么 |
| 任务流 | 把 LearningPathStep 按顺序投影，旁支显示资料 | 下一步应该怎么走 |
| 证据 | 文献、灵感、永久卡分层，突出支撑边 | 这个知识凭什么可信 |
| 地形/掌握度 | 用 mastery、路径状态影响高度和亮度 | 哪些知识已经掌握 |

布局不是换皮肤。没有真实路径时不能伪造任务流；没有真实时间字段时不能用随机值冒充时间线；证据视图必须区分观点和来源；地形必须接入真实掌握数据。

#### Card / Edge / Cluster 契约

- `Card` 统一承载 literature、fleeting、permanent 三种知识生命周期；
- `Edge` 表达 wikilink、related、prerequisite、derived、supports、contradicts 等关系语义；
- `Cluster` 表达知识域归属、颜色和排序。

WikiLink 保存时只在当前 Vault 解析真实目标卡片，并重建当前卡片的 WikiLink 出边；不存在的目标保留为待处理链接，不创建伪造 Edge。

#### 显式关系与系统发现关系分层

用户 WikiLink、路径关系、资料导入关系属于显式业务事实。LightRAG 或 Agent 发现的语义相关只能先进入候选层，以淡色辅助边或推荐关联展示，经过确认后才写入正式 Edge。

因此 Galaxy 不只是展示结果，还反向服务 Cognition 和 Learn：边密度用于判断关联能力，孤立节点形成知识缺口，缺少永久卡的星团表示尚未沉淀，前置关系进入学习顺序，证据边进入可信性判断。

### 15.4 RAG：不是“接向量库”，而是设计可信知识回路

AXIOM 的 RAG 设计包含四个层次：事实源、派生索引、引用回映射和学习反馈。

```text
PostgreSQL Card / Edge / Cluster
    -> 格式化为带 AXIOM 元数据的文档
    -> LightRAG 切块、向量、实体和关系
    -> 检索结果解析稳定文档身份
    -> 回映射到当前 Vault 的真实 Card
    -> 进入 Agent、Forge、Learn、Galaxy 和 Cognition
```

进入 RAG 的卡片不仅包含正文，还包含卡片 ID、类型、路径、星团和标签。稳定身份 `axiom:<vaultId>:card:<cardId>` 同时解决多 Vault 隔离和引用追踪。

相关卡片采用三段式策略：

```text
LightRAG mix 检索语义相关
    -> 不足时补显式 Edge
        -> 仍不足时补同 Cluster 上下文
```

三种信号各有职责：RAG 发现语义相近，Edge 尊重用户明确关系，Cluster 保留主题归属。

#### 为什么不把聊天流水全部写入 RAG

原始对话包含临时提问、错误理解、重复确认和未经用户确认的 AI 草稿。如果全部进入长期检索，系统会在未来召回噪音甚至用户已经纠正的误解。

AXIOM 采用：

```text
聊天进入会话记录
    -> Agent2 判断长期价值
    -> 用户确认或结构化为卡片/观察/记忆
    -> 再进入长期知识索引
```

这一边界体现了产品理念：知识不是自动堆积，而是经过理解、取证和确认后的结构。

### 15.5 资源推送：8 路查找器先找缺口，AI 再做审核

推送系统不是让大模型凭空回答“你接下来可以学什么”。`PushSuggestionEngine.findCandidates()` 先通过八类结构化查找器定位真实缺口：

| 查找器 | 发现的问题 | 主要算法信号 |
|---|---|---|
| WikiLink 缺失连接 | 正文已有链接但图谱无边 | WikiLink 解析 + existingEdges |
| 缺失卡片 | 引用了不存在的概念 | title 映射 + 去重集合 |
| 薄卡片 | 内容短、缺例子、连接度低 | 纯文本长度 + 关键词 + degree |
| 关键词相似 | 两卡高度相关但未连接 | 关键词交集 + Cluster + Edge 排除 |
| 路径缺口 | Step 无卡片或相邻步骤缺前置边 | Step/Card + prerequisite |
| 星团缺口 | 板块连接密度低或薄卡过多 | 内部边密度 + 薄卡计数 |
| 画像缺口 | 弱概念、缺前置、孤立节点 | Profile gaps + 目标细化 |
| 证据驱动下一步 | 已掌握概念仍缺边界或适用条件 | recentEvidence + boundary keywords |

这些查找器首先从知识库结构中得到候选，不依赖 LLM。随后 AI 只承担审核和表达优化：按置信度取候选，判断保留或丢弃，可调整标题、理由和置信度。来自明确证据链的 `displayLocked` 候选禁止 AI 改写核心理由；模型不可用时采用规则候选，并将置信度上限压低。

置信度只决定排序，不决定自动执行：

```text
85-100：强建议
65-84：普通建议
40-64：弱建议
0-39：默认不展示
```

即使高置信建议也需要用户接受。推送系统因此成为 AI 建议进入知识库之前的审核层：AI 可以发现缺口、提出连接、生成草稿和解释理由，但不能直接写入永久卡、批量改图谱或删除用户知识。

### 15.6 三层学习评估：从“听懂”到“掌握”

传统测验往往只回答“这道题得了多少分”。AXIOM 将学习效果拆成三个不同层次：

```text
第一层：理解层 -> 能否解释和迁移
第二层：沉淀层 -> 能否形成合格知识卡片
第三层：系统层 -> 整体知识结构是否健康
```

#### 第一层：费曼对话式评估

评估嵌入自然教学对话，要求学生用自己的语言复述、举新例子、说明区别、指出条件变化的影响。判断递进关系为：

| 学生表现 | 证据强度 |
|---|---|
| 能复述定义 | 初步理解 |
| 能举出新例子 | 基本掌握 |
| 能指出边界和反例 | 深入理解 |
| 能迁移到新场景 | 强掌握证据 |
| 能写成自己的卡片 | 可以进入沉淀审核 |

`feynman-card-capture.ts` 保留用户原文并将有效解释关联到卡片。系统还提供选择题、代码挑战、应用任务和辩论等不同验证入口，避免所有学科只用同一种题型。

#### 第二层：卡片升级质量审核

fleeting 卡升级为 permanent 时必须经过质量门槛。系统检查定义、边界、例子、证据、关系、应用和独立价值。失败返回明确缺失项并记录 PromotionAttempt，不是所有卡片都能一键成为永久知识。

这一层把“我觉得我懂了”变成“我能写出清晰、准确、必要的知识对象”。多次驳回还可以成为画像中的稳定卡点证据。

#### 第三层：系统学习健康度

系统层综合六维画像、Card 类型分布、图谱密度、星团覆盖、路径完成、测评通过和孤立节点等信号，判断知识结构是否空心化。例如：大量 literature/fleeting 但很少 permanent，说明输入很多但尚未沉淀；节点多但边密度低，说明知识仍然碎片化。

当前后端 `buildLearningProfileContext()` 已经聚合这些数据并用于教学策略，完整的集中式“学习健康度”前端面板仍属于待完善能力。提交稿据此只声明系统层计算基础已经形成，不声明完整面板已经交付。

三层评估不是三个独立模块，而是一条数据流：

```text
对话理解证据
    -> 卡片沉淀与质量审核
        -> 图谱、画像、路径和测评的系统评估
            -> 调整教学、插入复习、推送资源
                -> 再次评估
```

### 15.7 从技术动作到页面证据

复杂技术只有在页面上形成可验证变化，才算进入产品。

| 技术动作 | 页面证据 | 对学习的作用 |
|---|---|---|
| Session/Card 绑定 | Forge Current Focus | 保证对话服务于明确对象 |
| SSE 流式执行 | 逐字回复和工具状态 | 让等待过程可理解 |
| LightRAG 注入 | 引用与相关卡片 | 证明回答使用个人知识 |
| 资源编排 | 类型进度、预览和文件 | 把生成转成可使用材料 |
| BackgroundAnalyzer | Cognition 观察与历史 | 证明画像来自真实学习证据 |
| Graph update | Galaxy 节点、边和证据链 | 证明学习结果形成结构 |
| Path adjustment | Learn 状态、评估和下一步 | 证明学习计划会随结果变化 |
| Query invalidation | 多页面同步刷新 | 证明不同模块共享同一事实状态 |

这也是 AXIOM Space 的综合技术判断：**数据库负责保存事实，Agent Harness 负责组织行为，RAG 负责增强召回，领域规则负责知识准入，页面负责把运行证据呈现给用户。**
