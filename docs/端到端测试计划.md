# AXIOM Space 端到端测试计划

> 版本：v2.0 · 日期：2026-05-25 · 维护者：morewhyhan
> 副标题：**以「学习真的发生」为最高验收标准。**
> 形态：自动化测试规约（Spec），每条命题可被 Playwright / Vitest / DB 断言直接验证，或明标 🧑 由人工执行。
>
> **赌注声明**：本文档中每一个命题的真伪，决定 AXIOM 能否在赛题答辩中证明其核心承诺。
> 一个含糊的命题 = 一个不可验证的承诺 = 一个无法兑现的赌注。
> **不允许在本文档中出现以下模糊词**：明显、显著、流畅、合理、大致、约、左右、可能、应该、可观察、实质性、不错、良好、流畅。
> 凡出现量词必给数字。凡出现"差异"必给阈值（编辑距离 / 计数 / 百分比）。凡出现时间必给单位。

## 用户视角的使用说明（替用户讲一遍）

> 这是 AXIOM 在用户嘴里的样子。测试要验证的，是这段话里**每一句**都真的发生。

**我打开 AXIOM。**

**我告诉它我想学什么。** 不用填表单，我就直接说话：「我想学操作系统」「我想搞懂量子力学的基础」「我导入了一份论文，帮我吃透它」。

**它先反过来问我。** 不是讲课，而是像个老朋友一样问我「你以前听过『进程』吗？能用一句话说说吗？」我答得磕磕巴巴也没关系，它顺着我说的继续追问，几轮之后，它就摸清了我会什么、不会什么。

**它给我读的东西。** 根据它摸清的我的水平，它**生成或找到**针对我的学习资料——一份大纲清晰的讲解文档、一张思维导图、一套练习题、一份能直接拿去汇报的 PPT、一个我能套用的应用案例。它不会给我一本厚书让我从头读，而是直接把我**该看的那部分**摆在我面前。

**我开始读，开始想。** 读着读着我冒出疑问、灵感、半成型的想法，我把它们随手记下来——不是用什么"闪念盒"按钮，就是写一张卡，颜色是青色，自动落到我的知识星系里。

**它帮我把零散想法磨成清晰概念。** 我说「这张卡我想做扎实」，它问我「你怎么用自己的话定义它？能不能举个例子？它和哪个概念有关？什么时候你会用到它？」我自己回答这四个问题——它**不替我答**，只是问。等四个回答都写出来了，这张卡就从青色变成紫色，进入我的永久知识体系。

**我看到我的知识在长大。** 切换到星系视图，我看到一个 3D 的宇宙，每一颗紫色星是我亲手写的概念，星和星之间有连线——那是我自己用 `[[ ]]` 标出来的关联。星越多，连得越密，我越清楚自己懂了什么。

**它越来越懂我。** 每写一张新卡，它就读一次我的知识体系，悄悄更新对我的理解。下次我说"我想学内存管理"，它不会从零开始讲——它会说「你之前学过『进程』，内存管理和它的关系是这样的……」。**越学越快，越学越深。**

**我可以让它生成 PPT 拿去给老师汇报。** 我可以让它出题考我。我可以让它做一份对比矩阵帮我厘清几个相似概念。我可以把我整个知识库导出成 Obsidian 格式，永远属于我。

**我关上电脑。** 第二天，朋友问我「什么是进程？」——我能用我自己的话讲清楚，能举一个 AXIOM 里没出现过的例子，能说清它和线程的区别。

**这时候，AXIOM 才算真的工作了。**

## 0. 测试哲学

> **凡能说清楚的，便能验证；凡不能验证的，便不该写进测试计划。**
> —— 仿《逻辑哲学论》7

本计划遵循三条原则：

1. **每一条验收标准都是命题** —— 可被自动化判定为「真」或「假」，不留主观空间。
2. **每一条命题都从用户视角写出** —— 主语是「我」（用户），不是「系统」。
3. **不可观察的事情，不写进测试** —— 「体验流畅」「界面美观」不是命题，「LCP < 2.5s」「FPS ≥ 30」「DOM 节点 `data-status='active'`」才是。

> ✅ 合规命题：「我点击 Galaxy 中的节点后，3 秒内右侧编辑器加载出该卡片的内容。」
> ❌ 非合规命题：「Galaxy 交互体验流畅。」

---

### 0.1 三种断言（标签体系）

每一条命题前打标签，标明它由谁、用什么方式验证：

| 标签 | 名称 | 验证方式 | 谁来做 |
|------|------|--------|--------|
| 🤖 | **AUTO** | Playwright / Vitest / Prisma 查询 / DOM 断言 / 字符串 grep | 自动化测试套件 |
| 🧑 | **MANUAL** | 人工观察、录音、截图、人工标注 | 项目核心成员（morewhyhan + 队友）于答辩前 7 日内一次性执行 |
| 📊 | **STAT** | 跨用户 / 跨周期数据采集后统计判定 | 自动化采集 + 人工解读，作为里程碑指标 |

**未打标签的命题，默认 🤖 AUTO**。

**DoD 拆分**：
- 🤖 通过率必须 ≥ 95%（无 Blocker）
- 🧑 通过率必须 ≥ 90%，证据归档于 `docs/manual-verification/YYYY-MM-DD/`
- 📊 不阻塞答辩，作为产品成熟度参考

### 0.1+ 容忍度执行协议

> 本文档中多处出现「容忍度：连续 N 次执行中 ≥ M 次通过即可」，特指以下执行方式：

| 参数 | 定义 |
|------|------|
| **一次执行** | 同一用户、同一 Vault、同一测试命题，发送**相同输入**触发 Oracle/API，记录一次输出 |
| **连续 N 次** | N 次执行在同一测试 session 内完成，每次使用**新对话**（`/new` 或新 session），间隔 ≥ 10 秒（避免 LLM 缓存命中） |
| **通过判定** | N 次中 M 次通过 → 整条命题判 PASS。不要求连续通过，只要求通过次数达标 |
| **适用范围** | 仅适用于标注了「容忍度」的命题（均涉及 LLM 输出内容判定）。纯功能命题（DOM 查询、数据库断言、HTTP 状态码）**不适用**容忍度 |

---

### 0.2 命题语言规则（维特根斯坦契约）

每条命题必须满足以下五条：

| # | 规则 | 反例 | 正例 |
|---|------|------|------|
| 1 | **主语明确** | 「系统流畅」 | 「我点击 X 后 3s 内 Y 出现」 |
| 2 | **谓语可观察** | 「体验良好」 | 「DOM 中存在 `[data-status='active']` 元素」 |
| 3 | **量词给数字** | 「响应较快」 | 「响应 ≤ 800ms」 |
| 4 | **差异给阈值** | 「显著不同」 | 「编辑距离 / max(len) ≥ 0.5」 |
| 5 | **时间给单位** | 「很快」 | 「≤ 500ms」 |

**禁用词清单**（在本文档任何命题中出现即视为该命题不合规）：

```
明显 / 显著 / 流畅 / 合理 / 大致 / 约 / 左右
可能 / 应该 / 可观察 / 实质性 / 不错 / 良好 / 通常
有时 / 经常 / 一般 / 适当 / 充分 / 必要 / 基本
```

**例外**：如果某条命题确实只能用人工判定（如「Oracle 风格符合 Dijkstra」），必须打 🧑 标签，并附上**抽样规则**（抽几条、由谁判、判的依据）。

---

### 0.3 边界声明（凡不能验的，不许写）

本文档**不**承诺以下事项的可验证性：

| 不验证的事项 | 理由 |
|------------|------|
| 「用户主观感到学习更高效」 | 这是 30 天后人工访谈的事，超出 E2E 测试边界 |
| 「AI 生成内容质量很高」 | 「质量」未定义；我们只验证字数、结构、占位符存在与否 |
| 「界面美观」 | 美学不可断言 |
| 「Oracle 真的懂这个学科」 | LLM 输出的学科正确性不在本计划测试范围（属于 LLM 供应商责任） |
| 「LLM 输出的具体措辞/句式」 | LLM 具有非确定性；同一 prompt 在不同 session / 模型 / 温度下输出不同。本计划对 LLM 输出内容的测试采用**容忍度机制**：允许同一命题在连续 N 次（N≥3）执行中有 ≤1 次不通过（即通过率 ≥ 67% 即判为 PASS）。纯功能调用（是否调了 LLM、参数对不对、输出非空）不受此豁免 |

凡不在本文档命题中出现的事项，**视为我们不为其负责**。

### 0.4 弹性原则（与精确性同等重要）

> **精确性回答"什么算对"。弹性回答"对错之间的空间里，用户怎么活下去"。**
>
> 一条命题如果"通过 = 1，不通过 = 0"，它就是钢条。
> 一个用户实际遇到的世界，永远不是 1 或 0——他可能写一半就分心、网突然慢了、AI 那天说了句奇怪的话、他想跳过引导自己来。
> 弹性 = **当我没按系统预期的方式使用，系统不崩、不催、不删、不堵我的路**。

本计划在每一条硬命题旁边，保留一条**弹性命题**对应——它规定的不是"成功的样子"，而是"**不完美的样子里，用户仍然舒服的样子**"。

弹性命题汇总在 **L8 层**。它们的优先级**不低于** L0-L7 的硬命题：钢条没断 + 弹簧没断 = 产品成立。

具体的弹性维度：

| 弹性维度 | 含义 | 反例（系统当前可能犯的错） |
|---------|------|--------|
| **慢/失败弹性** | LLM 卡了、断网了，系统不白屏不报红 | 转圈 30 秒后直接报错 |
| **节奏弹性** | 用户慢思考、长停顿，系统不催不打断 | 30 秒不动就弹"你还在吗" |
| **不完美输入弹性** | 用户少写、错写、用同义词，系统不一刀切拒绝 | 校验失败就锁住，必须改对才放行 |
| **不强制路径弹性** | 用户可绕开任何一条主路径不被惩罚 | 必须先建卡才能对话；必须升级永久才能继续 |
| **数据量弹性** | 0 卡和 10000 卡都不崩，性能不抖 | 永久卡 = 0 时雷达图崩溃 |
| **撤销弹性** | 删错、点错、升级错了，都能后悔 | 删除即永久删 |
| **打扰弹性** | 推送、提示、通知有上限，可静音 | 每次启动弹 modal "你今天还没学" |
| **失败优雅** | API 错误是用户能理解的文案 | 屏幕显示 "Error 500: ECONNREFUSED at /handler.ts:42" |

## 1. 产品本质（测试前必读）

### 1.1 一句话定义

**AXIOM 是一个让我通过『自己写卡片』来真正学会一个领域的 AI 系统。**

### 1.2 三盒闭环

```
闪念盒 (Fleeting)  →  文献盒 (Literature)  →  永久盒 (Permanent)
   原始想法              AI 生成的资料            标准化概念卡片
        ↑                                              ↓
        └────── AI 读永久盒 → 更新画像 → 推荐更精准 ──┘
```

### 1.3 四个 Agent

| Agent | 角色 | 一句话职责 |
|-------|------|----------|
| Oracle（巨匠） | 学科奠基人 | 苏格拉底式追问，探我的知识边界 |
| Literature（文献） | 资料生成 | 根据我的画像生成 5+ 种学习资源 |
| Forge（锻造） | 卡片构建 | 引导我把闪念打磨成永久卡 |
| Guide（向导） | 迭代优化 | 读我的永久盒，更新我的画像 |

### 1.4 卡片四要素

**一张合格的永久卡片 = 定义 + 举例 + 关联 + 应用。缺一不可。**

### 1.5 费曼铁律

> **「我输出的才是我会的。」**
> 卡片必须由我自己写。AI 可以引导、追问、纠错，但**不能代我写卡片**。

### 1.6 8 维认知升级

1. 复杂情境整合 · 2. 多方立场分析 · 3. 洞察本质与表象 · 4. 逻辑严密推理
5. 认知偏差觉察 · 6. 反直觉思维 · 7. 价值冲突选择 · 8. 创造性输出

### 1.6+ 概念三原则（与卡片四要素同等位阶的硬性约束）

> **来源**：`2-感觉文档.md` ——「不是以"书"为单位，不是以"章节"为单位，而是以**清晰、准确、必要的概念**为单位。」
>
> 三原则适用于 **所有**进入永久盒的概念，**以及**从文献中提取的灵感节点。
> 三原则**不是建议**，是**准入门槛**——不满足，不准升级；不满足，不准提取。

| 原则 | 哲学定义 | 操作定义（可验证） |
|------|---------|------|
| **清晰** (Clear) | 概念有**确定的边界**——你能判定一个给定例子是属于它还是不属于它 | 1. 「定义」section 字数 ∈ [10, 150]；2. 「定义」含主谓宾完整句（结尾标点 `。` `.` `！` `!` 之一）；3. 「定义」不以概念名本身作为定义主体（拒绝循环定义——但概念名出现在从句/修饰语/举例中不违规，见 L2.6 校验 B3 判定规则） |
| **准确** (Accurate) | 与该学科**主流共识**一致；无误用、无歧义 | 1. 文献提取场景：概念名在原文出现次数 ≥ 1；2. 提取的定义与原文相关段落的 Levenshtein 相似度 ≥ 0.5；3. 升级永久卡场景：「关联」section 含至少 1 个有效 `[[wikilink]]`（被链接的卡存在或属于待建概念） |
| **必要** (Necessary) | 移除这个概念，**周围的概念无法被理解**——是承重节点，不是装饰术语 | 1. 文献提取场景：概念在原文出现次数 ≥ 2（非偶然提及）；2. 在 concept-map.json 中被其他概念引用 ≥ 1 次，或在 cluster 内 wikilink 入度+出度 ≥ 1（不孤立） |

#### 三原则的边界声明

- **清晰**可由前端校验**完全自动化**判定（字数 + 句式 + 循环检测）。
- **准确**只能**部分自动化**——子串匹配和 Levenshtein 相似度是代理指标；真正的"学科准确"需要 LLM 或人工，属于 §0.3 不承诺范围。
- **必要**通过引用频次 / 图谱连接判定——这是**结构准确**而非**语义准确**。

#### 反例（什么样的概念会被拒绝）

| 反例 | 违反哪条 | 原因 |
|------|---------|------|
| 「进程是进程」 | 清晰 | 循环定义 |
| 「这是一个很重要的东西」 | 清晰 | 无主语、无边界 |
| 一份文献只在脚注里提了一次的术语 | 必要 | 出现次数 < 2 |
| 用 LLM 凭空编出来的、原文没出现过的概念 | 准确 | 原文出现次数 = 0 |
| 「关联」section 留空的卡 | 必要 | 没有承重连接 |

### 1.7 三个 Skill

| Skill | 触发词 | 产物 |
|-------|--------|------|
| axiom-learning | 学 / 教 / 解释 / 什么是 / 帮我学 / 导入了文献 | profile.json + concept-map.json + 空模板卡（含 learningOrder） + 文献 |
| axiom-forge | 生成资源 / 出题 / 生成导图 / 应用案例 | **5 种严肃实现**：document / mindmap / quiz / case-study / ppt（其他类型可作为规划占位） |
| axiom-ppt | 生成 PPT / 演示文稿 | 真实 `.pptx` 文件入文献盒 |

> ⚠️ `3-硬性约束文档.md` 中曾标注「❌ PPT 自动生成不做」，但产品实际通过 axiom-ppt skill 实现——**以代码与本计划为准，待更新硬性约束文档**。

### 1.8 三盒在 UI 上的体现方式（重要）

**三盒不是三个按钮，不需要点击式入口。** 三盒通过以下方式被用户感知：

| 三盒 | UI 呈现 |
|------|---------|
| 闪念盒（Fleeting） | Galaxy 中**青色**节点 + Galaxy 控制面板的「类型过滤器」中 fleeting 项 |
| 文献盒（Literature） | Galaxy 中**粉色**节点 + Learn 模式的文献列表 |
| 永久盒（Permanent） | Galaxy 中**紫色**节点 + Cognition 模式的知识结构树 |

**测试时不应去找「闪念盒」「文献盒」「永久盒」三个按钮**——它们不存在。要验的是：颜色编码正确、类型过滤器可切换、数据流转后颜色随之变化。

## L0.3 — 越学越快（迭代加速）

### L0.3.0 — 公理与机制

> **L0.3 用户期望**（用户原话翻译为命题）：
> 我学的概念越多，AXIOM 对我越了解。当它对我的了解（画像精准度）达到某个阈值后，它**不再等我问**——它**主动**告诉我「下一个该学什么」。
> 当我问一个我没学过的新概念时，它**从我已经学过的概念里**找出关联点，让我**用已知去推演未知**。
> 学习的加速，**不是**来自系统跑得快，**而是**来自我不必从零开始——新概念能"接"在我已熟悉的某个旧概念上。

> **不变公理**（这是产品哲学的基石）：
> **新概念的理解，只能通过已知概念来建立。**
> 因此「越学越快」的本质 = `已知锚点 × 关联推演`。
>
> 如果系统**不能**读到我已知的概念，「越学越快」无法成立。
> 如果系统**不**主动用我已知的概念来介绍新概念，「越学越快」无法成立。
> 如果系统在我已知概念为空时**伪造**关联，则违反公理，属于"幻觉"，需被 L1.5 拦截。

### L0.3 — 边界声明

L0.3 验证的是「**已知锚点 + 关联推演机制**在产品中真实运转」。
L0.3 **不**验证以下三件事（它们属于其他层级）：

| L0.3 不验证 | 属于哪里 |
|------------|---------|
| 用户主观感受到学习变快 | §4 8 维认知专题（30 天后） + L0.4 灵魂检验 |
| LLM 生成内容学科正确 | §0.3 边界声明（LLM 供应商责任） |
| 推荐的概念是否"真的"是用户该学的 | 产品迭代，不在 E2E 测试范围 |

L0.3 通过 = 以下 6 组命题（a-f）**全部**为真。

## L1.4 — 个性化学习路径

> ⚠️ **实现现状**：当前实现是「**纯数据库聚合 + 简单算术**」，不调用 LLM、不读 user-profile、不读 cognition、不基于学习偏好。一个 cluster = 一条路径，步骤按 `createdAt` 升序排列。本节命题分三类：
> - **L1.4.a 当前实现可验命题**——必须为真
> - **L1.4.b UI 行为命题**——必须为真
> - **L1.4.c 赛题承诺 vs 实现差距**——这些命题**当前为假**，是已知缺口，列入测试报告 finding

### L1.4.a — 当前实现可验命题

| 命题 |
|------|
| `GET /api/learning/paths?topic=操作系统` 在 500ms 内返回（本地 SQL，不调 LLM）。 |
| 不传 topic 时，返回当前 vault **所有 cluster** 对应的路径列表。 |
| 传 topic 时，按 `cluster.name.includes(topic.toLowerCase())` 子串匹配过滤。 |
| 每条路径 `id` = clusterId，`name` = `<cluster名>学习路径`，`color` = cluster 色。 |
| 路径的 `steps[]` 顺序 = cluster 内卡片按 `createdAt asc` 排列。 |
| 步骤 `status` 映射：`permanent → done` / `fleeting → active` / 其他 → `pending`。 |
| 步骤 `mastery`：`permanent` = `min(100, 70 + content.length/20)`，`fleeting` = `min(60, content.length/5)`，其他 = 0。 |
| 路径 `progress` = `Math.round(doneCount / totalCount * 100)`。 |
| 路径 `difficulty`：permanent 占比 > 60% → `进阶`；卡数 > 5 → `综合`；否则 `基础`。 |
| 当过滤后 cluster ≥ 2 时，响应额外含一条 `id: 'cross-domain'` 路径，硬编码 8 步上限，按 `createdAt desc` 取最新 8 张跨簇卡。 |
| 跨域路径每步 `desc` 字段形如 `来自「<clusterName>」`。 |
| `activePath` = 首个含 active 步骤的路径 id，否则首路径 id，无路径时为 `null`。 |
| `activeStep` = 该路径中首个 active 步骤的 0-based 索引。 |
| 空 vault 返回 `{ paths: [], activePath: null, activeStep: 0 }`。 |
| 删除一张卡后重新拉取，对应步骤消失；新建一张卡后重新拉取，对应 cluster 路径新增一步。 |

### L1.4.c — ⚠️ 赛题承诺 vs 实现差距

> 这些命题当前为假，是**已知缺口**。"修好"的定义见每行右列。

| 命题（当前为假） | 来源 | 修好的硬定义（必含数字/算子） |
|----------------|------|---------|
| 路径生成读取 user-profile，依据 `level / learningStyle` 调整步骤顺序与难度。 | 赛题第 30 行 + 灵感构建 | 同一 user 切换 level（入门→进阶）后重拉路径，`steps` 顺序的 Levenshtein 编辑距离 ≥ `steps.length × 0.3`；`difficulty` 字符串改变。 |
| 路径**跳过**或**降权**我已 `mastered=true` 的概念。 | 5-学习方法文档 | 把 5 张卡标记 `mastered=true` 后重拉，这 5 个步骤的 `status='reviewed'`，或它们的 index ≥ `steps.length × 0.7`（落到后 30%）。 |
| `mastery` 反映**真实掌握度**，而非 `content.length` 近似。 | 1-基本灵感构建 | mastery 计算公式中至少含 1 个非长度信号：`quiz 正确率` ∨ `Forge 评分` ∨ `复习间隔`，且该信号权重 ≥ 0.3。 |
| 跨域路径基于**双向链接 / 概念关联**，而非按时间倒序硬取 8 条。 | 0-A3 赛题 | 跨域路径中相邻两步在 `edges` 表存在直接边的比例 ≥ 50%。 |
| 拖入 `.md` / `.txt` 文件**实际**生成路径或创建卡片。 | 我的设计意图 | 拖入文件后 5 秒内，`/api/vault/card` POST 请求发生 ≥ 1 次，返回 `type=literature` 卡，其 `content` 字段含原文件文本（子串匹配，命中率 100%）。 |
| 路径生成有 Agent（Guide / Literature）参与，不是纯 SQL。 | 1-基本灵感构建 | `GET /api/learning/paths` 请求 trace 中含 ≥ 1 次 LLM 调用记录（log 表 `audit_log` 可查）。 |
| 路径动态更新画像、画像反向优化路径（越学越快）。 | 2-感觉文档 | 学完 5 张永久卡后重拉路径，与第一次返回的 `steps` 顺序 Levenshtein 编辑距离 ≥ `steps.length × 0.2`。 |
| `learningOrder` 由 LLM 推理依赖关系生成（而非按字数）。 | L1.9 关联 | concept-map.json 含 `dependencies` 字段；按 dependencies 做拓扑排序后的顺序与 `learningOrder` 升序**完全一致**（0 个 inversion）。 |

> **结论**：当前路径规划是「**路径展示**」而非「**路径规划**」。赛题答辩前需要做的选择：（a）补齐上述差距；（b）调整文档措辞与之对齐。

## L1.9 — 文献驱动的灵感节点生成与学习排序 ★ 新增

> **用户场景**：「我刚下载了一份 50 页的论文 / 一本电子书 / 一个 Markdown 笔记。我不想从头读到尾。我希望 AI 替我看一遍，告诉我里面有哪些核心概念、它们之间什么顺序学最合适、然后给我准备好一组卡片，让我直接开始学。」
>
> 这是 axiom-learning Phase 1 的**完整端到端验证**——从用户提供文献到形成可学习的有序卡片网络。

### L1.9.a — 文献导入入口

| 命题 |
|------|
| Forge Chat 中输入「帮我学习这份文献」并附加文件（`.md` / `.txt` / `.pdf`），系统触发 axiom-learning skill。 |
| Learn 模式左侧的文件拖入区，**接通后端**——`material` 状态传给 `handleGenerate`，落库为 literature 卡。（当前为已知缺口，见 L1.4.c） |
| 粘贴长文本（≥ 2000 字）到对话框，系统识别为"导入文献"意图而非普通对话。 |
| 支持的格式：`.md` / `.txt` / `.pdf`（PDF 通过提取文本层）；不支持时返回明确错误，不静默失败。 |

### L1.9.b — AI 提取核心概念（**三原则准入**）

> **铁律**：从文献中提取的概念，**必须**同时满足「清晰 / 准确 / 必要」三原则（见 §1.6+）。
> 不满足的概念**不**进入 concept-map.json；**不**生成 fleeting 模板卡。
> 提取数量浮动：满足三原则的概念有多少就提取多少，不为凑数稀释，也不为简洁砍掉硬骨。

#### L1.9.b.基础命题 🤖

| 命题 |
|------|
| 导入后 120 秒内，文献盒新增 1 张 literature 卡，含文献全文或大纲摘要。 |
| AI 从文献中提取**满足三原则**的核心概念，写入 `fleeting/concept-map.json`。 |
| 满足三原则的概念数量 ≥ 3 且 ≤ 8（短文献低于 3 时返回提示「文献过短，建议合并多篇」，不强行凑数）。 |
| 每个提取的概念在 Galaxy 中以**青色节点**出现，初始 `status='pending'`，正文 section 留空。 |
| 提取的概念**不重复**——与已有永久卡同名概念**复用而非新建**（同名判定：标题字符串完全相等）。 |
| 概念之间的依赖关系作为 `[[wikilink]]` 写入 concept-map.json 的 `dependencies` 字段。 |

#### L1.9.b.清晰原则校验 🤖

> 每个被提取的概念会经过以下校验。未通过的**标记为 `low_confidence` 并降级到 `extras` 列表**，而非直接丢弃。用户可在审计面板中查看并手动复活。

| 命题 |
|------|
| 概念**名称**字符数 ∈ [2, 30]（避免过长或过短）。 |
| 概念**摘要描述**（concept-map.json 中的 `description` 字段）字数 ∈ [10, 150]。 |
| 概念**摘要描述**含完整句式（结尾标点 `。` `.` `！` `!` 之一）。 |
| 概念**摘要描述**字符串中**不包含**概念**名称**本身作为定义主体（拒绝循环定义：「进程是进程的一次执行」违规）。**例外**：概念名出现在解释性从句中不视为违规（如「进程调度是操作系统为**进程**分配 CPU 资源的策略」——此处「进程」是举例而非自我定义）。判定规则：概念名仅出现在从句/修饰语中（前后有逗号、顿号或「的」连接）时豁免。 |

| 命题 |
|------|
| Oracle 对话中（非 lecture-mode），连续 5 轮内 Oracle 回复中**至少 3 轮包含问号 `？` 或 `?`**——Oracle 应主动提问而非仅讲授。单条回复问号数量不限（避免为凑数而堆砌问号）。容忍度：连续 3 次抽样执行中 ≥ 2 次通过即可。 |

## L2.5 — AI 不代劳（费曼铁律）

### L2.5.机制声明

> AXIOM 区分两类卡片内容来源：
> - **user_typed**：用户键盘输入或粘贴
> - **user_accepted_ai**：AI 在对话中提出建议句，用户**显式点击「采纳此句」按钮**后写入
>
> **默认规则**：卡片 section 写入的内容应**语义上源自**用户输入。验证方式：section 中非格式行（排除 Markdown 语法行、YAML frontmatter）与用户 chat message 的**语义覆盖率** ≥ 80%（按行计：每行在用户历史消息中存在子串匹配或 Levenshtein ≥ 0.75 的改写），或该行有对应的「采纳」按钮点击日志。
>
> **改写容忍**：用户对 AI 建议的句子做了修改（改词、调语序、删减）后写入，视为用户内容而非 AI 代劳。判定标准：用户提交的文本与 AI 原始建议的 Levenshtein 相似度 ≤ 0.9（即至少 10% 的修改量）。
>
> **例外通道**：AI 可在 chat 中以引号块的形式**提议**句子，附带 `[采纳此句到 ⟨section⟩]` 按钮。用户点击后，该句以 `acceptedFromAI: true` 标记写入。

### L2.5.命题 🤖

| 命题 |
|------|
| 我在 Forge 编辑器留空一张永久卡的「定义」section，向 Forge Agent 输入「帮我写定义」。Agent 的回复字符串**不写入** card.content（PUT 请求体不含该 section 的新内容）；Agent 改为反问「你怎么理解这个概念」（回复字符串含问号 `？`）。 |
| `card.content` 中**标记为语义内容的 section（定义/举例/关联/应用 四个 section）**的每一行字符串（按换行符 split，排除空行、Markdown 格式行、YAML frontmatter）：**语义覆盖率 ≥ 80%**（每行要么在用户 chat message 中有子串匹配，要么 Levenshtein ≥ 0.75 的改写，要么对应 `card.audit_log` 表中有 `action='accept_ai_suggestion'` 记录）。未覆盖行 ≤ 总行数的 20% 且不超过 3 行。 |
| `card.content` 中 `acceptedFromAI=true` 的字符串占比 ≤ 默认 30%（按字符数计，阈值可在 `user_settings.ai_suggestion_quota` 中调整）。超过阈值时，Forge Agent 触发警告，DOM 出现 `[data-warning='cards-too-much-ai']` 元素；用户可点击该警告中的 `[data-action='raise-quota']` 将阈值调高 10%。 |
| `card.audit_log` 表记录每次写入的 `source` 字段：枚举值 `user_typed` / `user_accepted_ai`。 |
| `acceptedFromAI` 提议显示为 `<blockquote>` 块状元素，附带 `[data-action='accept-suggestion']` 按钮；只有点击该按钮才会触发 PUT 请求；未点击的提议不影响 card.content。 |

## L2.6 — 卡片四要素完整 + 显式升级机制

### L2.6.机制声明

> 闪念→永久不是自动判定，是**用户主动按按钮**触发。
> 这是费曼学习法在 UI 上的落点：**「按下按钮」= 「我认为我懂了」**。
> 但**懂不等于合格**。系统在升级时还要再校验一次三原则（清晰 / 准确 / 必要），把住永久盒的入口。

### L2.6.命题 🤖

| 命题 |
|------|
| Forge 编辑器右上角 DOM 存在按钮 `[data-action='promote-to-permanent']`，文案含「升级为永久卡」字样。 |
| 点击该按钮时，前端依次执行**四要素校验**和**三原则校验**；任一不满足则弹出 modal，DOM 中显示 `[data-warning='upgrade-blocked']` 元素，文案指出**具体违反的校验编号**；不发起 PUT 请求。 |

#### 四要素校验（卡片结构完整）

| 命题 |
|------|
| **校验 A1**：4 个 section（定义 / 举例 / 关联 / 应用）全部非空。 |
| **校验 A2**：每个 section 字符数 ≥ 30。 |
| **校验 A3**：「关联」section 含至少 1 个合法 `[[wikilink]]`（正则 `\[\[[^\]]+\]\]` 匹配）。 |

#### 三原则校验（概念质量门槛，对应 §1.6+）

| 命题 |
|------|
| **校验 B1（清晰）**：「定义」section 字数 ∈ [10, 150]。 |
| **校验 B2（清晰）**：「定义」section 含完整句式（结尾标点 `。` `.` `！` `!` 之一）。 |
| **校验 B3（清晰·拒绝循环定义）**：「定义」section **不能以卡片标题本身作为定义的主体**。判定规则：（1）若定义的主谓结构中主语 = 卡片标题且谓语仅含「是」「指」「就是」「即」等连接词 + 标题本身或其同义反复 → 违规；（2）标题出现在从句、修饰语、举例中（如「进程调度是操作系统为**进程**分配 CPU 的策略」）→ 不违规。自动判定方式：定义句中标题出现次数 ≥ 2 且无其他实义动词 → 判违规；否则通过。 |
| **校验 B4（准确）**：「举例」section 字数 ≥ 30 且**不与「定义」section 重复**（两 section Levenshtein 相似度 ≤ 0.8，避免抄定义）。 |
| **校验 B5（必要）**：「关联」section 中的 `[[wikilink]]` 至少有 1 个指向**真实存在**的卡片（非 dangling）。这保证此概念**承载**了至少一条与外部的连接。 |

| 升级成功后，5 秒内 Galaxy 中对应节点颜色从青色（fleeting）变为紫色（permanent），无需用户手动刷新。 |
| 升级**不可由 AI 触发**——`audit_log` 中此次类型变更的 `actor` 字段必须为 `user`，不允许 `agent` / `system`。 |
| 升级失败 modal 中提供 `[data-action='show-which-failed']` 详情按钮，展开后显示**每条**校验的 PASS/FAIL 状态，让用户知道具体哪一条没过。 |

#### 弹性条款（呼应 L8.3）

| 命题 |
|------|

## L4.1 — Oracle 苏格拉底式 + 上下文边界

### L4.1.边界声明

> **「苏格拉底式」是一个隐喻**，本身不可被自动化判定。
> 我们只能测它的**代理指标**——即"看起来像苏格拉底式"的可观察特征。
> 这些代理指标是**必要条件**，不是**充分条件**：通过代理 ≠ 真的苏格拉底；不通过 = 一定不是。
> L4.1 通过 = 全部代理指标通过 + 🧑 抽样人工标注通过。

### L4.1.a — 代理指标 🤖

> **非确定性声明**：以下命题测试 LLM 输出模式，受 prompt / 模型 / 温度影响。每条命题默认容忍度：连续 3 次执行中 ≥ 2 次通过即可。

| 命题 |
|------|
| Oracle 每条回复字符串中，问号 `？` 或 `?` 出现次数 ≥ 1。容忍度：连续 5 条回复中允许 ≤ 1 条无问号（Oracle 可能先做简短确认再追问）。 |
| 连续 20 条 Oracle 回复中，以 `？` 或 `?` 结尾的比例 ∈ [30%, 80%]——既要求足够追问（≥30%），又避免为凑数硬塞问号（≤80%）。 |
| Oracle 回复首句**不含**以下讲授式短语（任一命中即该轮扣分，但连续 3 轮中允许 ≤1 次命中——LLM 偶尔回退到讲授式可接受）：`让我解释` / `我来讲讲` / `下面介绍` / `首先要明白` / `定义是` / `综上所述` / `总而言之`。**注意**：`这是一个` 从黑名单中移除——因为"这是一个很好的问题""这是一个常见的误解"等是追问的合理前奏。**例外**：当用户上一条消息含"讲""告诉""解释""直接"之一时，黑名单豁免该轮（用户主动要求讲授）。 |
| 当用户上一条消息含某概念 X 时，Oracle 下一条回复中复用 X 字符串次数 ≥ 1（证明 Oracle 在听，不是发预设稿）。 |
| 当用户输入「我完全不懂」时，Oracle 回复字符串中**不含**学科专业术语清单中的任何一项（学科术语清单由 cluster 标签定义）；且回复包含至少 1 个问号。容忍度：连续 3 次中 ≥ 2 次。 |
| 当用户输入一个明显错误的描述时（如「进程就是文件」），Oracle 下一条回复中**至少**包含一个反例或对比说明（含「但是」「然而」「区别在于」「不太准确」「其实」中至少一个关键词）。容忍度：连续 3 次中 ≥ 2 次。 |
| 当用户输入「我懂了」时，Oracle 下一条回复**必含**一个追问，且追问要求用户做以下之一：举例、复述、应用到具体场景（关键词：`举个例子` / `用你的话` / `如果遇到` / `试着` 中至少一个）。容忍度：连续 3 次中 ≥ 2 次。 |

## L5.3 — 编辑器（Forge）+ 自动保存 + 崩溃恢复

### L5.3.机制声明

| 维度 | 规则 |
|------|------|
| 自动保存触发 | 用户停止输入 **3 秒**后发起 PUT 请求（防抖） |
| 强制全量保存 | 即使用户持续输入，每 **30 秒**强制 PUT 一次 |
| 本地草稿 | 每次按键写入浏览器 IndexedDB，key = `draft:<cardId>` |
| 崩溃恢复 | 打开卡片时优先读 IndexedDB，对比服务端 `updatedAt`。本地草稿更新则弹 modal 询问「检测到本地草稿，是否恢复」 |
| 多设备冲突解决 | Last-Write-Wins by `updatedAt`；写入时若 `baseUpdatedAt < 服务端 updatedAt`，前端弹冲突 modal（三选一：保留本地 / 用服务端 / 手动合并） |

### L5.3.命题 🤖

| 命题 |
|------|
| Live / Read 模式切换前后，`textarea` / 渲染区文本内容 hash 一致。 |
| Ctrl+S 触发 PUT 请求；从按键到返回 200，网络往返耗时 ≤ 500ms（服务端处理耗时单独记录在响应头 `X-Process-Time` 供监控，不参与前端超时判定）；UI 显示 `[data-status='saved']` 元素。 |

| **自动保存（防抖）**：连续输入不超过 3 秒间隔时，PUT 请求不发起；停止输入 3-4 秒（含浮动）内发起 1 次 PUT；UI 显示 `[data-status='auto-saved']` + 时间戳。 |

## L5.13 — 错误边界、降级、纠正路径、安全

| 命题 |
|------|

| LLM API Key 失效时，Agent 返回友好错误，**不泄露**原始堆栈（响应体不含 `at /node_modules/`、`stack:` 等）。 |

| 用户 A 无法通过任何接口读到用户 B 的卡片、Vault、Session。 |

| **Rate Limiting**：同一 IP 对 Agent 接口（`/api/agent/chat`）连续请求 30 次/分钟后，返回 429 或队列降级，不导致 API Key 额度耗尽。⚠️ 若产品未实现 Rate Limiting，记入 finding 而非判 FAIL。 |
| **内容注入 via WikiLink**：卡片中写入 `[[<script>alert(1)</script>]]`，Backlinks 面板渲染后**不执行脚本**。 |
| **SSRF 防护**：Agent 的 `web_search` / `web_fetch` 工具**不可访问**内网地址（`localhost` / `127.0.0.1` / `192.168.x.x` / `10.x.x.x`）。⚠️ 若 web 工具仅供 Agent 内部使用（用户不可直接调用），此项降级为代码审计而非 E2E 测试。 |

# 🟡 L8 弹性与降级 — 用户在不完美里仍然舒服

> **L8 用户期望**：
> 我不是机器。我会慢、会跳步、会改主意、会断网、会编辑一半去厕所。
> AXIOM 不应当因为我没按它的剧本走，就**惩罚**我（删数据 / 弹 modal / 卡死 / 报错）。
> 它应该像一个**有耐心的朋友**——我快它就快，我慢它就慢，我错了它告诉我但不堵我，我想跳步它默许。
>
> **L8 验证的是钢条之间的弹簧。**
> L0-L7 验证"对的样子"，L8 验证"错的样子里，用户仍然活得下去"。

## L8.1 — LLM 慢 / 失败的弹性 🤖

| 命题 |
|------|
| LLM 响应未返回首 token 超过 **5 秒**时，UI 出现 `[data-status='thinking']` 元素（骨架屏 / 旋转图标 + "正在思考..." 文案），**不白屏**。 |
| LLM 响应未返回首 token 超过 **15 秒**时，UI 增加显示 `[data-action='cancel-llm']` 按钮，用户可中断；中断后已发送的用户消息**保留**在输入框（不丢失）。 |
| LLM 响应超过 **30 秒**仍未返回，前端将 15 秒时的 cancel 按钮变为**强制取消按钮**（红色，`[data-action='force-cancel-llm']`），**不自动 cancel**——已等待 30 秒的部分回复可能仍可用，由用户决定是否放弃。用户主动取消后显示 `[data-action='retry-llm']` 按钮 + 友好提示文案。 |
| LLM 返回 HTTP 4xx/5xx 时，UI 显示**用户语言**错误（"AI 服务暂时不可用，要不要换个模型？" / "请稍后再试"）；不显示 raw error。 |
| LLM 失败时，session 历史**完整保留**——刷新页面后能继续上次对话。 |
| 连续 3 次 LLM 失败后，自动启用 `degraded-mode`：UI 顶部显示降级提示，禁用资源生成等重量级功能，仍允许浏览既有卡片与编辑。 |

| 降级时保留 `polishedAt` 历史和 4 个 section 内容，不清空——再次升级时仍可用原内容。 |
| 文献盒 export ZIP 含**已删除**卡片，但放在 `trash/` 子目录，frontmatter 标 `deleted: true`。 |
| 误操作"采纳 AI 建议"后，可点 `[data-action='unaccept-suggestion']`，在 **10 秒内**撤回该次采纳；超时则需手动编辑卡片。 |

## L8.7 — 打扰弹性 🤖

| 命题 |
|------|
| "下一个学什么"推送区域**永远是被动展示**——不弹 modal、不发系统通知、不发邮件。 |
| 用户连续 **7 天未点击**推送中任意卡片，推送内容根据画像**重新排序**（将 `interests` 匹配度高的概念置顶），数量**不降**——不点击不代表不学，是推送内容不匹配的信号。连续 14 天未点 → 自动隐藏推送区域，仅保留入口按钮。 |

| 命题 |
|------|
| 用户在设备 A 编辑卡片 C 并保存；用户在设备 B **5 分钟后**打开 C，看到的是最新版本（无冲突，**不弹** modal）。 |

## L8.10 — 失败优雅 🤖

| 命题 |
|------|
| 所有 API 错误响应体含 4 个标准字段：`code` (string)、`message` (用户友好中文)、`userAction` (用户能做什么，如"请稍后重试" / "联系支持")、`retryable` (boolean)。 |
| 前端**不显示** stack trace、HTTP status code 原文、错误 ID 等技术信息；这些信息记录到浏览器 console 供开发者调试，但 UI 只显示 `message + userAction`。 |

| 任何 panic / unhandled exception 被全局 ErrorBoundary 捕获，显示一张友好降级页（含"刷新 / 回首页 / 报告问题"三个按钮）；不暴露原始报错。 |

# 4. 8 维认知升级验证（特别专题）

> 用户使用系统 30 天后，应该在 8 个维度上**可观测**地提升。

| 维度 | 观测命题 |
|------|---------|
| 1. 复杂情境整合 | 我能针对一个跨学科问题，从永久盒中调出 ≥3 张不同领域的卡片并阐述其关联。 |
| 2. 多方立场分析 | 我的永久卡中至少 1 张含「不同视角对比」section。 |
| 3. 洞察本质与表象 | Oracle 多次追问后，我能区分某概念的「定义」和「常见误解」。 |
| 4. 逻辑严密推理 | 我的卡片间存在「前提 → 推理 → 结论」式的链式引用。 |
| 5. 认知偏差觉察 | 我至少 1 次主动标注「我之前以为 X，其实是 Y」。 |
| 6. 反直觉思维 | 我至少 1 张永久卡的「应用」section 含反直觉案例。 |
| 7. 价值冲突选择 | 我至少 1 次面对 Oracle 的开放性问题给出明确立场。 |
| 8. 创造性输出 | 我**关闭系统后**，能向第三人讲清某概念，并举出系统中**未出现过**的例子。 |

**通过标准**：30 天后，≥6/8 维度的命题为真。
