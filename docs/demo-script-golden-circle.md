```
│                                                       │
│  ┌─ 问题：一个 Agent 做不到所有事 ────────────────┐  │
│  │  教学、评估、画像分析、卡片引导、路径规划        │  │
│  │  → 所以我们需要 5 个专业子 Agent 可搭配召唤      │  │
│  └──────────────────────────────────────────────────┘  │
│                                                       │
│  ┌─ 问题：多个 Agent 需要通信和编排 ──────────────┐  │

│  └──────────────────────────────────────────────────┘  │
│                                                       │
├─────────────────────────────────────────────────────┤
│  第一层：双 Agent 底座                                │
│  Agent A (前台教学 · 对话+工具+召唤子Agent)          │
│  Agent B (后台监控 · 每轮对话后静默分析)              │
├─────────────────────────────────────────────────────┤
│  第二层：5 种子 Agent 角色                            │
│  Oracle · Profile · Forge · Guide · Assess           │
├─────────────────────────────────────────────────────┤
│  第三层：基础设施                                     │
│  编排引擎 · 消息总线 · 认知层 · 记忆 · 工具系统       │
└─────────────────────────────────────────────────────┘
```

**🎙️**：
> 我们给她取名 **Hermes**。
> 为什么自己写一套，而不是用 LangChain 或 AutoGen？
> 因为市面上的 Agent 框架解决的是"调用多个 API"的问题——
> 而我们的问题是：**需要一个前台 Agent 专心教学，同时一个后台 Agent 静默分析学生。**
> 没有现成框架支持这个模式。
>
> 所以 Hermes 有三层：
> **第一层，双 Agent 底座** — 前台教、后台分析，互不干扰。
> **第二层，5 种子 Agent** — 教学、评估、画像、卡片、路径，各司其职。
> **第三层，基础设施** — 编排引擎、消息总线、认知层、记忆引擎。
>
> 下面我用实际使用来证明，这三层分别怎么工作。

**📌**：
`Hermes Agent 三层架构 · 完全自研 · 不基于 LangChain/AutoGen`

## 附录 B：4 个 FAQ 防反驳准备

> 评委可能会在答辩环节质疑。这些问答不是视频内容，是给答辩人准备的口头弹药。

**Q1：你们和直接用 Claude/GPT 有什么区别？**
> A：市面上的产品你打开就是输入框，你得先知道问什么。AXIOM 反过来 —— Agent 主动向你提问，根据你的反馈调整难度。而且我们的 Agent B 在后台每轮分析画像，这是单一 LLM 调用做不到的。本质上，**我们不是在调用一个 AI，是在搭建一个教学系统**。

**Q2：自研架构有必要吗？用 LangChain 不是更快？**
> A：LangChain 解决的是"调多个 API"的问题。我们的需求是 **"一个前台教学，一个后台分析，互不干扰"** —— LangChain 没有这个模式。而且自研意味着我们可以精确控制工具调用的权限、消息路由的优先级、以及编排工作流的粒度。**不是为了自研而自研，是为了满足需求不得不自研。**

**Q3：6 维画像怎么证明是准的？**
> A：我们有两个机制。一是**置信度机制** — 数据不够的维度不显示，宁可藏着也不误导。二是**观察清单** — 画像中的每个维度变化都有对应的 AI 观察记录，不是黑箱打分，是可追溯的诊断。您可以点开`观察清单`看每一条记录。

**Q4：讯飞星火在你们系统里起了什么实际作用？**
> A：系统的核心推理、资源生成、评估诊断都可以通过讯飞星火 API 驱动。我们会前端的模型选择暴露为环境变量，默认使用讯飞星火 Spark Max 模型。本次演示的所有资源生成和对话，都可以跑在讯飞上。我们在文档中有详细的适配层说明和性能对比。
