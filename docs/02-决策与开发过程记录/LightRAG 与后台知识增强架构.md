# LightRAG 与后台知识增强架构

> 本文档说明 AXIOM Space 为什么接入 LightRAG、Redis 和后台任务系统，以及它们如何把“存卡片”升级成“可召回、可关联、可持续生长的个人知识系统”。
> 适合：答辩材料、技术说明、后续功能规划。

---

## 一句话结论

AXIOM 的主数据库负责保存用户真实数据，LightRAG 负责把卡片加工成可检索的知识索引，Redis 负责把耗时索引任务放到后台执行。

用户体验上的变化是：

> 用户写下的卡片和导入的资料，不只是被保存起来，而是会被系统理解、索引、关联，并在后续 AI 对话、路径规划、知识图谱和推荐中被重新调用。

---

## 一、为什么需要这套增强层

传统笔记系统的问题是：资料存进去了，但系统并不真正理解它。

用户体验上会出现三个断点：

1. **存了很多，但 AI 想不起来**
   用户明明写过某个概念，之后问 AI 时，AI 仍然像第一次见到这个主题。

2. **资料很多，但关系靠用户手工连**
   只有用户显式写了 `[[链接]]`，图谱才出现边；隐含关系难以被发现。

3. **导入资料后，没有真正进入学习闭环**
   文档只是被保存，不能稳定参与路径规划、个性化问答和知识缺口分析。

LightRAG 的作用就是补上这个断点：它把 AXIOM 中的卡片和资料加工成一个可以被 AI 检索和推理的知识索引。

---

## 二、三层数据职责

### 1. Postgres：主数据库

Postgres 是 AXIOM 的事实源。

它保存：

- 用户、认证、Vault
- 卡片原文、标题、类型、颜色、星团归属
- 学习路径、学习步骤、学习消息
- AI 对话记录
- RAG 同步状态

**原则：Postgres 里的数据才是原始数据，LightRAG 可以重建。**

### 2. LightRAG：知识索引层

LightRAG 保存的是派生数据。

它加工并保存：

- 文本切块
- embedding 向量
- 实体
- 关系
- 检索索引
- 文档处理状态

它不是 AXIOM 的主数据库，而是 AI 用来“想起相关知识”的增强记忆层。

### 3. Redis：后台任务层

Redis 不存长期知识。

它负责保存临时任务：

- 哪张卡片需要同步到 LightRAG
- 哪个 Vault 需要重新索引
- 任务是否完成、失败、重试

用户保存卡片时，不需要等待 LightRAG 完整抽取完成；任务会进入 Redis，由后台 worker 慢慢处理。

## 三、当前已经接入的真实链路

当前已经跑通的链路是：

```text
用户创建/修改卡片
    ↓
Postgres 保存原始卡片
    ↓
Prisma card.create/update/upsert 自动触发 RAG 入队
    ↓
Redis 保存 rag.indexCard 后台任务
    ↓
worker 消费任务
    ↓
LightRAG 接收卡片文本
    ↓
Ollama bge-m3 生成向量
    ↓
DeepSeek 执行实体/关系抽取
    ↓
LightRAG 完成 processed
    ↓
Postgres ragDocumentIndex 标记 indexed
```

这条链路已经具备几个关键特性：

- **统一触发**：所有 `card.create/update/upsert` 都会自动进入 RAG 队列，不依赖某一个前端页面。
- **后台执行**：保存卡片不会被 RAG 处理阻塞。
- **真实完成状态**：不是收到 LightRAG 200 就标记完成，而是等待 `track_status` 变成 processed。
- **更新可重建**：卡片内容更新时，会删除旧 LightRAG 派生文档，再重新插入，避免旧知识污染。
- **对话可召回**：AI 对话前可以查询 LightRAG，并把检索结果注入 Agent prompt。

## 四、用户能感受到什么

### 1. AI 对话更懂我的知识库

用户问：

> 我之前写过的“进程和线程的区别”是什么？

系统不再只靠模型记忆或当前对话，而是先到 LightRAG 中查相关卡片，再带着检索结果回答。

用户感受到的是：

- AI 能引用自己写过的卡片
- AI 回答更贴近个人知识库
- 学得越多，AI 越懂自己

### 2. 导入资料不只是保存，而是进入知识网络

用户导入一份文档后，系统可以把文档生成的卡片同步进 LightRAG。

后续可以支持：

- 问这份资料里的内容
- 让 AI 找资料之间的共同概念
- 从资料中发现适合打磨成永久卡的点

### 3. 卡片更新后，AI 会使用新内容

用户修改卡片后，旧索引会被替换。

这避免了一个很糟糕的体验：

> 用户明明改了卡片，但 AI 仍然回答旧内容。

### 4. 图谱可以不只靠手写链接

当前 AXIOM 图谱主要依赖数据库里的显式边，比如 `[[WikiLink]]` 和学习路径边。

LightRAG 后续可以提供隐含关系：

- 两张卡片讲同一个概念
- 某个概念是另一个概念的前置知识
- 某些卡片属于同一个主题簇
- 某些资料之间存在冲突或互补

这可以增强 2D/3D 知识图谱，让它更像一个“活的知识网络”。

## 五、LightRAG 可以继续增强的体验

### 1. 知识库问答

页面能力：

- 用户直接问整个 Vault
- 回答显示引用来源
- 引用可以跳回具体卡片

体验价值：

> 用户不需要记得资料放在哪里，只要问问题，系统会从自己的知识库里找证据。

### 2. 相关卡片推荐

页面能力：

- 打开某张卡片时，自动显示“相关卡片”
- 分成显式关联和语义关联
- 支持一键建立 WikiLink

体验价值：

> 用户写卡片时，系统主动告诉他“这张卡可能和哪些知识有关”。

### 3. 知识缺口发现

页面能力：

- 检测某个主题下缺少关键概念
- 检测只有资料卡，没有永久卡的区域
- 检测图谱中孤立节点

体验价值：

> 系统不只展示已有知识，还能指出“你这里还没学完整”。

### 4. 路径规划增强

页面能力：

- Path 页面生成学习路径时，先读取 LightRAG 中的已有知识
- 根据用户已经掌握的卡片跳过基础内容
- 根据薄弱区域补充复习步骤

体验价值：

> 路径不再像通用课程大纲，而是根据用户自己的知识库生成。

### 5. 图谱增强

页面能力：

- 在 2D/3D 图谱中叠加 LightRAG 抽取的隐含边
- 支持“显式链接 / 语义相关 / 前置关系 / 同主题关系”切换
- 支持按关系强度筛选

体验价值：

> 用户看到的不只是自己手动连出的图，而是系统帮他发现的知识结构。

### 6. 对话沉淀

页面能力：

- 不把所有聊天流水直接进 RAG
- 只把重要对话总结为卡片或记忆
- 用户确认后沉淀进知识库

体验价值：

> 聊天不会变成噪音，真正重要的内容会沉淀为长期知识。

### 7. 重复与冲突检测

页面能力：

- 检测两张卡片是否表达相同概念
- 检测某些卡片观点是否冲突
- 提供合并或保留差异的建议

体验价值：

> 用户的知识库会越来越干净，而不是越写越乱。

## 六、和 Hermes Agent 架构的关系

LightRAG 不是替代 Hermes Agent，而是增强 Hermes Agent 的长期知识能力。

```text
Hermes Agent
    负责对话、教学、追问、资源生成、路径规划

Postgres
    负责保存用户真实数据

LightRAG
    负责把卡片加工成可召回知识索引

Redis + worker
    负责把耗时任务放到后台稳定执行
```

在答辩中可以这样解释：

> Hermes 是智能体大脑，Postgres 是事实仓库，LightRAG 是长期知识检索层，Redis 是后台调度系统。四者组合后，系统不是一次性聊天工具，而是一个会随用户学习持续成长的认知系统。

## 七、答辩时可以强调的技术亮点

### 亮点一：主数据和派生索引分离

AXIOM 不把 LightRAG 当主数据库。

这样设计的好处：

- 用户原始数据安全、可控
- RAG 索引坏了可以重建
- 不会被第三方索引结构锁死
- 业务数据和 AI 检索数据边界清晰

### 亮点二：后台异步索引

RAG 处理很慢，涉及 LLM 抽取和向量生成。

AXIOM 用 Redis + worker 把它放到后台：

- 前端保存不卡顿
- 任务失败可重试
- 批量导入可以逐步处理
- 后续可以扩展文档解析、图谱重算、画像更新

### 亮点三：本地免费 embedding

当前使用：

- DeepSeek 负责 LLM 抽取
- Ollama + bge-m3 负责本地 embedding

好处：

- embedding 不消耗外部 API 费用
- 数据向量化在本地完成
- 后续可替换更强的本地模型

### 亮点四：RAG 不是简单向量搜索

LightRAG 不只是做 embedding 相似度搜索，还会抽取实体和关系。

这和 AXIOM 的知识图谱目标天然匹配：

- 卡片是知识节点
- 链接是显式关系
- LightRAG 可以补充隐含关系
- 图谱可以成为学习路径和认知画像的基础

## 八、后续落地优先级

### P0：让用户看见 RAG 是否生效

要做：

- 卡片右侧显示 RAG 状态：未同步、索引中、已索引、失败
- AI 对话显示“已参考知识库”
- 失败时允许用户点击重试

原因：

> 底层能力再强，如果用户看不见，就会觉得系统没有变化。

### P1：AI 对话引用卡片来源

要做：

- Agent 回答后显示引用卡片
- 点击引用跳转到 Forge 卡片
- 区分“来自知识库”和“模型推断”

原因：

> 这是答辩中最直观的 RAG 价值证明。

### P2：卡片相关性推荐

要做：

- Forge 打开卡片时查询 LightRAG
- 推荐相关卡片
- 支持一键建立链接

原因：

> 这能直接提升用户整理知识的效率。

### P3：图谱隐含关系叠加

要做：

- Galaxy 增加关系来源筛选
- 显式边和 LightRAG 隐含边分层显示
- 支持按强度过滤

原因：

> 这会让知识图谱从“展示工具”升级成“发现工具”。

### P4：学习路径增强

要做：

- Path 页面生成路径前先查询 LightRAG
- 根据已有卡片判断起点
- 根据缺口生成下一步

原因：

> 这是把 RAG 价值从问答扩展到个性化学习规划。

---

## 九、当前边界

当前系统不会把所有聊天流水直接写入 LightRAG。

原因：

- 聊天内容噪音高
- 每句话都入库会污染知识索引
- 用户临时想法不一定值得长期保存

更合理的策略是：

> 对话先进入 Postgres；只有经过总结、确认、卡片化之后，才进入 LightRAG。

这符合 AXIOM 的核心理念：知识不是被动堆积，而是经过用户理解和打磨后的长期结构。

---

## 十、答辩口径

可以用这段话概括：

> AXIOM 不是把资料简单存进数据库，而是把用户写下的卡片和导入资料转化为可召回、可关联、可持续更新的个人知识网络。Postgres 保存真实原文，Redis 负责后台任务，LightRAG 负责抽取实体关系和向量索引，Hermes Agent 在对话和路径规划中调用这层知识。这样系统会随着用户学习不断变聪明，真正形成个人化、长期化的学习智能体。
