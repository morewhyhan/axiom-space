# 知识图谱布局优化审视报告

本文档记录对当前知识图谱布局、连线、装饰元素、旋转和相机交互的审视结论。它面向产品和开发，不面向最终用户。

相关文件：

- `components/three/galaxy-canvas.tsx`：Three.js 场景、节点、连线、布局算法、相机和交互。
- `components/galaxy/galaxy-layout-panel.tsx`：`LAYOUTS` 按钮列表。
- `components/galaxy/galaxy-controls.tsx`：旋转、Bloom、彗星、银河带、过滤等控制。
- `stores/mode-store.ts`：`GraphLayoutMode` 状态和持久化。
- `server/api/routes/galaxy.ts`：图谱节点、边、星团数据来源。
- `types/galaxy.ts`：前端图谱数据类型。

---

## 核心结论

当前知识图谱的问题不是某一个布局参数不对，而是三件事混在了一起：

1. **布局模式的产品边界不清**：有些模式是总览，有些是关系分析，有些是学习路径，有些是状态比较，但 UI 里全部平铺成同一层级。
2. **布局动画和连线渲染没有形成稳定的一体化机制**：节点位置是 GSAP 动画，连线是独立几何体，需要手动刷新。当前做法会带来性能和同步风险。
3. **相机交互没有按模式适配**：点击节点后统一进入 3D focus 行为，导致 2D/2.5D 视图里视角很怪。

用户看到的现象是：

- 切换布局时，节点先变，线条慢半拍。
- 星系、平面、环形、同心、矩阵之间边界感不足。
- 矩阵看起来像“奇怪地排开”，而不是可理解的分析视图。
- 点击节点后，相机在某些模式中不该靠近却靠近。
- 银河带、彗星被做成开关，反而增加了用户负担。

## 当前实现概览

### 布局模式

当前 `GraphLayoutMode` 一共有 10 种：

- `galaxy`
- `flat`
- `radial`
- `concentric`
- `layered`
- `matrix`
- `task-flow`
- `timeline`
- `mastery`
- `evidence`

`GalaxyLayoutPanel` 直接把这 10 种模式平铺在 `LAYOUTS` 面板里。用户看到的是一组按钮，但没有看到它们属于不同使用场景。

### 场景结构

当前 Three.js 场景大致由这些对象组成：

- 背景星云 `nebulaGroup`
- 远处星空 `starField`
- 银河带 `milkyWay`
- 彗星粒子
- 节点组 `nodesGroup`
- 连线组 `linksGroup`
- 学习路径 `learningPath.group`
- HTML 标签层 `labelOverlay`

节点和连线不是同一个对象层级下的动态绑定关系。节点移动后，连线必须重新计算几何体。

### 布局切换流程

当前切换布局时大致流程是：

1. `applyGraphLayout(mode)`
2. 计算目标点位 `computeLayoutTargets(mode)`
3. `animateNodesTo(targets, duration)` 用 GSAP 移动节点
4. 动画期间 `layoutAnimating = true`
5. 每帧 `refreshLinkGeometry()`
6. 动画结束后再刷新一次连线

这个思路方向是对的：动画期间确实需要刷新连线。但当前刷新方式代价很高。

## 关键问题分析

## 1. 连线跟随慢，不只是视觉问题

### 当前做法

当前每次刷新连线时，会对每条线执行：

- 根据 source/target 当前坐标重新生成曲线点。
- dispose 旧 geometry。
- 创建新的 `BufferGeometry`。
- `setFromPoints(points)`。
- 如果有 flow，则更新 flow 的 points。

这意味着布局动画期间，每一帧都可能为所有连线创建和销毁大量几何对象。

### 为什么用户会看到“线慢”

表面看代码已经在动画期间每帧刷新连线，但用户仍然可能看到线慢，原因主要有三类：

1. **每帧重建几何体太重**  
   节点多、边多时，频繁 dispose/create 会造成 GC 压力和帧率下降。帧率一掉，线条更新就会显得跟不上。

2. **GSAP ticker 和 render loop 的时序不完全等价**  
   节点位置由 GSAP 更新，连线由 Three.js 动画循环刷新。如果某一帧里连线计算发生在节点位置更新之前，线会短暂使用上一帧的位置。

3. **连线可见性和几何刷新是两套逻辑**  
   `applyLayoutLinkVisibility()` 负责显示/隐藏和透明度，`refreshLinkGeometry()` 负责位置。切换时先改变可见性，再移动节点，用户容易看到旧几何线突然出现，然后再跟随。

### 应该怎么改

连线应该进入“动态缓冲”模式，而不是每帧销毁重建。

建议方案：

1. 每条线创建时固定一份 `BufferGeometry`。
2. 动画时只更新 position attribute 数组。
3. 不在动画循环里 dispose/create geometry。
4. 在布局切换开始时，把所有需要显示的线立即切换为动态刷新状态。
5. 布局稳定后，再决定是否重算高质量曲线。

更进一步，可以分两级：

- 动画期间：使用轻量直线或低分段曲线，保证贴合。
- 动画结束：重建更漂亮的曲线，用于静态展示。

这样能保证切换时“关系不断裂”，同时保留美观。

---

## 2. 装饰元素应该成为模式规则，而不是用户开关

### 当前做法

控制面板里有：

- 彗星开关
- 银河带开关
- 彗星速度

但从产品语义上看：

- 银河带是世界背景，不是用户任务。
- 彗星是星系视图装饰，不是分析功能。

### 当前问题

用户不应该思考“我要不要打开银河带”。这会把本来应该由系统决定的视觉风格，变成一个干扰性设置。

彗星如果在分析型布局里出现，会影响用户阅读关系。

### 应该怎么改

规则应该变成：

- 银河带默认永远显示。
- 银河带不再出现在控制面板中。
- 彗星默认开启。
- 彗星只在 `galaxy` 模式显示。
- 彗星速度不作为默认面板里的核心设置，可以藏到高级设置，甚至直接固定。

实现上可以在 `applyGraphLayout(mode)` 里统一调用视觉规则：

```ts
setModeAtmosphere(mode)
```

例如：

```ts
milkyWay.visible = true
comets.visible = mode === 'galaxy'
```

不要让每个按钮自己控制视觉语义。

## 3. 自动旋转策略需要重做

### 当前做法

当前 `PLANAR_LAYOUT_MODES` 包含：

- `flat`
- `radial`
- `concentric`
- `task-flow`
- `timeline`

这些模式进入后会关闭 OrbitControls 的旋转能力，并且自动旋转被视为关闭。

### 当前问题

这和用户期望不完全一致。

平面、任务流、时间线确实应该稳定不动。但环形和同心天然适合围绕中心轻微运动。尤其是同心模式，如果节点围绕中心轻微转动，会更容易表达“以某个知识点为中心的关系场”。

### 应该怎么改

不要用“2D/平面”一个集合决定所有旋转。

应该把模式拆成独立交互配置：

```ts
type RotationPolicy = 'camera-orbit' | 'graph-spin' | 'none'
```

建议：

| 模式 | 旋转策略 |
|------|----------|
| 星系 | 缓慢 camera orbit 或整体场景展示旋转 |
| 平面 | 不旋转 |
| 环形 | 轻微围绕中心旋转 |
| 同心 | 围绕当前中心轻微旋转 |
| 分层 | 不旋转 |
| 矩阵 | 不旋转 |
| 任务流 | 不旋转 |
| 时间线 | 不旋转 |
| 地形 | 默认不旋转，可手动斜视 |
| 证据 | 不旋转 |

注意：如果采用旋转 `nodesGroup` 和 `linksGroup` 的方式，标签渲染必须使用 `getWorldPosition()`，不能再只读 `node.position`。否则节点转了，HTML 标签会留在旧位置。

## 4. 点击节点后的相机行为必须按模式分开

### 当前做法

现在点击节点会统一调用：

```ts
focusNode(node)
```

`focusNode()` 又会调用：

```ts
frameSelection(selection, node, duration)
```

这套逻辑会计算选中节点和邻居的包围盒，然后移动 `controls.target` 和 `camera.position`。

### 当前问题

这对 3D 星系视图是合理的，但对 2D/2.5D 视图不合理。

平面模式下，用户最需要的是保持俯视图，看清楚节点在网络中的位置。点击节点后如果相机移动到某个局部，用户会丢失全局方向。

同心模式下，点击新节点更应该重新以它为中心排布，而不是只把相机飞过去。

任务流和时间线里，点击节点应该高亮上下文，不应该打乱用户对顺序的理解。

### 应该怎么改

引入 `focusBehaviorByMode`：

| 模式 | 点击节点后的主行为 |
|------|-------------------|
| 星系 | 相机靠近选中节点和邻居 |
| 平面 | 保持俯视，高亮一跳/二跳关系 |
| 环形 | 暂停旋转，高亮节点所在扇区和跨区连线 |
| 同心 | 以该节点为新中心，重新展开圈层 |
| 分层 | 保持视角，高亮上下层链路 |
| 矩阵 | 保持视角，高亮当前格子和相关节点 |
| 任务流 | 保持视角，高亮前置/后续步骤 |
| 时间线 | 保持视角，高亮前后时间上下文 |
| 地形 | 可轻微靠近，但保留高度视角 |
| 证据 | 保持视角，高亮证据链 |

实现上，`focusNode()` 不应该直接决定相机，而应该派发：

```ts
handleNodeFocus(node, layoutMode)
```

然后按模式选择：

- `focusNodeIn3D()`
- `focusNodeInPlanar()`
- `recenterConcentricLayout()`
- `highlightEvidenceChain()`
- `highlightTaskContext()`

## 5. 重置视角不应该总是回星系

### 当前做法

控制面板里的“重置视角”会调用 `resetCameraView()`，并把布局模式设回 `galaxy`。

### 当前问题

如果用户正在平面模式、矩阵模式或同心模式中，只是想回到这个模式的默认视角，结果却被带回星系，会打断任务。

### 应该怎么改

拆成两个概念：

1. **重置当前视角**：回到当前模式默认相机。
2. **返回星系总览**：切回星系模式。

当前 `RESET VIEW` 更像“返回星系总览”，不应该承担所有重置含义。

## 各模式详细问题与建议

## 星系模式

### 当前优点

星系模式是当前最完整的模式。它有星团、中心节点、背景、彗星、银河带和整体氛围。

星团内节点按层级半径分布，核心节点更靠近星团中心，整体有宇宙结构感。

### 当前问题

星系视图承担了太多事情：展示、美观、关系、高亮、相机 focus、装饰开关都混在这里。

真实语义连线在默认总览里隐藏较多，这对整体美观有利，但用户可能不清楚哪些连接是真实知识关系，哪些只是星团连接。

### 建议

星系只负责总览和品牌感：

- 保持慢速自转。
- 彗星只在这里显示。
- 默认只显示低强度星团连接和必要的跨星团关系。
- 点击节点后可以进入 3D focus。
- “打开卡片”和“查看关系”应作为点击后的明确动作。

---

## 平面模式

### 当前优点

它用简化力导向逻辑让有关联的节点靠近，方向正确。

### 当前问题

力导向计算使用了 `allLinks`，其中包括人工的核心到星团、星团到节点连接，也包括真实语义边。这样会让“展示结构”和“真实知识关系”混在一起。

用户在平面模式里想看的通常是真实知识关系，而不是星团装饰关系。

### 建议

平面模式应该优先使用真实语义边：

- `wikilink`
- `related`
- `prerequisite`
- `derived`
- `counter`

星团归属可以影响初始位置或颜色，但不应该主导力导向。

相机保持俯视，点击节点只高亮邻居。

---

## 环形模式

### 当前优点

它已经有按星团扇区排列的基础。

### 当前问题

当前环形更像“星团子节点按弧线展开”，不是严格的环形关系图。中心、扇区边界和跨区连接不够明确。

### 建议

环形模式应该明确两层：

- 内圈：星团或核心主题。
- 外圈：具体卡片节点。

跨星团连线从圆内穿过，用户一眼看到桥接关系。

允许默认轻微旋转。点击后暂停旋转。

---

## 同心模式

### 当前优点

它按图遍历把一跳、二跳、三跳和外部节点分层，概念正确。

### 当前问题

当前中心节点来自 `lockedNode || pressedNode || hoveredNode || core`。但是用户在同心模式里点击新节点后，当前逻辑主要是 focus 相机和高亮，并不会自然重新计算同心布局。

这会导致用户以为“我点了新中心”，但图没有真正围绕新节点重排。

### 建议

同心模式点击节点时：

1. 设置该节点为中心。
2. 重新计算同心布局。
3. 节点和连线同步动画。
4. 视角保持俯视或轻微 2.5D。
5. 默认轻微围绕中心旋转。

---

## 分层模式

### 当前优点

它能按节点类型把永久卡、灵感卡、文献卡分层，符合知识加工的直觉。

### 当前问题

它目前更像按类型摆放，不是真正的依赖层级。没有明确表达“资料 -> 灵感 -> 永久知识”的方向和证据链。

### 建议

补充层级轴：

- 文献/证据在下。
- 灵感/加工中节点在中。
- 永久卡/结论在上。

连线方向和透明度应表达“支撑”或“沉淀”关系。

默认静止。

---

## 矩阵模式

### 当前问题最明显

用户觉得矩阵奇怪是合理的。

当前矩阵使用三维格子：

- x：cluster
- y：type
- z：degree band

但画面上没有坐标轴、没有格子边界、没有标签说明。用户不知道 x/y/z 分别代表什么，所以会觉得它只是乱排。

### 建议

矩阵模式只有在补齐这些元素后才成立：

- 横向轴标签：星团/主题。
- 纵向轴标签：卡片类型/知识层级。
- 第三维或颜色：连接强度。
- 每个格子有轻量边界。
- 点击格子可以聚焦该格子的节点集合。

如果短期没有时间打磨，建议矩阵先降级为实验模式，不作为演示重点。

---

## 任务流模式

### 当前优点

它已经能接入 `learningPath.steps`，如果有学习路径，会按步骤排序。

### 当前问题

没有学习路径时，它会 fallback 到按语义排序的前 24 个节点。这种 fallback 容易让任务流看起来像“随便排队”。

### 建议

任务流模式必须依赖明确路径：

- 学习路径
- 目标任务
- 先修关系
- 推荐步骤

如果没有路径，应显示空状态或提示选择学习路径，而不是自动伪造队列。

---

## 时间线模式

### 当前问题

当前 `GalaxyNode` 类型没有 `createdAt` 或 `updatedAt`。

`server/api/routes/galaxy.ts` 的 `/nodes` 也没有返回时间字段。

因此当前时间线并不是按真实时间排序，而是按 hash 排序。这不应该作为正式时间线。

### 建议

后端补充：

- `createdAt`
- `updatedAt`

前端类型补充：

- `createdAt?: string`
- `updatedAt?: string`

时间线按真实时间排序，并按类型分轨。没有真实时间前，时间线不应作为核心展示模式。

---

## 地形模式

### 当前问题

当前地形高度主要来自节点类型和学习路径状态。如果没有学习路径状态，它基本是在用类型猜测掌握程度。

这会让“地形”缺少可信数据来源。

### 建议

地形模式应该接入真实学习状态：

- 掌握度
- 复习次数
- 最近学习时间
- 测验结果
- 学习路径完成状态

否则建议暂时叫“状态地形”或放在实验视图，不要强调它代表真实掌握。

---

## 证据模式

### 当前优点

它按类型把文献、灵感、永久卡放在不同高度，方向正确。

### 当前问题

它没有真正使用边的类型来区分“支撑”“引用”“反例”“派生”。如果所有关系都只是普通连接，证据模式就无法证明“这个结论为什么可信”。

### 建议

证据模式需要重点使用 edge type：

- `prerequisite`
- `derived`
- `counter`
- `wikilink`
- `citation`
- `evidence`

点击永久卡时，只突出支撑它的文献和证据。点击文献时，突出它支撑的结论。

默认静止，强调阅读证据链。

---

## 建议的技术重构方向

## 1. 引入布局配置表

当前模式逻辑分散在多个函数里。建议引入统一配置：

```ts
const LAYOUT_PROFILES = {
  galaxy: {
    projection: '3d',
    motion: 'camera-orbit',
    atmosphere: { milkyWay: true, comets: true },
    focus: 'camera-selection',
    linkPolicy: 'overview',
  },
  flat: {
    projection: '2d',
    motion: 'none',
    atmosphere: { milkyWay: true, comets: false },
    focus: 'highlight-neighborhood',
    linkPolicy: 'semantic',
  },
}
```

每种模式应该同时定义：

- 布局算法
- 相机默认位置
- 点击行为
- 旋转策略
- 连线显示策略
- 装饰元素规则
- 是否需要真实业务数据

这样不会再出现“平面、环形、同心都被粗暴当成 planar”的问题。

## 2. 把连线更新改成动态 buffer

目标：

- 动画期间不 dispose/create geometry。
- 节点和连线同一帧贴合。
- 大图切换时不掉帧。

建议实现：

1. 创建线时保留固定长度 points buffer。
2. 每帧只写 `position.array`。
3. `position.needsUpdate = true`。
4. 动画结束后再做高质量曲线。

## 3. 把 focus 从相机逻辑里拆出来

当前 `focusNode()` 同时做了：

- 停止旋转
- 计算选中集合
- 移动相机
- 设置高亮
- 设置标签

建议拆成：

- `selectNode(node)`
- `highlightSelection(node, mode)`
- `updateLabels(node, mode)`
- `applyFocusCamera(node, mode)`
- `maybeRecomputeLayoutFromSelection(node, mode)`

这样同心模式点击节点时才能重排，而平面模式点击节点时可以只高亮。

## 4. 视觉装饰进入模式规则

不要把银河带和彗星继续作为普通开关暴露。

建议：

- 删除银河带开关。
- 删除彗星开关或放到高级设置。
- `applyGraphLayout()` 内部统一控制彗星显示。
- 彗星默认只在星系模式显示。

## 5. 布局 UI 按用途分组

`LAYOUTS` 应从技术列表变成认知视角列表：

```text
总览：星系
关系：平面 / 环形 / 同心
学习：分层 / 任务流 / 证据
分析：矩阵 / 时间线 / 地形
```

这样用户能理解“为什么有这么多模式”。

---

## 推荐实施顺序

### 第一阶段：先修用户马上能感知的问题

1. 连线和节点同步动画。
2. 彗星只在星系显示。
3. 银河带常开并移除开关。
4. 平面/任务流/时间线点击节点不再相机俯冲。
5. 环形/同心默认允许轻微中心旋转。

### 第二阶段：修模式边界

1. LAYOUTS 按用途分组。
2. 同心模式点击节点后重新以该节点为中心排布。
3. 矩阵补坐标轴、格子边界和标签。
4. 时间线接入真实 `createdAt/updatedAt`。
5. 任务流没有路径时不伪造队列。

### 第三阶段：修数据可信度

1. 地形接入真实掌握度。
2. 证据模式使用更明确的边类型。
3. 平面模式区分真实语义边和展示辅助边。
4. 每个模式提供对应的 tooltip 或一句话说明。

---

## 验收标准

### 布局切换

- 切换任意布局时，节点和连线必须同步移动。
- 不应出现线条明显滞后、断开或最后才跳到正确位置。
- 大于 100 个节点时，切换动画仍应保持可接受帧率。

### 装饰元素

- 银河带始终存在。
- 彗星默认只在星系模式出现。
- 分析模式中没有彗星干扰。

### 旋转

- 星系默认慢速自转。
- 环形和同心默认轻微中心旋转。
- 平面、矩阵、分层、任务流、时间线、证据默认静止。

### 点击节点

- 星系点击节点可以相机靠近。
- 平面点击节点只高亮关系，不打乱俯视结构。
- 同心点击节点后，该节点成为新中心。
- 任务流点击节点高亮前后步骤。
- 证据点击节点高亮证据链。

### 模式理解

- 用户能在 3 秒内理解当前模式是用来干什么的。
- 每个模式有明确的使用场景和边界。
- 矩阵、时间线、地形如果数据或视觉边界不完整，不应作为演示重点。

---

## 最终建议

不要把布局模式理解为“十种排列算法”。应该把它们理解为“十种认知视角”。

短期最值得打磨的是：

1. 星系：默认入口和整体展示。
2. 平面：真实关系分析。
3. 同心：单个节点上下文。
4. 环形：跨主题连接。
5. 证据：知识可信度。
6. 任务流：学习路径。

矩阵、时间线、地形不是不能要，而是需要更强的数据和视觉边界。否则它们会削弱整体品质。

如果只做一轮优化，优先修“连线同步 + 相机按模式适配 + 装饰默认规则”。这三项会立刻改善用户感知。
