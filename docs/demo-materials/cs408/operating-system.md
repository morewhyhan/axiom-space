## 4.1 操作系统概述

### 4.1.1 操作系统的定义

操作系统（Operating System, OS）是计算机系统中最基本、最重要的系统软件。它管理计算机的硬件和软件资源，为应用程序提供运行环境，并为用户提供用户界面。

**操作系统的四个角色**：
1. **资源管理者**：管理 CPU、内存、文件、I/O 设备等资源
2. **程序运行环境**：为应用程序提供抽象的资源视图
3. **用户接口**：提供命令行、图形界面等交互方式
4. **系统管家**：提供系统保护、安全认证等服务

### 4.1.2 操作系统的目标与功能

**资源管理**：
- **处理机管理**：进程调度、分配 CPU 时间
- **存储器管理**：内存分配、虚拟内存管理
- **文件管理**：文件存储、目录组织、权限控制
- **设备管理**：I/O 调度、缓冲、设备驱动

**程序控制**：
- 程序的加载、执行、终止
- 系统调用接口

**用户接口**：
- 命令接口（Shell、cmd）
- 图形用户接口（GUI）
- 程序接口（系统调用、API）

### 4.1.3 操作系统的特征

**并发（Concurrency）**：
- 多个程序在宏观上同时执行
- 单核 CPU 中在微观上交替执行
- 与并行（Parallelism）的区别：并发是交替执行，并行是同时执行

**共享（Sharing）**：
- **互斥共享**：一个资源一次只能被一个进程使用（如打印机）
- **同时共享**：一个资源可以被多个进程同时使用（如磁盘文件）

**虚拟（Virtualization）**：
- 物理资源抽象为逻辑资源
- 虚拟内存（把物理内存抽象为更大的虚拟地址空间）
- 虚拟处理器（每个进程认为自己独占 CPU）

**异步（Asynchronism）**：
- 程序执行的顺序和速度不可预知
- 进程以不可预知的速度向前推进
- 需要通过同步机制保证正确性

### 4.1.4 操作系统的发展历史

**手工阶段（1940s-1950s）**：
- 没有操作系统，程序员直接使用机器语言
- 插线板控制、纸带输入

**批处理阶段（1950s-1960s）**：
- **单道批处理**：内存中只有一道程序，CPU 等待 I/O
- **多道批处理**：内存中同时存在多道程序，CPU 在 I/O 等待时切换

**分时系统（1960s-1970s）**：
- 时间片轮转，每个用户获得交互式体验
- 代表：UNIX（1969年，贝尔实验室）

**实时系统**：
- 必须在规定时间内完成任务
- 硬实时：航空、医疗（错过截止时间就是灾难）
- 软实时：视频播放（偶尔错过可接受）

**现代操作系统**：
- 个人计算机：Windows、macOS、Linux
- 移动设备：Android（Linux内核）、iOS
- 服务器/云：Linux（Ubuntu、CentOS）、Windows Server

## 4.2 进程与线程

### 4.2.1 进程的概念

**进程（Process）**：程序的一次执行过程，是系统进行资源分配和调度的基本单位。

**程序 vs 进程**：

| 对比 | 程序 | 进程 |
|------|------|------|
| 概念 | 静态的指令集合 | 程序的一次动态执行 |
| 存储 | 保存在磁盘上 | 在内存中运行 |
| 生命周期 | 永久存在 | 从创建到终止 |
| 资源 | 不占用资源 | 占用 CPU、内存等资源 |
| 多个 | 一个程序可以有多个进程 | 一个进程对应一个程序的执行 |

**进程的组成**：

1. **程序段**：可执行的代码
2. **数据段**：程序运行时使用的数据
3. **进程控制块（PCB, Process Control Block）**：进程管理和控制所需的信息

**PCB 的内容**：
- **进程标识符**：PID（进程号）、PPID（父进程号）
- **进程状态**：就绪、运行、阻塞
- **程序计数器**：下条指令的地址
- **CPU 寄存器**：通用寄存器、栈指针
- **调度信息**：优先级、调度队列指针
- **内存管理信息**：页表指针
- **统计信息**：CPU 时间、使用资源

### 4.2.2 进程的状态与转换

**三种基本状态**：

```
          ┌──────────┐
          │   创建    │
          └────┬─────┘
               │ 创建
               ▼
    ┌──────────────────┐
    │    就绪（Ready）  │←──────────────┐
    └────────┬─────────┘               │
             │ 调度                     │
             ▼                          │
    ┌──────────────────┐    时间片用完   │
    │  运行（Running）  │───────────────┘
    └────────┬─────────┘
             │ 等待 I/O 或事件
             ▼
    ┌──────────────────┐
    │  阻塞（Blocked）  │───────────────┘
    └──────────────────┘  I/O 完成或事件发生
```

**五种状态**（增加创建和终止状态）：

```
创建 → 就绪 ↔ 运行 → 终止
              ↕
             阻塞
```

**状态转换**：
- **就绪→运行**：调度程序选中该进程
- **运行→就绪**：时间片用完或被高优先级进程抢占
- **运行→阻塞**：进程请求 I/O 或等待事件
- **阻塞→就绪**：I/O 完成或事件发生
- **创建→就绪**：进程创建完成，可以运行
- **运行→终止**：进程执行完毕或被强制终止

### 4.2.3 进程控制

**原语（Primitive）**：操作系统实现进程控制的基本操作。原语在执行过程中不可中断（关中断执行）。

**进程创建**：
1. 分配 PID 和 PCB
2. 分配进程地址空间
3. 初始化 PCB（状态设为就绪）
4. 将进程插入就绪队列

**进程终止**：
1. 回收进程占用的所有资源
2. 撤销 PCB

**进程阻塞**：
1. 保存进程的 CPU 上下文到 PCB
2. 修改进程状态为阻塞
3. 将进程插入相应事件的等待队列
4. 转调度程序

**进程唤醒**：
1. 将进程从等待队列移出
2. 修改进程状态为就绪
3. 将进程插入就绪队列

**进程切换**：
1. 保存当前进程的 CPU 上下文（PC、寄存器等）
2. 更新当前进程的 PCB
3. 恢复新进程的 CPU 上下文
4. 更新内存管理信息（如刷新 TLB）

### 4.2.4 线程（Thread）

**线程**：CPU 调度的基本单位，是进程内的一个执行流。

**进程 vs 线程**：

| 对比 | 进程 | 线程 |
|------|------|------|
| 资源分配 | 进程是资源分配的单位 | 线程不拥有独立资源 |
| 调度 | 线程是调度的单位 | 同一进程内线程切换轻量 |
| 地址空间 | 有独立地址空间 | 共享进程地址空间 |
| 切换开销 | 大（涉及地址空间切换） | 小（不切换地址空间） |
| 通信 | 需要 IPC 机制 | 可以直接读写共享内存 |
| 系统开销 | 创建进程开销大 | 创建线程开销小 |

**多线程模型**：

| 模型 | 用户线程 → 内核线程 | 说明 |
|------|-------------------|------|
| 多对一 | 多个用户线程 → 1个内核线程 | 线程切换快，但不能利用多核 |
| 一对一 | 1个用户线程 → 1个内核线程 | 利用多核，切换开销大 |
| 多对多 | 多个用户线程 → 多个内核线程 | 结合两者优点 |

**用户级线程 vs 内核级线程**：

| 比较 | 用户级线程 | 内核级线程 |
|------|-----------|-----------|
| 管理者 | 用户空间线程库 | 操作系统内核 |
| 切换 | 不需要系统调用 | 需要系统调用 |
| 对内核透明 | 是 | 否 |
| 多核利用 | 不能（内核只看到1个线程） | 能 |
| 阻塞影响 | 一个用户线程阻塞 → 所有线程阻塞 | 一个线程阻塞不影响其他线程 |

### 4.2.5 协程（Coroutine）

协程是一种用户态的轻量级线程，由程序员在代码中控制切换点。

**协程的特点**：
- 不依赖内核的调度
- 切换开销极低（不涉及系统调用）
- 通过 yield 和 resume 显式切换
- 通常基于事件循环或生成器实现

## 4.3 处理机调度

### 4.3.1 调度的层次

**高级调度（作业调度，Long-term Scheduling）**：
- 从外存的后备队列中选择作业加载到内存
- 频率低（秒级到分钟级）
- 控制多道程序的道数

**中级调度（内存调度，Medium-term Scheduling）**：
- 进程在内存和磁盘间换入换出（Swapping）
- 缓解内存压力

**低级调度（进程调度，Short-term Scheduling）**：
- 从就绪队列中选择一个进程占用 CPU
- 频率高（毫秒级）
- 直接影响系统性能

### 4.3.2 调度的时机

- 当前进程主动放弃 CPU（等待 I/O、执行完毕、调用阻塞原语）
- 当前进程被抢占（时间片用完、高优先级进程就绪）
- 发生中断（I/O 中断、时钟中断、系统调用）

**非抢占式调度**：进程一旦获得 CPU，除非主动放弃，否则一直运行。

**抢占式调度**：调度程序可以强行终止当前进程的执行。

### 4.3.3 调度算法评价指标

| 指标 | 定义 | 追求 |
|------|------|------|
| CPU 利用率 | CPU 忙的时间比例 | 最大化 |
| 吞吐量 | 单位时间完成的进程数 | 最大化 |
| 周转时间 | 进程从提交到完成的时间 | 最小化 |
| 平均周转时间 | 所有进程周转时间的平均值 | 最小化 |
| 等待时间 | 进程在就绪队列中等待的总时间 | 最小化 |
| 响应时间 | 从提交到首次响应的时间 | 最小化 |

**周转时间计算**：
```
周转时间 = 完成时间 - 提交时间（到达时间）
带权周转时间 = 周转时间 / 实际运行时间
```

### 4.3.4 常见调度算法

**1. 先来先服务（FCFS, First Come First Served）**：
- 按到达顺序调度
- 非抢占式
- 优点：实现简单、公平
- 缺点：短作业可能较长作业等待（convoy effect）

**2. 短作业优先（SJF, Shortest Job First）**：
- 选择运行时间最短的进程
- 可以是非抢占式或抢占式（SRTN, Shortest Remaining Time Next）
- 优点：最小化平均等待时间
- 缺点：可能造成长作业饥饿

**3. 优先级调度**：
- 选择优先级最高的进程
- 可以是静态优先级或动态优先级
- 问题：低优先级进程可能饥饿
- 解决方案：老化（Aging）—— 等待时间越久，优先级逐渐提高

**4. 时间片轮转（RR, Round Robin）**：
- 每个进程运行一个固定时间片
- 抢占式
- 优点：响应时间短，适合分时系统
- 缺点：时间片过大 → 退化为 FCFS；时间片过小 → 切换开销大

**5. 多级反馈队列（MFQ, Multi-level Feedback Queue）**：
- 设置多个就绪队列，不同队列有不同的优先级和时间片大小
- 新进程进入最高优先级队列
- 用完时间片未完成 → 降到下一级队列
- 综合了多种调度算法的优点

### 4.3.5 调度算法例题

**例题**：
> 有 4 个进程 P1-P4，到达时间和服务时间如下：
>
> | 进程 | 到达时间 | 服务时间 |
> |------|---------|---------|
> | P1 | 0 | 7 |
> | P2 | 2 | 4 |
> | P3 | 4 | 1 |
> | P4 | 5 | 4 |
>
> 分别用 FCFS、SJF（非抢占）、SRTN（抢占式SJF）、RR（时间片=2）求各进程的周转时间和平均周转时间。

**FCFS 解法**：

```
调度顺序：P1(0-7), P2(7-11), P3(11-12), P4(12-16)

进程  到达  开始  结束  周转
P1    0     0     7     7
P2    2     7     11    9
P3    4     11    12    8
P4    5     12    16    11

平均周转时间 = (7+9+8+11)/4 = 8.75
```

**SJF（非抢占）解法**：

```
0时刻：就绪队列 P1(7) → 选P1运行
P1完成后(7)：就绪队列 P2(4), P3(1), P4(4) → 选P3(最小)运行
P3完成后(8)：就绪队列 P2(4), P4(4) → 选P2(P4也可)
P2完成后(12)：P4开始运行(12-16)

进程  到达  开始  结束  周转
P1    0     0     7     7
P3    4     7     8     4
P2    2     8     12    10
P4    5     12    16    11

平均周转时间 = (7+4+10+11)/4 = 8
```

**RR(q=2) 解法**：

```
P1: (0-2) → 运行2，剩余5
P2: (2-4) → 到达，运行2，剩余2
P3: (4-5) → 到达，运行1（完成）
P1: (5-7) → 运行2，剩余3
P4: (7-9) → 到达，运行2，剩余2
P2: (9-11) → 运行2（完成）
P1: (11-13) → 运行2，剩余1
P4: (13-15) → 运行2（完成）
P1: (15-16) → 运行1（完成）

进程  完成  周转
P1    16    16
P2    11    9
P3    5     1
P4    15    10

平均周转时间 = (16+9+1+10)/4 = 9
```

## 4.4 进程同步与互斥

### 4.4.1 基本概念

**临界资源（Critical Resource）**：
一次只能被一个进程访问的资源（如打印机、共享变量）

**临界区（Critical Section）**：
访问临界资源的代码段

**互斥（Mutual Exclusion）**：
同一时刻最多只能有一个进程在临界区中执行

**同步（Synchronization）**：
多个进程按照一定的先后顺序执行

### 4.4.2 临界区问题的要求

1. **互斥**：同一时刻最多有一个进程在临界区中
2. **前进（Progress）**：当没有进程在临界区时，应允许等待进入的进程进入
3. **有限等待（Bounded Waiting）**：进程在发出进入请求后，等待时间有上限

### 4.4.3 软件实现方法

**Peterson 算法**：
```c
// 进程 i 的代码
int flag[2] = {0, 0}; // 是否想进入临界区
int turn;              // 轮到谁

// 进程 i（另一个是 1-i）
flag[i] = true;
turn = 1 - i;
while (flag[1-i] && turn == 1-i)
    ;  // 等待
// 临界区
// ...
flag[i] = false;  // 退出
```

- 正确性：满足互斥、前进、有限等待
- 缺点：只适用于两个进程，需要忙等待

### 4.4.4 硬件实现方法

**中断屏蔽**：
```c
关中断;
// 临界区
开中断;
```
- 简单，但实现复杂（多核无法同时屏蔽所有 CPU 的中断）

**Test-and-Set 指令**：
```c
// TSL 是原子操作
boolean TestAndSet(boolean *lock) {
    boolean old = *lock;
    *lock = true;  // 加锁
    return old;
}

// 使用
while (TestAndSet(&lock))
    ;  // 忙等待
// 临界区
lock = false;
```

**Swap 指令**：
```c
// 原子交换
void Swap(boolean *a, boolean *b) {
    boolean temp = *a;
    *a = *b;
    *b = temp;
}

// 使用
key = true;
while (key == true)
    Swap(&lock, &key);
// 临界区
lock = false;
```

### 4.4.5 信号量（Semaphore）

信号量是 Dijkstra 提出的经典同步机制。信号量 S 是一个整型变量，由 P 和 V 两个原语操作。

**信号量定义**：
```c
typedef struct {
    int value;          // 信号量值
    struct process *list; // 等待队列
} semaphore;
```

**P 操作（wait）**：
```c
void wait(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        // 将进程加入等待队列
        // 阻塞该进程
    }
}
```

**V 操作（signal）**：
```c
void signal(semaphore *S) {
    S->value++;
    if (S->value <= 0) {
        // 从等待队列中取出一个进程
        // 将其唤醒（放入就绪队列）
    }
}
```

**信号量实现互斥**：
```c
semaphore mutex = 1;

// 进程 P1
P(&mutex);
// 临界区
V(&mutex);

// 进程 P2
P(&mutex);
// 临界区
V(&mutex);
```

**信号量实现同步**：
```c
// 要求：P2 必须等 P1 执行完某段代码后才能执行
semaphore sync = 0;

// P1
// 执行代码...
V(&sync);  // 发出信号

// P2
P(&sync);  // 等待信号
// 执行后续代码...
```

### 4.4.6 经典同步问题

**1. 生产者-消费者问题**：

问题描述：生产者向缓冲区放入数据，消费者从缓冲区取出数据。缓冲区满时生产者等待，缓冲区空时消费者等待。

```c
semaphore mutex = 1;     // 互斥访问缓冲区
semaphore empty = n;     // 空闲缓冲区数量
semaphore full = 0;      // 满缓冲区数量

// 生产者
void producer() {
    while (true) {
        produce_item();
        P(&empty);       // 申请空位
        P(&mutex);       // 进入临界区
        put_item_to_buffer();
        V(&mutex);       // 退出临界区
        V(&full);        // 增加满位
    }
}

// 消费者
void consumer() {
    while (true) {
        P(&full);        // 申请满位
        P(&mutex);       // 进入临界区
        get_item_from_buffer();
        V(&mutex);       // 退出临界区
        V(&empty);       // 增加空位
        consume_item();
    }
}
```

> **注意**：P 操作的顺序不能互换。如果先 P(mutex) 再 P(empty)，当缓冲区满时，生产者持有 mutex 但无法申请到 empty，而消费者需要 mutex 才能取数据，造成死锁。

**2. 读者-写者问题**：

问题描述：多个读者可以同时读共享数据，但写者必须独占访问。

**读者优先方案**：
```c
semaphore rw = 1;        // 写者互斥
semaphore mutex = 1;     // 保护 read_count
int read_count = 0;      // 读者数量

// 读者
void reader() {
    while (true) {
        P(&mutex);
        read_count++;
        if (read_count == 1)
            P(&rw);      // 第一个读者加锁
        V(&mutex);
        
        read_data();
        
        P(&mutex);
        read_count--;
        if (read_count == 0)
            V(&rw);      // 最后一个读者解锁
        V(&mutex);
    }
}

// 写者
void writer() {
    while (true) {
        P(&rw);          // 申请写权限
        write_data();
        V(&rw);
    }
}
```

**3. 哲学家就餐问题**：

问题描述：5 个哲学家围坐圆桌，每人之间有一根筷子。哲学家需要拿起左右两根筷子才能吃饭。

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};

// 哲学家 i（可能导致死锁的版本）
void philosopher(int i) {
    while (true) {
        P(&chopstick[i]);           // 拿左筷子
        P(&chopstick[(i+1)%5]);     // 拿右筷子
        eat();
        V(&chopstick[i]);
        V(&chopstick[(i+1)%5]);
        think();
    }
}
// 问题：所有哲学家都拿了左筷子 → 死锁
```

解决方案：
1. 只允许 4 个哲学家同时进餐
2. 奇偶号哲学家拿筷子顺序相反
3. 只有当两根筷子都可用时才拿起

### 4.4.7 管程（Monitor）

管程是一种高级的同步机制，将共享数据和操作封装在一起。

**管程的特性**：
1. 管程内的过程互斥执行
2. 管程内可以设置条件变量，支持 wait 和 signal 操作
3. 一次只有一个进程能进入管程

Java 中的 `synchronized` 关键字、C# 中的 `lock` 语句都是管程的实现。

## 4.5 死锁

### 4.5.1 死锁的定义与条件

**死锁（Deadlock）**：一组进程中的每个进程都在等待只能由该组中其他进程才能引发的事件。

**死锁的四个必要条件**（必须同时满足）：

1. **互斥条件**：资源一次只能被一个进程使用
2. **不可剥夺条件**：已分配的资源在未使用完前不能被强制剥夺
3. **请求与保持条件**：进程在请求新资源的同时保持已分配的资源
4. **循环等待条件**：存在进程-资源的环形等待链

### 4.5.2 死锁的处理策略

**四种策略**：

| 策略 | 方法 | 资源利用率 | 实现难度 |
|------|------|-----------|---------|
| 预防 | 破坏四个必要条件之一 | 低 | 简单 |
| 避免 | 动态检测安全状态 | 中 | 中等 |
| 检测 | 允许死锁发生，定期检测 | 高 | 中等 |
| 恢复 | 检测到死锁后解除 | 高 | 较难 |

### 4.5.3 死锁预防（Deadlock Prevention）

**破坏互斥条件**：
- 将资源改为共享（不可行，很多资源本质是互斥的）

**破坏不可剥夺条件**：
- 进程请求资源失败时，释放所有已占有的资源
- 缺点：可能导致前功尽弃

**破坏请求与保持条件**：
- 进程在运行前一次性申请所有需要的资源
- 缺点：资源利用率低，容易饥饿

**破坏循环等待条件**：
- 对所有资源类型进行编号，进程按递增顺序请求资源
- 缺点：资源编号难以确定

### 4.5.4 死锁避免（Deadlock Avoidance）

**安全状态**：存在一种进程执行顺序（安全序列），使得所有进程都能顺利完成。

**银行家算法**（Dijkstra提出）：

**数据结构**：
- `Available[m]`：各资源的可用数量
- `Max[n][m]`：各进程对每种资源的最大需求
- `Allocation[n][m]`：各进程已分配的资源数
- `Need[n][m]`：各进程还需要的资源数（Need = Max - Allocation）

**进程请求资源**：
1. 检查请求 ≤ Need，否则出错
2. 检查请求 ≤ Available，否则等待
3. 试分配：修改 Available、Allocation、Need
4. 检查安全性：运行安全算法，如果安全则正式分配，否则回滚

**安全算法**：
```
Work = Available
Finish = {false, false, ..., false}
while (存在 i 使 Finish[i]==false && Need[i] <= Work):
    Work = Work + Allocation[i]
    Finish[i] = true
if (所有 Finish[i] == true):
    系统处于安全状态
else:
    系统处于不安全状态
```

**例题**：
> 系统中有 5 个进程 P0-P4，3 类资源 A(10), B(5), C(7)。当前 Allocation 和 Need 如下：
> Allocation: P0(0,1,0), P1(2,0,0), P2(3,0,2), P3(2,1,1), P4(0,0,2)
> Need: P0(7,4,3), P1(1,2,2), P2(6,0,0), P3(0,1,1), P4(4,3,1)
>
> 求 Available，并判断当前是否安全。
>
> 解：
> Available = (10,5,7) - Σ(Allocation) = (10,5,7) - (7,2,5) = (3,3,2)
>
> 安全检查：
> - P1: Need(1,2,2) ≤ Work(3,3,2) ✓ → Work=(5,3,2)
> - P3: Need(0,1,1) ≤ Work(5,3,2) ✓ → Work=(7,4,3)
> - P0: Need(7,4,3) ≤ Work(7,4,3) ✓ → Work=(7,5,3)
> - P2: Need(6,0,0) ≤ Work(7,5,3) ✓ → Work=(10,5,5)
> - P4: Need(4,3,1) ≤ Work(10,5,5) ✓ → Work=(10,5,7)
>
> 安全序列：P1, P3, P0, P2, P4 → 安全

### 4.5.5 死锁检测与恢复

**资源分配图（Resource Allocation Graph, RAG）**：
- 圆形节点 = 进程
- 方形节点 = 资源（圆点表示实例数量）
- 从进程到资源的边 = 请求边
- 从资源到进程的边 = 分配边

**死锁检测**：
- 如果资源分配图中有环 → 可能有死锁
- 每个资源类型只有单个实例时，有环 = 有死锁
- 每个资源类型有多个实例时，需要进一步分析

**死锁恢复**：
1. **进程终止**：终止所有死锁进程或逐个终止直到死锁解除
2. **资源剥夺**：从进程中剥夺资源给其他进程
3. **进程回滚**：回滚到安全状态

## 4.6 内存管理

### 4.6.1 内存管理的目标

1. **地址空间独立**：每个进程有独立的地址空间
2. **地址映射**：将逻辑地址转换为物理地址
3. **内存分配**：为进程分配和回收内存
4. **内存保护**：进程不能访问其他进程的内存
5. **内存扩充**：虚拟内存技术

### 4.6.2 地址空间与地址转换

**逻辑地址（Logical Address）**：
- CPU 生成的地址（程序视角）
- 也称为虚拟地址

**物理地址（Physical Address）**：
- 实际访问内存的地址（硬件视角）

**地址转换**：
```
逻辑地址 → 地址转换（MMU） → 物理地址
```

**地址转换的时机**：
- **编译时**：编译时就确定物理地址（早期单道批量系统）
- **加载时**：加载程序时重定位（代码中不含绝对地址）
- **运行时**：程序运行过程中动态转换（需要 MMU 硬件支持）

### 4.6.3 连续内存分配

**单一连续分配**：
- 内存分为系统区和用户区
- 缺点：只支持单道程序

**固定分区分配**：
- 内存分为若干个固定大小的分区
- 缺点：内碎片（分区内部碎片）

**动态分区分配**：
- 根据进程需要动态划分分区
- 三种常见分配算法：

| 算法 | 查找策略 | 优点 | 缺点 |
|------|---------|------|------|
| 首次适应（First Fit） | 找到第一个够大的空闲块 | 简单、快速 | 容易产生碎片 |
| 最佳适应（Best Fit） | 找到刚好够用的最小空闲块 | 内部碎片最小 | 需要排序，产生小碎片 |
| 最差适应（Worst Fit） | 找到最大的空闲块 | 碎片较小 | 大分区被快速分割 |

**外部碎片（External Fragmentation）**：各分区之间的小空闲块无法被利用。
**内部碎片（Internal Fragmentation）**：分配给进程的块中未被使用的空间。

**解决外部碎片的方法**：
- **紧凑（Compaction）**：移动所有进程，使空闲块连成一片

### 4.6.4 分页存储管理

**基本思想**：
- 将逻辑地址空间划分为固定大小的页（Page）
- 将物理内存划分为同样大小的页框（Frame）
- 通过页表将逻辑页号映射到物理页框号

**地址结构**：
```
逻辑地址 = 页号(P) + 页内偏移(W)
物理地址 = f(P) + W
```

**页号与页内偏移的计算**：
```
页号 = 逻辑地址 / 页大小（取整）
页内偏移 = 逻辑地址 % 页大小
```

**页表**：
```
页表：页号 → 页框号（块号）

页号 0 → 页框号 2
页号 1 → 页框号 5
页号 2 → 页框号 7
...
```

**地址转换过程**：
```
逻辑地址 → 提取页号 → 查页表得页框号 → 拼接页内偏移 → 物理地址
```

**例题**：
> 页面大小为 4KB，逻辑地址为 8197，求页号和页内偏移。
>
> 解：
> 4KB = 4096B
> 页号 = 8197 / 4096 = 2
> 页内偏移 = 8197 % 4096 = 5
> 如果页表显示页号2对应页框号6，则物理地址 = 6×4096 + 5 = 24581

### 4.6.5 分段存储管理

**基本思想**：
- 按程序逻辑将地址空间划分为多个段（代码段、数据段、堆、栈）
- 每个段有独立的基址和长度限制
- 段表将段号映射到段基址和段长

**段地址结构**：
```
逻辑地址 = 段号(S) + 段内偏移(D)
```

**段表**：
```
段号 → (段基址, 段长)

段号 0(代码段) → (1000, 800)
段号 1(数据段) → (2000, 400)
段号 2(堆栈段) → (3500, 200)
```

**分段 vs 分页**：

| 比较 | 分页 | 分段 |
|------|------|------|
| 划分方式 | 硬件划分（固定大小） | 程序逻辑划分（大小可变） |
| 对程序员 | 透明 | 可见 |
| 地址空间 | 一维 | 二维（段号+偏移） |
| 保护 | 同一页内的权限相同 | 不同段有不同权限 |
| 共享 | 难以共享 | 容易共享（如共享代码段） |
| 碎片 | 有内部碎片 | 有外部碎片 |

### 4.6.6 段页式存储管理

结合段和页的优点：
1. 按段划分逻辑地址空间（段内连续）
2. 段内再分页（段内不连续）
3. 地址映射：段号 → 页表基址 → 页号 → 页框号 → 物理地址

**地址结构**：
```
逻辑地址 = 段号(S) + 段内页号(P) + 页内偏移(W)
```

## 4.7 虚拟内存

### 4.7.1 虚拟内存的基本概念

**虚拟内存（Virtual Memory）**：将内存和磁盘结合起来，为进程提供一个比物理内存大得多的地址空间。

**虚拟内存的基础——局部性原理**：
- 程序在运行过程中，一段时间内只访问一部分数据
- 不需要将整个程序同时加载到内存

**虚拟内存的核心思想**：
- 进程运行时，只将需要的部分加载到内存
- 当前不需要的部分留在磁盘上
- 当访问的页面不在内存时，从磁盘调入
- 当内存不足时，将部分页面换出到磁盘

### 4.7.2 请求分页（Demand Paging）

**页表项的扩展**：
```
┌──────┬─────┬─────┬─────┬─────┬──────────┐
│ 页框号 │ 存在位│ 修改位│ 访问位│ 保护位│ 其他控制位│
└──────┴─────┴─────┴─────┴─────┴──────────┘
```

- **存在位**：页面是否在内存中（0 → 缺页）
- **修改位**：页面是否被修改过（脏页，换出时需要写回磁盘）
- **访问位**：页面是否被访问过（用于页面置换）

**缺页中断（Page Fault）**：

过程：
1. CPU 访问虚拟地址，MMU 查页表
2. 发现存在位为 0（页面不在内存）
3. CPU 陷入操作系统（缺页中断）
4. 操作系统选择空闲页框
5. 如果没有空闲页框，执行页面置换算法
6. 从磁盘将所需页面读入选中的页框
7. 更新页表（存在位置1）
8. 重新执行引起缺页的指令

**有效访问时间**：
```
EAT = (1 - p) × 内存访问时间 + p × 缺页处理时间
```
其中 p 为缺页率

**例题**：
> 内存访问时间 100ns，缺页处理时间 5ms，缺页率为 1%，求有效访问时间。
>
> 解：
> EAT = 0.99 × 100ns + 0.01 × 5ms
>     = 99ns + 50000ns = 50099ns ≈ 0.05ms
> 缺页使速度降低了 500 倍！

### 4.7.3 页面置换算法

**1. 最佳置换（OPT, Optimal Replacement）**：
- 淘汰未来最长时间不会被访问的页面
- 无法实现（无法预测未来），作为理论上的性能参考

**2. 先进先出（FIFO）**：
- 淘汰最早进入内存的页面
- 实现简单
- **Belady 异常**：增加页框数反而增加缺页率

**3. 最近最久未使用（LRU, Least Recently Used）**：
- 淘汰最久未使用过的页面
- 符合局部性原理，性能接近 OPT
- 硬件实现成本高（需要计数器或栈）

**4. 时钟置换（Clock / NRU, Not Recently Used）**：
- 每个页面有一个使用位（访问时置1）
- 时钟指针循环扫描页面：
  - 如果使用位为1，清0并继续
  - 如果使用位为0，淘汰该页面
- 是 LRU 的近似，容易实现

**5. 最不常用（LFU, Least Frequently Used）**：
- 淘汰被访问次数最少的页面
- 需要计数器记录访问频率

**例题**：

缺页次数：12

### 4.7.4 帧分配

**分配策略**：
- **固定分配**：每个进程分配固定数量的页框
- **可变分配**：随着进程运行调整分配数量

**置换范围**：
- **全局置换**：从整个系统挑选要置换的页面
- **局部置换**：从当前进程自己的页面中置换

**分配算法**：
- **等分分配**：所有进程均分
- **按比例分配**：根据进程大小分配
- **优先级分配**：高优先级进程获得更多页框

**抖动（Thrashing）**：
- 进程频繁缺页，大部分时间都花在页面置换上
- 页面置换开销远大于实际计算
- 原因：分配给进程的页框数太少，无法满足局部性需求
- 解决方法：工作集模型

**工作集（Working Set）**：
- 进程在时间窗口 Δ 内访问的页面集合
- 工作集大小 = 时间窗口内访问的不同页面数
- 操作系统应该保证分配给进程的页框数 ≥ 工作集大小

## 4.8 文件系统

### 4.8.1 文件与文件系统

**文件（File）**：一组有意义的信息或数据的集合。

**文件属性**：
- 文件名、标识符、类型、大小
- 创建时间、最后修改时间、最后访问时间
- 文件权限（读、写、执行）
- 文件所有者和组

**文件系统（File System）**：
- 负责文件的组织、存储、检索、保护和共享
- 逻辑结构：文件目录
- 物理结构：文件在磁盘上的存储方式

### 4.8.2 文件的逻辑结构

**有结构文件（记录文件）**：
- **顺序文件**：记录按顺序排列，存取快但插入困难
- **索引文件**：建立索引表加速查找
- **索引顺序文件**：索引文件 + 顺序文件的折中

**无结构文件（流式文件）**：
- 文件由字节流组成，无内部结构
- UNIX/Linux 的文件采用此方式

### 4.8.3 文件目录

**目录结构的发展**：

1. **单级目录**：所有文件在一个目录中（名字冲突问题）
2. **两级目录**：每个用户一个目录（但不能嵌套）
3. **树形目录**：允许嵌套目录（现代操作系统使用）
4. **无环图目录**：允许共享（通过软/硬链接）

**目录项的内容**：
- 文件名
- 文件控制块（FCB）或索引节点指针
- 文件属性

**路径**：
- **绝对路径**：从根目录开始的完整路径（如 /home/user/file.txt）
- **相对路径**：从当前目录开始的路径（如 ./file.txt）

**目录操作**：
- 创建、删除目录
- 列目录
- 改变当前目录

### 4.8.4 文件的物理结构

文件在磁盘上的存储方式：

**1. 连续分配**：
```
文件 → 连续的磁盘块：块10, 块11, 块12, 块13, ...
```
- 优点：随机访问快，顺序读性能好
- 缺点：外部碎片，文件难扩展

**2. 链接分配**：
```
文件 → 块10 → 块25 → 块34 → 块15 → ...
```
- **隐式链接**：每个块存有下一个块的指针
  - 缺点：只支持顺序访问
- **显式链接（FAT）**：所有链接信息集中存放在文件分配表（FAT）中
  - FAT 表：每个表项存放下一块的地址
  - 优点：支持随机访问（读 FAT 即可）
  - 缺点：FAT 表占用内存空间

**3. 索引分配**：
```
文件 → 索引块 [10, 25, 34, 15, ...]
```
- 索引块存放指向文件各数据块的指针
- 优点：支持随机访问，没有外部碎片
- **多级索引**：索引块存指向下一级索引块的指针（UNIX 文件系统使用）
- **混合索引**：小文件用直接索引，大文件用多级索引

**UNIX/Linux 文件存储结构（inode）**：
```
inode
├── 12个直接块指针 → 小文件直接访问
├── 1个一级间接块指针 → 指向索引块，索引块指向数据块
├── 1个二级间接块指针 → 指向索引块，...指向索引块，...指向数据块
└── 1个三级间接块指针 → 超大文件使用
```

### 4.8.5 文件存储空间管理

**位图法（Bit Map）**：
- 每个磁盘块对应一个位（0=空闲，1=已占用）
- 内存中维护位图，查找空闲块快

**空闲链表法**：
- 将空闲块链接起来
- 空闲表：使用空闲块范围表

### 4.8.6 文件共享与保护

**硬链接（Hard Link）**：
- 多个目录项指向同一个 inode
- 文件删除（unlink）时，减少引用计数
- 不能跨文件系统

**软链接（符号链接，Symbolic Link）**：
- 创建一个特殊文件，内容指向原文件的路径
- 可以跨文件系统
- 原文件删除后，软链接失效

**文件访问控制**：
- **ACL（访问控制列表）**：为每个用户/组设置权限
- **UNIX 权限模型**：所有者/组/其他人 + 读/写/执行

## 4.9 设备管理

### 4.9.1 I/O 系统概述

**I/O 设备的分类**：

按传输速度：
- 低速（键盘、鼠标）：几字节/秒
- 中速（打印机、扫描仪）：几千字节/秒
- 高速（磁盘、显示器）：几兆字节/秒以上

按数据传输方式：
- 字符设备：按字符传输（键盘、鼠标）
- 块设备：按块传输（磁盘）

按共享属性：
- 独占设备（打印机）
- 共享设备（磁盘）
- 虚拟设备（通过 SPOOLing 将独占设备虚拟化）

### 4.9.2 I/O 控制方式

**1. 程序直接控制方式**：
```c
// CPU 循环检查设备状态
while (!device_ready())
    ;  // 忙等待
data = read_device();
```
- CPU 利用率极低

**2. 中断驱动方式**：
```
CPU 发起 I/O 操作后，继续执行其他程序
设备完成 I/O 后发送中断
CPU 响应中断，执行中断处理程序
```
- CPU 利用率较高
- 每字节/每字都要中断，对高速设备开销太大

**3. DMA 方式**：
```
CPU 设置 DMA 控制器（缓冲区地址、传送字节数）
DMA 控制器直接在设备和内存之间传输数据
传输完成后发送中断通知 CPU
```
- 按块传输，每块只中断一次
- CPU 仅在开始和结束时参与

**4. 通道控制方式**：
- 通道是专门负责 I/O 的处理器
- 有自己的指令系统
- 可以执行通道程序，独立管理 I/O 操作

### 4.9.3 I/O 软件层次

```
用户级 I/O 软件（库函数：printf, scanf, read, write）
        ↓
设备无关的操作系统层（缓冲区管理、设备命名、保护）
        ↓
设备驱动程序（与硬件交互，处理中断）
        ↓
中断处理程序
        ↓
硬件设备
```

### 4.9.4 缓冲技术

**缓冲的目的**：
- 平衡 CPU 和 I/O 设备的速度差异
- 减少中断次数
- 提高数据传输出效率

**缓冲区类型**：
- **单缓冲**：一个缓冲区，处理时间和传输时间可以重叠一部分
- **双缓冲**：两个缓冲区交替使用，进一步减少等待时间
- **循环缓冲**：多个缓冲区组成环形
- **缓冲池**：系统管理的多个缓冲区集合

### 4.9.5 SPOOLing 技术

**SPOOLing（假脱机操作，Simultaneous Peripheral Operation Online）**：

将独占设备（如打印机）虚拟化为共享设备。

**实现**：
- 当进程请求打印机时，SPOOLing 系统在磁盘上为其分配一个输出文件
- 进程的输出暂存在磁盘上（输出井）
- SPOOLing 守护进程依次将输出井中的数据发送到打印机

**优点**：
- 将独占设备变为共享设备
- 进程不需要等待慢速的物理设备

### 4.9.6 磁盘调度算法

磁盘访问时间 = 寻道时间 + 旋转延迟 + 传输时间

寻道时间占主要部分，磁盘调度算法的核心是优化寻道时间。

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| FCFS | 按请求顺序 | 公平 | 平均寻道长 |
| SSTF | 选择离当前位置最近的请求 | 平均寻道短 | 饥饿（远的可能一直等） |
| SCAN（电梯算法） | 沿一个方向移动到尽头，然后反向 | 磁头利用率高 | 两端请求等待长 |
| C-SCAN（循环扫描） | 沿一个方向移动到尽头后快速返回 | 等待时间更均匀 | 返回时不处理请求 |
| LOOK | SCAN 的改进，移到最远请求就反向 | 更高效 | - |

**例题**：
> 磁盘有 200 个磁道，当前磁头在 100 号磁道，向 0 方向移动。
> 请求序列：98, 183, 37, 122, 14, 124, 65, 67
> 分别用 FCFS、SSTF、SCAN 计算寻道距离。
>
> FCFS：|100-98| + |98-183| + |183-37| + |37-122| + |122-14| + |14-124| + |124-65| + |65-67|
>       = 2 + 85 + 146 + 85 + 108 + 110 + 59 + 2 = 597
>
> SSTF：100→98(2) → 65(33) → 67(2) → 37(30) → 14(23) → 122(108) → 124(2) → 183(59)
>       总距离 = 2+33+2+30+23+108+2+59 = 259
>
> SCAN（向0方向）：100→98(2)→65(33)→37(28)→14(23)→0(14) → 67(67)→122(55)→124(2)→183(59)
>       总距离 = 2+33+28+23+14+67+55+2+59 = 283

## 4.11 补充：操作系统深入专题

### 4.11.1 系统调用实现原理

系统调用是用户程序请求操作系统服务的唯一合法途径。

**系统调用的过程**（以 read 为例）：
```
用户程序调用 read() → C库函数 → 陷入内核（trap）
  ① 将系统调用号和参数存入指定寄存器
  ② 执行 trap/int 指令（切换到内核态）
  ③ 内核根据系统调用号跳转到对应服务例程
  ④ 执行实际的内核操作
  ⑤ 将结果返回给用户程序
  ⑥ 执行 iret/return-from-trap（切换回用户态）
```

**用户态与内核态的切换**：
- **用户态**：受限的指令执行环境，不能执行特权指令
- **内核态**：可以执行所有指令，访问所有资源
- 切换开销：几百纳秒（保存/恢复上下文、TLB刷新）

**Linux 系统调用的数量**：
- x86_64 架构：约 350 个系统调用
- 常见调用：read、write、open、close、fork、execve、mmap

### 4.11.2 中断系统深入

**中断优先级与中断屏蔽**：

中断处理过程中，中断优先级决定了先处理哪个中断。

**中断屏蔽字**：
- 每个中断源设置一个屏蔽位
- 屏蔽高优先级中断 → 实现互斥
- 在临界区中关中断 → 防止死锁

**中断嵌套**：
```
低级中断 → 正在处理
    ↑ 高级中断到达 → 暂停低级，处理高级 → 恢复低级
    ↑ 更高级中断 → 继续嵌套...
```

**Linux 中断处理**：
- **上半部（Top Half）**：快速响应，只做最少必要工作
- **下半部（Bottom Half）**：延后处理耗时操作（软中断、tasklet、工作队列）

### 4.11.3 实时调度算法

实时系统要求任务在截止时间（Deadline）之前完成。

**硬实时 vs 软实时**：
- **硬实时**：错过截止时间 → 灾难性后果（航空航天、医疗）
- **软实时**：偶尔错过可接受（视频播放、在线游戏）

**速率单调调度（RMS, Rate Monotonic Scheduling）**：
- 静态优先级调度
- 周期越短（速率越高），优先级越高
- 可调度条件：U = Σ(C_i/T_i) ≤ n(2^(1/n) - 1)
  - C_i：任务执行时间，T_i：任务周期
  - 当 n→∞ 时，U_max ≈ ln(2) ≈ 69%

**最早截止时间优先（EDF, Earliest Deadline First）**：
- 动态优先级调度
- 截止时间越近，优先级越高
- 可调度条件：U = Σ(C_i/T_i) ≤ 1
- 比 RMS 利用率更高，但需要更多开销

**例题**：
> 两个实时任务 T1(周期=10, 执行时间=3), T2(周期=15, 执行时间=6)
> 用 RMS 和 EDF 判断是否可调度。
>
> RMS：
> U = 3/10 + 6/15 = 0.3 + 0.4 = 0.7
> n = 2 时 U_max = 2(2^(1/2)-1) ≈ 0.828
> 0.7 < 0.828 → RMS 可调度
>
> EDF：
> U = 0.7 < 1 → EDF 可调度

### 4.11.4 大页内存（Huge Pages）

现代 CPU 的默认页大小为 4KB。对于大内存应用（数据库、虚拟机），使用更大页面可以提高性能。

**大页的优势**：
- 减少 TLB 缺失次数（同样大小的 TLB 覆盖更多内存）
- 减少页表级数
- 提高内存访问性能

**大页大小**：
| 架构 | 标准页 | 大页 | 巨页 |
|------|-------|------|------|
| x86_64 | 4KB | 2MB | 1GB |
| ARM64 | 4KB | 2MB | 1GB |

**透明大页（THP, Transparent Huge Pages）**：
- Linux 内核自动合并相邻页为大页
- 应用程序无需修改

**性能对比示例**（数据库场景）：
```
4KB 页：访问 2GB 数据需要 512K 页表项，TLB 覆盖有限
2MB 页：访问 2GB 数据需要 1024 页表项，TLB 覆盖大幅增加
    测试表明：大页可使数据库查询性能提升 5-15%
```

### 4.11.5 内存映射文件（mmap）

mmap 将文件直接映射到进程的虚拟地址空间，实现文件 I/O 和内存操作的统一。

**mmap 的工作原理**：
```
虚拟地址空间 ← 映射 → 文件
进程读写这段内存 → 内核自动读写文件（按页进行）
```

**mmap 的优点**：
- **零拷贝**：读写文件不需要从内核到用户空间再复制
- **延迟加载**：只加载被访问的部分（通过缺页机制）
- **高效共享**：多个进程映射同一个文件时共享物理内存

```c
// 将文件映射到内存
void *addr = mmap(NULL, length, PROT_READ | PROT_WRITE,
                  MAP_SHARED, fd, 0);
// 像操作内存一样操作文件
memcpy(addr, data, size);
// 解除映射
munmap(addr, length);
```

**mmap 的关键应用**：
- **文件 I/O 高性能实现**：数据库存储引擎
- **进程间通信（IPC）**：多个进程映射同一文件
- **动态库加载**：共享库的代码段通过 mmap 加载到进程空间

**mmap vs 传统文件 I/O**：

| 对比 | 传统I/O (read/write) | mmap |
|------|--------------------|------|
| 数据拷贝 | 内核→用户缓冲区（至少一次拷贝） | 无需拷贝 |
| 系统调用 | 每次读/写都需要 | 一次 mmap + 缺页 |
| 随机访问 | 需要 seek | 任意偏移直接访问 |
| 小文件 | 效率尚可 | 可能浪费页空间 |

### 4.11.6 虚拟化技术

**虚拟化的层次**：

| 虚拟化类型 | 说明 | 代表 |
|-----------|------|------|
| 指令集虚拟化 | 模拟不同架构的指令集 | QEMU（纯软件模拟） |
| 硬件辅助虚拟化 | CPU硬件支持VM运行 | Intel VT-x, AMD-V |
| 操作系统级虚拟化 | 共享内核，隔离用户空间 | Docker, LXC |

**Hypervisor（虚拟机监控器）类型**：

**Type 1（裸机型）**：
```
硬件 → Hypervisor → 多个虚拟机
```
- 直接运行在硬件上
- 性能损耗小
- VMware ESXi, Xen, KVM

**Type 2（宿主型）**：
```
硬件 → 宿主机OS → Hypervisor → 多个虚拟机
```
- 运行在操作系统之上
- 方便，但性能损耗大
- VirtualBox, VMware Workstation

**容器技术（Container）**：

容器与虚拟机共享宿主机的操作系统内核

```
VM：硬件 → Hypervisor → 多个Guest OS → 应用
容器：宿主机OS → Docker Engine → 多个容器应用
```

| 对比 | 虚拟机 | 容器 |
|------|--------|------|
| 启动时间 | 分钟级 | 毫秒级 |
| 镜像大小 | GB级 | MB级 |
| 资源占用 | 每个VM有完整OS | 共享宿主机内核 |
| 隔离性 | 强（完全隔离） | 弱（共享内核） |
| 性能 | 有损耗 | 接近原生 |

**Docker 核心技术**：
- **Namespace**：隔离进程、网络、文件系统等视图
- **Cgroup**：限制和监控资源使用（CPU、内存、磁盘）
- **UnionFS**：分层镜像存储，高效复用

**KVM（基于内核的虚拟机）**：
- Linux 内核模块，将 Linux 变成 Type 1 Hypervisor
- 使用 Intel VT-x / AMD-V 硬件加速
- QEMU + KVM 是 Linux 上最流行的虚拟化组合

### 4.11.7 Linux 内核架构概览

**Linux 内核架构图**：
```
┌─────────────────────────────────────────────────────┐
│                    系统调用接口                        │
├─────────────────────────────────────────────────────┤
│       进程管理      │   内存管理    │   文件系统      │
│   ┌──────────────┐ │ ┌──────────┐ │ ┌────────────┐ │
│   │ 调度器 (CFS) │ │ │页面分配器 │ │ │ VFS        │ │
│   │ 进程/线程    │ │ │ 虚拟内存  │ │ │ ext4/btrfs │ │
│   │ IPC/信号     │ │ │  页换出   │ │ │  网络FS    │ │
│   └──────────────┘ │ └──────────┘ │ └────────────┘ │
├────────────────────┼──────────────┼────────────────┤
│    网络协议栈      │  设备驱动    │  安全模块       │
│  ┌──────────────┐  │ ┌──────────┐ │ ┌────────────┐ │
│  │ TCP/IP协议栈 │  │ │字符设备   │ │ │ SELinux    │ │
│  │ 网络调度     │  │ │块设备     │ │ │ AppArmor   │ │
│  │ 防火墙       │  │ │网络设备   │ │ │ Capabilities││
│  └──────────────┘  │ └──────────┘ │ └────────────┘ │
├────────────────────┴──────────────┴────────────────┤
│                 中断处理 / 定时器                     │
├─────────────────────────────────────────────────────┤
│              硬件抽象层（体系结构相关代码）             │
└─────────────────────────────────────────────────────┘
```

**Linux 调度器（CFS, Completely Fair Scheduler）**：
- 基于虚拟运行时间（vruntime）的调度策略
- 使用红黑树管理就绪进程
- vrunime 最小的进程被优先调度
- 优先级由 nice 值决定（-20 ~ 19）
- 时间复杂度：O(log n)（n为就绪进程数）

**内核同步机制**：
| 机制 | 特点 | 适用场景 |
|------|------|---------|
| 自旋锁 | 忙等待，不睡眠 | 短临界区，关抢占 |
| 互斥锁 | 睡眠等待 | 长临界区 |
| 读写锁 | 读者并发，写者独占 | 读多写少 |
| RCU | 无锁读，延迟释放写 | 读极多，写极少 |
| 信号量 | 计数值控制 | 资源计数 |
| 原子操作 | 硬件保证 | 简单计数器 |

**设备驱动模型的三个组件**：
- **总线（Bus）**：设备连接的方式（PCI、USB、I2C）
- **设备（Device）**：物理设备或虚拟设备
- **驱动（Driver）**：控制设备的软件代码
