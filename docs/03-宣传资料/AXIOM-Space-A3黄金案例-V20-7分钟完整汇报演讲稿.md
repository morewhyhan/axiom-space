# 0:40-1:40｜二、行业痛点与需求分析

## PPT 上屏内容

主标题：

> 现有 AI 学习工具，  
> 解决了“输入”，没解决“掌握”

三列对比：

| 工具 | 能做什么 | 留下的问题 |
|---|---|---|
| 通用 AI | 解释、问答、生成 | 学生是否真的会，仍靠自觉 |
| 课程平台 | 组织内容、布置任务 | 很难按个人缺口改变路径 |
| 笔记工具 | 保存资料、整理笔记 | 不会主动追问和审核理解 |

底部结论：

> 学习系统最该回答的问题：学生凭什么算学会？

## 视频画面

继续停留在仓库总览或 Dashboard。画面中可以看到资料、卡片、图谱、路径入口。

可以用局部放大强调：这不是一次聊天，而是一个长期学习空间。

## 口播

我们先看行业痛点。

通用 AI 很强，它可以解释一道题，生成一份资料，也可以陪学生聊天。但它通常只能保证知识被输入给学生。学生有没有真正吸收，哪里没懂，是否能独立迁移，仍然需要学生自己判断。

课程平台能组织课程和任务，但它往往是固定路径。不同学生真正卡住的地方不同，可平台很难根据一个学生的真实回答，动态改变提问方式、资源类型和下一步路径。

笔记工具能保存资料，但保存不等于掌握。很多学生的问题不是没有笔记，而是笔记越来越多，自己却越来越不知道哪些只是看过，哪些真的会了。

所以 AXIOM 面对的核心需求不是“再多生成一些内容”，而是建立一套机制：让学生的真实输出成为证据，让系统知道他到底会了什么、没会什么，以及下一步应该怎么学。

## 本段价值点

把项目从“功能展示”抬到“教育问题”：不是内容不足，而是掌握不可见、不可验证、不可延续。

# 1:40-2:40｜三、整体系统架构与方案设计

## PPT 上屏内容

主标题：

> AXIOM 的设计：  
> 把一次学习变成证据闭环

流程图：

```text
课程资料
  -> 知识节点
  -> 灵感卡
  -> 苏格拉底式追问
  -> 学生费曼输出
  -> Agent B 记录证据
  -> 审核
  -> 永久卡
  -> 画像 / 资源 / 路径 / 图谱更新
```

旁边三条设计原则：

> 画像必须有证据  
> 掌握必须经审核  
> 推荐必须可追溯

## 视频画面

按照真实使用顺序展示：

1. 进入仓库；
2. 初次画像对话；
3. 查看画像变化；
4. 导入学习资料；
5. Dashboard 显示知识图谱和三色卡片统计。

如果录制时不能完整展示画像对话，也要至少展示“画像证据”和“画像变化”页面。

## 口播

AXIOM 的整体设计不是一个聊天框，而是一个长期学习仓库。

第一层是课程资料。学生导入课程讲义、文章或项目材料后，系统会把资料变成可以追踪的知识节点，而不是只把文件保存起来。

第二层是学习画像。系统不是用问卷给学生贴标签，而是通过对话和学习表现，形成一组可以被证据修正的教学假设。学生为什么卡住、适合什么解释方式、下一步应该怎么追问，都要回到真实证据。

第三层是卡片与图谱。文献卡表示知识从哪里来；灵感卡表示学生接触过但尚未掌握；永久卡表示学生已经通过输出和审核，证明这项知识在当前流程中已经掌握。

第四层是双 Agent 工作台。Agent A 在前台用苏格拉底式提问逼学生输出；Agent B 在后台保留学生原话、更新画像和写回卡片。

最后是审核与反馈。学生讲清以后，系统审核是否达到掌握标准。通过后，灵感卡升级为永久卡；这份证据会继续改变后续资源推荐、学习路径和知识图谱。

这就是 AXIOM 的系统方案：不是让 AI 替学生学习，而是让 AI 帮学生暴露盲区、保留证据、验证掌握，并持续改变下一步。

## 本段价值点

评委要听懂系统架构，但不能陷入技术堆砌。架构要被解释成一个学习闭环。

## 4:15-4:45｜4.5 苏格拉底式提问与费曼输出

### PPT 上屏内容

> 通用 AI 帮你输入，  
> AXIOM 逼你输出

角色标签：

> Agent A：苏格拉底式追问  
> 学生：费曼式输出  
> Agent B：记录证据

### 视频画面

展示 Agent A 提问，学生回答。问题最好包含预测、为什么、例子、反例、边界。

### 口播

进入工作台后，Agent A 不会直接把标准答案讲完，而是持续追问。

比如，它会问：这段代码会调用哪个方法？为什么？如果参数类型换掉会怎样？如果对象结构频繁变化，Visitor 还合适吗？

这些问题的目的不是为难学生，而是逼他把自己脑子里的理解说出来。

学生需要用自己的话完成费曼式输出。只有他能把机制讲清楚，系统才有证据判断他是否真的理解。

### 价值点

这是和通用 AI 的核心差异：AXIOM 不满足于“答案出现”，它要看到“学生输出”。

## 4:45-5:00｜4.6 Agent B 记录与审核入口

### PPT 上屏内容

> AI 可以整理表达，  
> 但不能替学生制造掌握

### 视频画面

展示 Agent B 记录学生原话、写回灵感卡、更新画像，随后展示审核入口或审核状态。

### 口播

同时，Agent B 会在后台保留学生原话，提取关键证据，更新画像，并把当前理解写回灵感卡。

但这里有一条边界：Agent B 不能替学生生成“我的理解”，更不能直接把灵感卡改成永久卡。

因为永久卡代表已经掌握。掌握必须来自学生自己的输出，并且必须经过审核。

### 价值点

AI 的整理能力和判断权限被拆开。它可以整理证据，但不能伪造掌握。

# 5:00-5:50｜五、技术创新与难点攻克

## PPT 上屏内容

主标题：

> 技术创新：  
> 让 AI 的学习判断可追溯、可验证、可复用

四个创新点：

> 1. 双 Agent 分工：追问与记录分离  
> 2. 证据约束画像：每个判断都有来源  
> 3. 掌握审核机制：灵感卡不能直接变永久卡  
> 4. 向量数据库旧知迁移：旧理解服务新问题

## 视频画面

优先展示：

1. 审核失败；
2. 继续打磨；
3. 审核通过；
4. 灵感卡升级为永久卡；
5. 相关旧永久卡被召回；
6. 图谱中掌握状态变化。

## 口播

AXIOM 的技术创新，不是简单把大模型接进学习软件，而是解决 AI 学习系统里几个最难的问题。

第一个是双 Agent 分工。Agent A 面向学生，负责提问和追问；Agent B 在后台记录原话、更新画像和写回卡片。这样学生可以专注表达，系统负责沉淀证据。

第二个是证据约束画像。画像不是静态标签，而是由学生原话、学习表现、评估结果不断修正。每一条画像判断都要能追溯来源。

第三个是掌握审核机制。灵感卡代表尚未掌握，永久卡代表已经掌握。AI 不能因为一段文字看起来流畅，就把它包装成掌握。学生必须讲清机制、处理边界，并能应用到新情境，才可以通过审核。

第四个是向量数据库带来的旧知迁移。过去已经掌握的永久卡，可以被语义检索找回，用来帮助当前问题搭桥。比如学习 Visitor 时，系统可以召回 Java 重载、重写、多态分派这些旧知识。

但相似不等于掌握。旧知识只能帮助追问，不能替当前卡片通过审核。当前知识仍然需要学生重新输出、重新验证。

这些设计共同解决了一个难点：让 AI 不只是生成内容，而是参与一个可追溯、可验证的学习判断过程。

## 本段价值点

技术创新必须讲成“解决了什么难题”，不要讲成技术名词堆砌。

# 最终录制提醒

## 一镜到底必须覆盖的产品动作

```text
进入仓库
  -> 初次画像对话式构建
  -> 查看画像变化
  -> 导入学习资料
  -> 仪表盘查看知识图谱与三色卡片
  -> 点击灵感卡进入工作台
  -> Agent A 苏格拉底式提问
  -> 学生费曼式输出
  -> Agent B 记录证据
  -> 审核失败
  -> 继续打磨
  -> 审核通过
  -> 灵感卡升级为永久卡
  -> 主动资源生成
  -> 系统资源推送
  -> 六类资源预览
  -> 知识图谱多视角
  -> 长期档案与价值收束
```

## 录制时最容易丢掉的价值

1. 不要把画像讲成问卷，要讲成证据约束的教学假设；
2. 不要把三色卡片讲成分类，要讲成掌握状态边界；
3. 不要把 Agent B 讲成自动笔记，要讲成原话证据沉淀；
4. 不要把审核讲成卡片管理，要讲成防止 AI 伪造掌握；
5. 不要把资源讲成六个格式，要讲成按缺口和偏好补学习；
6. 不要把向量库讲成搜索，要讲成旧知迁移但不替代当前审核；
7. 不要只讲学生效率，还要讲教师干预和课程证据复用。

## 如果时间不够，优先保留

1. 开头问题；
2. 画像证据；
3. 三色卡片；
4. 苏格拉底追问与学生输出；
5. 审核失败到通过；
6. 永久卡代表掌握；
7. 资源推送有证据；
8. 结尾价值。

六类资源预览和图谱多模式可以压缩，但不能完全删除。
