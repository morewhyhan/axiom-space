# AXIOM Space A3 黄金案例：完整口播版

## 1. 使用说明

本文件用于实际录制，配套验收规格见：

[10-A3演示视频脚本-黄金案例.md](./10-A3演示视频脚本-黄金案例.md)

旁白原则：

1. 不重复朗读页面文字。
2. 不用旁白替页面证明功能。
3. 每段只说明“发现了什么、因此改变什么、证据在哪里”。
4. 学生和 AI 的长回答以屏幕录制为主，旁白只抓住关键变化。
5. 最终视频控制在 7 分钟内。

### 适时验证的录制节奏

相关操作完成并到达自然检查点后，先验证，再使用结论推进下一阶段：

```text
完成阶段 -> 打开证据 -> 看结论/来源/变化 -> 再让结论驱动下一阶段
```

不能把所有画像、评估和路径变化留到视频结尾第一次展示，也不需要每说一句话就机械打开画像。判断标准是：一个画像结论在被路径、资源或推送使用之前，评委已经看见它及其依据。

## 2. 七分钟完整脚本

### 0:00-0:25 开场与基线失败

页面操作：

```text
进入《软件设计模式》仓库。
打开 Visitor 基线任务。
学生预测 Java 代码输出为 PDF。
运行代码，控制台实际输出 Document。
页面显示“未通过”。
```

学生台词：

> 实际对象是 Pdf，所以我觉得应该输出 PDF。如果输出 Document，我不知道为什么，程序运行时不是知道真实类型吗？

旁白：

> 这是一名计算机专业的大专学生。他能背出 Visitor 的标准结构，却解释不了为什么需要 `accept`。我们先运行一段真实代码。预测和实际输出发生冲突，说明他的问题不是没见过这个模式，而是没有理解方法分派的中间过程。

必须拍到：学生原始回答、真实控制台输出、基线评估失败。

### 0:25-1:20 对话诊断与画像假设

页面操作：

```text
展示三轮诊断对话。
系统区分重写、重载基础与 Visitor 机制缺口。
打开画像假设卡和竞争假设。
三轮诊断结束后展开画像证据抽屉。
点击“具体缺口”和“认知加工”，回看来源消息、置信度和待验证状态。
用户确认其中一项，查看确认记录及影响范围，然后再生成路径。
```

学生关键台词：

> 老师跳过一个“为什么”继续讲时，我会停在那一步。Visitor 为什么要调用两次没想通，后面的 UML 和优缺点我就都记不牢。

> 我知道重写会看真实对象，也知道重载是同名不同参数。但我一直以为重载也会看参数对象的真实类型。

旁白：

> 系统没有把“跟不上”直接判断成基础差。它把可能原因拆成三个待验证假设：Java 基础整体薄弱、缺少编译期和运行期的过程模型，或者只是没有背熟 Visitor。当前证据更支持第二种，但这个判断仍然可以被后续任务推翻。

> 页面上可以看到八个画像维度。每项判断都带有原始对话、置信度和待验证状态，用户也可以确认、纠正或删除。

必须拍到：不少于六维画像、证据来源、竞争假设、置信度、用户纠正入口。

本阶段验证完成条件：评委能把“误以为重载看运行时参数类型”准确追溯到学生的原始回答，而不是只看到系统结论。

### 1:20-1:50 个性化路径变化

页面操作：

```text
展示完整《软件设计模式》课程资料库。
打开默认课程路径与小林当前路径的对比。
突出新增、跳过和调整顺序的步骤。
路径生成完成后、进入辅导之前打开变更详情，逐项显示引用了哪个画像字段。
```

旁白：

> 个性化不是生成一条看起来不同的目录。默认路径会继续讲 Visitor 的角色和 UML，但系统发现小林真正缺少的是重载与重写的过程模型，所以跳过重复内容，插入类型分派实验，并把每一步的完成条件改成代码预测和机制解释。

> 右侧显示了每次调整引用的画像证据。这证明路径变化不是随机生成，而是由刚才确认的学习缺口触发。

必须拍到：完整课程目录、默认路径、个性化路径、至少一个新增步骤、一个跳过步骤、一个顺序变化及其依据。

### 1:50-3:15 针对性辅导

页面操作：

```text
进入“为什么 Visitor 需要 accept”卡片。
AI 依次追问重载选择、第一次动态分派和 visit(this)。
播放双重分派调用动画。
学生解释被写入右侧卡片。
这一组解释完成后、进入评估之前打开卡片变更记录，核对新增段落、来源消息和保存状态。
```

旁白一：

> 系统没有重新讲一遍完整定义，而是把小林卡住的过程拆成三个节点。第一步只判断编译器根据什么选择重载签名；第二步观察运行时对象如何进入具体元素的 `accept`；第三步再看 `visit(this)` 如何把具体元素类型交给访问者。

旁白二：

> 这里可以同时看到代码、调用轨迹和学生自己的解释。系统每次只闭合一个因果缺口，确认以后才继续，而不是让他再次面对一段连续长讲解。

学生最终解释：

> `element.accept(visitor)` 先根据元素的真实类型进入 `Pdf.accept`。在 Pdf 类内部，`this` 的类型明确为 Pdf，所以会选择 `visit(Pdf)`；访问者的真实类型再决定执行哪个具体实现。最终行为同时取决于元素类型和访问者类型。

必须拍到：真实多轮对话、可播放动画、代码轨迹、右侧卡片实时更新及来源消息。

---

### 3:15-4:05 多智能体与多模态资源

页面操作：

```text
点击生成巩固资源。
展示六个 Agent 的任务、输入、状态、产物和交接。
依次打开文档、类图、动画、题组、Java 项目和案例阅读。
展开技术证据抽屉。
资源生成批次完成后打开质量报告；存在警告时先处理，不能直接进入评估。
```

旁白：

> 资源生成不是一个模型一次性吐出六个标题。诊断 Agent 确定缺口，资料 Agent 提供课程和语言规范依据，路径 Agent 决定下一步，资源 Agent 生成内容，评估 Agent 准备迁移任务，安全 Agent 检查代码和答案。

> 每个资源都说明为什么适合当前学生。这里不再重复 UML，而是让编译期类型、运行时类型和实际调用结果同时可见。动画能够真实播放，Java 项目也能实际运行。

必须拍到：独立任务 ID、Agent 交接、六类可打开资源、真实动画、代码运行、生成进度。

### 4:05-5:15 费曼评估与迁移

页面操作：

```text
学生第一次只说“单分派需要双重分派”。
系统拒绝通过，并显示缺失项。
第一轮评估结束后打开正式记录，核对原始回答与逐项失败原因。
系统切换到陌生 AST 场景。
学生预测调用轨迹、解释变化成本并判断适用边界。
Java 执行结果与固定量规显示通过。
第二轮通过后先查看正式评估；画像差异、路径影响、卡片资格和推送可以在随后“本轮学习结果”检查点集中展示，但不能留到视频结尾才第一次出现。
```

旁白一：

> 用户说出“双重分派”这个词还不能算掌握。第一次回答没有说明哪一步由编译期类型决定，也没有说明 Visitor 的变化成本，所以系统拒绝通过。

旁白二：

> 第二次评估换成学生没有见过的 AST 场景。他不仅要预测调用轨迹，还要解释新增操作和新增元素类型分别会修改什么，并判断什么时候不该使用 Visitor。

> 辅导 Agent 不能直接宣布自己教会了学生。评估使用固定量规和隐藏任务，代码预测还必须通过 Java 实际运行这个硬门槛。

必须拍到：第一次失败、具体缺失项、陌生迁移任务、逐项量规、真实代码结果、正式评估记录。

本阶段验证完成条件：同一个评估 ID 同时出现在评估记录、画像变化、卡片资格和下一步路径中。

### 5:15-5:50 防幻觉与永久知识

页面操作：

```text
展示包含技术错误的候选永久卡片。
点击提炼，系统拒绝。
拒绝后打开审核报告，展示错误位置、执行反证和来源，再进行修正。
展示 Java 执行反证和课程来源。
展示答案与解析冲突的资源被拦截。
修正后重新校验并升级永久知识。
升级后打开卡片证据详情，确认使用的是刚才的正式评估记录。
```

旁白：

> 这里我们故意放入两类错误。第一类把 Java 重载错误地说成依据运行时参数类型选择，系统使用真实执行结果反证；第二类题目答案和解析互相冲突，结构化校验会直接阻止资源进入完成状态。

> 修正以后，系统重新检查课程依据、代码结果、学生迁移回答和正式评估记录。只有这些证据一致，卡片才能进入永久知识库。

必须拍到：两类不同错误、失败原因、执行或来源证据、重新校验记录、永久卡片升级成功。

### 5:50-6:20 画像、路径和推送更新

页面操作：

```text
打开画像字段前后对比。
显示旧任务被跳过、新任务被插入。
打开推送箱和知识图谱。
刷新页面，确认状态仍然存在。
```

旁白：

> 学习结束后，系统不是简单弹出一句“画像已更新”。这里能看到具体字段从什么变成什么，以及是哪一条评估证据触发了变化。

> Visitor 的基础 UML 不再重复推送。新的任务转向 Visitor 与其他模式的选择边界。画像、路径、卡片、推送和图谱引用的是同一份掌握证据，刷新页面后仍然保持一致。

必须拍到：字段旧值与新值、触发证据、路径变化、推送理由、图谱新节点、刷新后持久化。

### 6:20-6:45 成熟仓库与长期形态

页面操作：

```text
切换到“小林·设计模式学期档案”。
先拍到覆盖周期、学习事件数量、证据时间线和原始记录入口。
依次打开六个画像维度，每个维度拍到 3 至 5 条合并后的动态子维度。
重点打开“哪里会卡 / 核心阻塞机制”的证据详情，拍到观察、分析、竞争解释、鉴别依据、干预和验证标准。
再展示画像演化时间线、课程掌握地图、永久知识和下一阶段计划。
选中一条历史记录，展开原始误区、干预、首次评估和跨会话复测。
```

旁白：

> 前面的干净仓库证明了一次学习闭环如何发生。这里切换到同一学生的学期档案，展示多次闭环积累以后，产品最终承载的长期形态。

> 六个维度是稳定的教学决策框架，里面的子维度不是预设表单。系统根据长期对话、测评、卡片和路径行为自行发现并合并。例如，它没有停留在“小林容易卡在 Visitor”，而是进一步形成“关键因果前提未闭合时，后续信息难以进入”这一可验证机制。

> 对用户来说，画像不只告诉他系统观察到了什么，还会说明目前怎样理解、排除了哪些可能，以及接下来会怎样帮助他。这里的结论不是给用户定型；每一条都有适用范围、置信度和复测条件，可以被后续证据修正。

> 更关键的是，这些画像不是只用于展示。它被压缩进下一轮教学提示词，所以系统会实际跳过已经掌握的 UML，保留解释深度但缩短单次因果跨度，并用陌生迁移验证干预是否有效。

> 我们不看节点有多少，而是追踪一条历史变化：系统最初如何判断、学生接受了什么干预、哪次评估没有通过、后续复测是否保持，以及这些证据怎样影响下一阶段计划。

> 这一步不是用一个漂亮的总分证明长期有效，而是让评委看到长期档案怎么工作：每个状态都能回到原始会话、评估记录和复测结果，后续计划也不是凭空生成的。

必须拍到：六个固定维度、每维动态子维度、观察与分析、竞争解释、干预与验证、覆盖周期、原始记录入口、画像变化时间线、掌握地图、一条历史证据链、跨会话复测状态、下一阶段推荐依据。

## 3. 旁白总检查

录制前逐句检查：

- [ ] 每个“系统发现”都有页面证据。
- [ ] 每项关键结论都在被下一步使用前展示了对应证据，没有拖到结尾才第一次验证。
- [ ] 相关变化在自然检查点集中展示，普通对话不会反复弹出笼统的“画像已更新”。
- [ ] 每个“个性化”都有默认方案对照。
- [ ] 每个“掌握”都有陌生任务或确定性结果。
- [ ] 每个“多智能体”都有角色输入、产物和交接。
- [ ] 每个“多模态”都有实际可播放或可运行内容。
- [ ] 每个“防幻觉”都有具体错误、拦截原因和复验结果。
- [ ] 没有声称“完全了解用户”或“对所有学生有效”。
- [ ] 成熟仓库展示覆盖周期、事件数量、证据时间线和原始记录入口；旁白不主动解释素材属性。
- [ ] 长期结果展示的是历史证据链和复测状态，不是只展示节点数量。
- [ ] 没有朗读大段页面文字。
- [ ] 总时长不超过 7 分钟。
