# AXIOM Space A3 演示视频脚本 10：黄金案例

## 1. 文档定位

这不是一份功能导览，也不是要求产品机械照着台词运行的固定剧本。

它同时承担三件事：

1. 作为 A3 演示视频的用户故事和录制脚本。
2. 作为“产品是否成立”的端到端验收案例。
3. 作为赛题要求与页面证据之间的对照表。

本案例只证明一个最小但完整的产品命题：

> AXIOM Space 能通过对话提出并验证关于学生学习障碍的假设，依据已确认的学习画像实施有差异的干预，再用学生的解释、辨析、验证和迁移表现证明干预是否有效，并据此更新后续计划。

视频中的每个强结论都必须具备以下证据链：

```text
原始对话证据
-> 系统判断（含置信度和待验证项）
-> 因判断而改变的计划或资源
-> 学生可观察的学习输出
-> 评估结果
-> 画像、路径、卡片和推送的同步变化
```

旁白不能替页面作证。页面没有显示的结论，不得在视频中声称已经做到。

### 1.1 适时可见验证原则

所有关键状态变化最终都必须被页面验证，并且要在该结论被用于支撑下一步决策之前展示。相关变化可以在一个自然阶段结束后集中查看，不要求每发生一个微小变化就立刻切换面板。

推荐节奏：

```text
完成一组相关操作
-> 到达自然检查点
-> 打开对应证据面板
-> 展示结论、来源、变化和影响范围
-> 再让该证据驱动下一阶段
```

不要求每发送一句普通对话都切换画像。若一个阶段没有提取到新证据，也不应每次都笼统弹出“画像已更新”。核心约束是“先展示证据，再使用结论”，而不是机械追求即时切换。

| 自然检查点 | 应查看的证据 | 必须验证的内容 |
|---|---|---|
| 诊断对话结束、生成路径之前 | 画像证据抽屉 | 新增/修改字段、原始话语、事实/推断、置信度 |
| 用户完成画像确认或纠正 | 画像字段详情 | 修正前后值、确认记录、哪些输出将受影响 |
| 路径生成完成、进入辅导之前 | 路径变更详情 | 新增、跳过、调序步骤及引用的画像证据 |
| 一组有效解释完成、进入评估之前 | 当前理解卡 | 新写入内容、来源消息、保存状态 |
| 一批资源生成完成、正式使用之前 | 协作记录与质量报告 | Agent 输入/产物/交接、资源依据、校验结果 |
| 每轮正式评估结束 | 正式评估记录 | 原始回答、量规、逐项结果、确定性测试输出 |
| 本轮学习结束、生成下一步之前 | 画像前后对比 | 旧缺口、新掌握、触发评估 ID、仍待验证项 |
| 卡片晋级或被拒之后 | 审核报告 | 通过/拒绝原因、来源、冲突证据 |
| 下一步推荐展示时 | 推送依据详情 | 已掌握证据、剩余缺口、为什么不再推旧内容 |

视频结尾可以汇总回放，但汇总不能成为关键结论第一次出现的地方。

### 1.2 双仓库演示策略

视频使用两个用途不同的仓库：

| 仓库 | 作用 | 可以证明什么 |
|---|---|---|
| A：小林·Visitor 黄金案例（干净仓库） | 从基线错误开始，现场跑通一次闭环 | 诊断、干预、评估和状态变化之间存在可见因果链 |
| B：小林·设计模式学期档案（成熟仓库） | 展示多次闭环积累后的长期产品形态 | 系统能够承载历史证据、画像演化、课程进度和知识沉淀 |

仓库 B 不能替代仓库 A。只展示成熟仓库，评委会怀疑结果是预先摆好的；只展示干净仓库，评委又看不到产品长期使用后的价值。

仓库 B 不需要在视频中特地解释它的素材属性。更重要的是让它看起来像一个严肃的长期学习档案：页面应显示覆盖周期、学习事件数量、证据时间线和原始记录入口，让评委能沿着一条历史链追溯，而不是只看一个静态结果页。

旁白口径要克制：可以说“学期档案”“长期学习记录”“多次闭环积累后的产品形态”，不要主动说“模拟出来的长期效果”，也不要声称“真实学生使用一个学期后已经被证明显著提升”。可信度来自可追溯证据链，而不是靠标签自证。

## 2. 黄金案例的学生

### 2.1 基本身份

姓名：小林（虚构案例）
学历：高职专科二年级
专业：计算机应用技术
当前课程仓库：《软件设计模式》
长期使用范围：计算机专业课程，包括 Java 程序设计、软件工程、设计模式、数据库和 Web 开发等
本次演示切片：Visitor（访问者模式）中的“双重分派为什么需要两次调用”

赛题要求至少准备一门完整高校专业课程的初始知识库。本案例录制时使用完整《软件设计模式》课程资料库，覆盖面向对象原则、UML、GoF 23 种模式、重构案例和综合项目，但视频只深入演示一个知识卡点。

不要在视频中声称 Visitor 是“公认最难的设计模式”。这个结论没有统一标准，也容易被评委用 Interpreter、Abstract Factory 等反驳。稳妥表述是：

> Visitor 的双重分派是设计模式课程中概念门槛较高、最容易“记住结构却没理解机制”的代表性环节。

### 2.2 初始状态

小林对自己的描述是：

> 老师讲的每句话我好像都听到了，但只要中间有一个“为什么”没有想明白，我就会一直停在那里。老师已经讲到后面，我还在想前一步。最后整节课都变得模模糊糊。我不是完全不想学，我是越想弄明白，越跟不上。

他在 Visitor 上的初始回答是：

> Visitor 就是把操作从对象里拿出来，元素写一个 `accept`，访问者写很多 `visit`。但是为什么不能直接 `visitor.visit(element)`，非要 `element.accept(visitor)` 再调一次，我完全想不明白。我能照着写，但不知道这两层到底做了什么。

### 2.3 待验证的心理机制

这里不做医学诊断，也不把学生贴成固定“心理人群”。系统应提出可被证伪的学习机制假设：

> 小林可能具有较强的反思型、因果链式加工倾向。他需要先补齐前提和因果关系，才愿意接受下一步结论。当课堂跳过一个关键前提时，未解决的问题持续占用工作记忆，后续信息无法稳定进入，最终表现为反应慢、跟不上和自我怀疑。这可能是“教学节奏与加工方式不匹配”，不等于理解能力低。

系统还必须保留并排查其他解释：

| 竞争假设 | 如何区分 |
|---|---|
| 前置知识不足 | 给出前置概念小测；若基础概念也无法识别，则先补基础 |
| 术语障碍 | 换成生活化表述后再次询问；若立即理解，则优先建立术语映射 |
| 注意力维持困难 | 比较短时任务、长段讲解和分段讲解下的表现 |
| 只是不熟练 | 给足时间后检查能否独立解释和迁移 |
| 因果完整性要求高 | 故意省略一个关键前提，观察学生是否停在缺口而无法继续 |

### 2.4 可以声称什么，不能声称什么

可以声称：

```text
系统根据三段具体回答，提出“因果链缺口导致节奏失配”的高置信度假设。
系统用后续问题和任务验证了这个假设。
系统因此改变了讲解颗粒度、学习顺序和评估方式。
```

不能声称：

```text
系统完全看透了学生。
系统比学生更懂他自己。
系统仅凭三言两语确定了人格或心理类型。
纯文字对话系统识别了微表情。
```

产品的专业性不来自“读心”，而来自“有证据地提出假设，并主动寻找反证”。

## 3. 成立与失败标准

### 3.1 本案例闭环成立的最小标准

本案例同时满足以下条件，才可判为通过：

1. 系统从自然对话构建不少于 6 个维度的画像，每个关键判断可回到原始话语。
2. 系统展示至少一个竞争假设，并通过追问或任务修正置信度。
3. 个性化路径与通用路径存在可见差异，而且系统说明差异来自哪条画像证据。
4. 干预前，学生无法解释原理或完成迁移题；干预后，学生能够解释、举例、说明边界、验证并迁移。
5. 同一条评估证据同时改变任务状态、画像、知识卡片和后续推送，不出现互相矛盾。
6. 错误或证据不足的内容不能进入永久知识；修正并通过评估后才能进入。
7. 至少 5 类个性化资源由可见的多智能体协作生成，并通过质量检查。

### 3.2 一票否决

出现以下任一情况，不得在视频中声称“因材施教已经成立”：

- 画像判断错误，用户纠正后系统仍沿用旧判断。
- 路径、资源和推送没有引用画像证据，换一个学生仍会得到同样结果。
- 用户只说“满意”或复述原话，系统就判定掌握。
- 费曼评估已经通过，但永久卡片仍说缺少另一套评估证据。
- 画像显示已经掌握，推送仍把同一概念当作未解决问题。
- 题目答案、解释、代码或动画互相矛盾，但审核仍显示“完成”。
- 页面没有可见证据，只由旁白宣称后台已经分析或更新。

## 4. 核心对照设计

旧脚本只展示“系统生成了一条路径”。黄金案例必须展示“为什么这条路径只适合这个学生”。

### 4.1 干预前基线

系统先给出一段可实际运行的 Java 基线代码：

```java
class Document {}
class Pdf extends Document {}

class Exporter {
    void visit(Document document) { System.out.println("Document"); }
    void visit(Pdf pdf) { System.out.println("PDF"); }
}

Document element = new Pdf();
new Exporter().visit(element);
```

系统要求学生先预测输出，再解释原因。小林的初始回答：

```text
实际对象是 Pdf，所以我觉得应该输出 PDF。
如果输出 Document，我不知道为什么；程序运行时不是知道真实类型吗？
```

页面记录：

```text
能复述 Visitor 结构：是
能区分重载与重写：否
能解释编译期类型与运行时类型：否
能解释 accept 的作用：否
能判断适用边界：否
结论：尚未掌握
```

### 4.2 计划差异对照

页面必须同时展示“通用方案”和“本次个性化方案”的差异摘要：

| 通用教学顺序 | 小林的个性化顺序 |
|---|---|
| 给出 Visitor UML 和角色定义 | 先运行一段“预测错”的重载代码，暴露真实缺口 |
| 背诵 Element、Visitor、accept、visit | 把编译期重载选择和运行期重写分派拆成两个因果节点 |
| 照着标准结构写代码 | 用调用轨迹解释第一次和第二次分派分别决定什么 |
| 做模式名称选择题 | 比较“经常加操作”和“经常加元素类型”两种变化方向 |
| 统一讲解长度 | 一次只推进一个因果节点，确认后再继续 |

个性化依据必须显示：

```text
依据 P-03：用户遇到未解释前提时会停留，后续信息进入失败。
依据 P-05：用户给足时间后能形成较深解释，不宜简单降难度。
因此：不降低知识深度，降低单次推进跨度；显式补齐因果前提；增加边界与反例校验。
```

如果页面无法展示这组差异，本案例最重要的“个性化”证据就没有成立。

## 5. 七分钟演示脚本

实际等待时间可通过剪辑压缩，但关键输入、输出、失败过程和前后变化不得剪掉。每一幕同时写明“评委可能怎么挑刺”和“页面如何反证”。

### 0:00-0:25 开场：先让学生预测错

画面：进入《软件设计模式》仓库，打开 Visitor 基线任务，实际运行第 4.1 节的 Java 代码。

学生先预测：

```text
实际对象是 Pdf，所以应该输出 PDF。
```

控制台实际输出：

```text
Document
```

页面评估：

```text
未通过
会背 Visitor 结构，但不能区分重载选择与运行时重写。
```

旁白：

> 小林上课时能记住每个类，却总觉得 `accept` 是多余的。先不告诉他答案，我们用一段真实运行的代码找到他具体卡在哪一步。

评委可能挑刺：错误回答是事先编的。
页面反证：展示学生提交时间、程序实际运行输出和自动评估记录；不能用字幕伪造控制台。

### 0:25-1:20 对话诊断：区分基础不足与机制缺口

系统提问 1：

```text
老师跳过一个“为什么”继续讲时，你会先跟着后面走，还是会停在那一步继续想？
```

学生回答：

```text
我会停住。Visitor 里面为什么要两次调用没想通，后面的 UML、优缺点我就都记不牢。
```

系统提问 2：

```text
我先排除 Java 基础问题：重写和重载分别发生在什么情况下？不确定也可以直接说。
```

学生回答：

```text
重写是子类改父类的方法，运行时会看真实对象。重载是同名但参数不同。
但我一直以为重载也会看参数对象的真实类型。
```

系统提问 3：

```text
如果把代码分成“编译器先选签名”和“运行时再找实现”两个步骤，你愿意逐步预测每一步吗？
```

学生回答：

```text
可以。我需要知道每一步到底由哪个类型决定，不能只看一张最终 UML 图。
```

系统输出带证据的画像假设卡：

| 画像维度 | 当前判断 | 证据 | 置信度 |
|---|---|---|---|
| 专业与目标 | 计算机应用技术；能在项目中判断和使用模式 | 用户自述 | 已确认 |
| 前置基础 | 知道继承、接口、重写和重载的定义 | 回答 P-02 | 中高 |
| 具体缺口 | 误以为 Java 重载依据参数运行时类型选择 | 基线代码与 P-02 | 高 |
| 认知加工 | 偏因果链式，需要看见中间机制 | P-01、P-03 | 中，待验证 |
| 节奏负荷 | 一个隐含前提未闭合会阻塞后续内容 | P-01 | 中高 |
| 情绪状态 | 把跟不上归因于自己基础差 | 用户自述 | 中 |
| 资源偏好 | 代码运行轨迹、时序动画、对照实验 | 尚待学习效果验证 | 低 |
| 掌握标准 | 能预测调用、解释机制、权衡变化方向并迁移 | 师生共同确认 | 已确认 |

诊断阶段结束后、画像被用于生成路径之前，必须打开画像卡。不得只显示“画像已更新”的通知。依次展开“具体缺口”和“认知加工”两个字段，回看原始消息并确认其状态分别为“高置信度事实判断”和“中置信度待验证假设”。

系统同时显示竞争假设：

```text
H1：Java 多态基础整体薄弱，置信度 0.35
H2：知道术语，但缺少“编译期重载 + 运行时重写”的过程模型，置信度 0.72
H3：只是没背熟 Visitor 结构，置信度 0.20
下一步：用两个只改变声明类型的对照程序区分 H1 与 H2。
```

系统声明边界：

```text
这是关于学习障碍的暂时假设，不是人格或心理诊断。
若后续代码预测没有改善，系统必须降低置信度并重新判断。
涉及情绪和认知风格的字段须由用户确认；用户可以修改、删除或停止使用这些字段参与推荐。
```

评委可能挑刺：三句话就判断心理机制，属于 AI 读心。
页面反证：显示竞争假设、置信度、原始证据和“待验证”状态；允许学生纠正，后续任务可推翻判断。

A3 踩点：自然语言画像、不少于 6 维、画像可修正、学习历史与目标抽取。

### 1:20-1:50 路径生成：证明计划真的因人而异

画面先展示完整《软件设计模式》课程库的目录、文档数和来源，再进入约 32 步课程路径中的“行为型模式”任务组。

页面并列：

| 默认课程路径 | 小林本轮插入的诊断路径 |
|---|---|
| Visitor 意图 | Java 重载究竟按什么类型选择 |
| Visitor 角色与 UML | 同一对象、不同声明类型的输出对照 |
| Visitor 标准代码 | 第一次分派：谁选择具体 `accept` |
| 优缺点 | 第二次选择：为什么 `this` 能锁定 `visit(Pdf)` |
| 模式应用题 | “加操作”和“加元素类型”的变化成本实验 |

页面必须显示：

```text
插入原因：画像 K-03 显示缺口不在 UML，而在重载选择机制。
保留深度：不降低课程难度，只缩小单次因果跨度。
完成条件：陌生代码预测 + 机制解释 + 变化方向权衡 + 新场景迁移。
```

评委可能挑刺：所谓个性化只是把通用路径换了几个词。
页面反证：至少展示一个新增步骤、一个跳过步骤、一个顺序变化，并分别引用画像证据；最好提供“恢复默认路径”用于现场对照。

A3 踩点：完整专业课程资料、动态路径、明确步骤顺序、画像驱动规划。

### 1:50-3:15 针对性辅导：闭合双重分派的因果链

画面：进入“为什么 Visitor 需要 `accept`”理解卡。资料依据显示课程讲义和 Java 方法调用规范的对应章节。

AI 第一步只问重载：

```text
Document element = new Pdf();
exporter.visit(element);
```

| 智能体设计、开发、测试和部署全流程 | 系统开发说明书、架构图、测试报告 |
| AI Coding 工具使用说明 | 配套文档显著位置 |
| 可稳定运行的源码、数据集和模型配置 | 提交工程与部署说明 |
| 大学生学习需求调研 | 需求分析文档、样本与调研结论 |

“全部踩点”不等于把全部材料念进视频。视频负责证明产品闭环，PPT 负责解释创新与技术，文档负责给出工程和合规证据。

### 6.2 证据强度边界

评委很可能追问：“一个精心编排的案例，凭什么证明产品成立？”对此不能偷换概念。

| 证据层级 | 能证明什么 | 仍不能证明什么 |
|---|---|---|
| 七分钟黄金案例 | 系统具备跑通一次可验证个性化闭环的能力 | 对所有学生都有效 |
| 三种对照画像复测 | 相同知识点会因不同障碍生成不同干预 | 长期保持和真实成绩提升 |
| 跨知识点、隔日无提示复测 | 画像与干预策略能够迁移，掌握有一定保持 | 大规模教育效果 |
| 真实用户试点与基线对照 | 相对通用 AI 或普通课程的效果差异 | 更广泛人群中的稳定效果，仍需持续研究 |

视频中只主张第一层已经成立。若提交材料能补充后三层数据，再逐级扩大产品效果主张。

## 7. 录制前产品验收表

以下条目必须在真实页面逐项通过。任何一项失败，都先修产品，不用旁白遮盖。

### 7.1 画像证据链

- [ ] 画像关键字段能够显示来源消息或证据 ID。
- [ ] 系统区分“事实、推断、待验证假设”。
- [ ] 用户纠正画像后，旧判断不再继续驱动路径和资源。
- [ ] 评估通过后，画像旧缺口不会仍显示为未解决。
- [ ] 更新日志明确显示“哪个字段从什么变成什么、依据是什么”。
- [ ] 有决策价值的画像证据在被路径、资源或推送使用之前已经展示，不拖到结尾。
- [ ] 普通对话不会反复制造笼统的“画像已更新”提示；相关变化可在自然检查点集中展示。

### 7.2 个性化干预

- [ ] 页面能展示通用方案与个性化方案的差异。
- [ ] 路径、卡片、资源和推送引用同一组画像证据。
- [ ] 换成“基础不足型学生”测试时，系统产生不同方案。
- [ ] 32 步完整课程路径可以保留；视频只需聚焦其中一个任务组。

### 7.3 统一评估证据

- [ ] 第一次不完整解释被拒绝。
- [ ] 完整费曼解释和迁移任务产生一条正式通过记录。
- [ ] 该记录同步驱动任务、画像、卡片和推送。
- [ ] 永久卡片审核能够识别该记录。
- [ ] 重复提交不会产生互相冲突的多条掌握事实。

### 7.4 资源真实性与安全

- [ ] 题目答案与选项、解析一致。
- [ ] 代码经过实际执行，输出与讲解一致。
- [ ] 动画、图示和文本使用同一组示例数据。
- [ ] 学术结论可追溯到课程资料或可靠来源。
- [ ] 警告或未验证内容不能显示为审核完成。
- [ ] 错误资源修正后可以重新校验并通过。

### 7.5 评委反方测试

- [ ] 后测更换业务域、类名和任务类型，不是原题换数字。
- [ ] 至少一项后测结论由真实 Java 编译运行验证。
- [ ] 用第二名“Java 基础确实薄弱”的对照学生重跑，系统不得生成同一路径。
- [ ] 人为纠正一次画像，确认旧判断不再驱动后续输出。
- [ ] 给 Agent 一份与课程讲义冲突的材料，确认来源冲突会暴露而非静默合并。
- [ ] 关闭旁白重看录屏，评委仍能从页面判断每一项结论。
- [ ] 断网、模型失败或资源校验失败时，页面不伪装成成功。
- [ ] 辅导 Agent 的输出不会作为“已经掌握”的直接证据。
- [ ] 故意让语义评分与代码执行冲突，确认确定性失败能够阻断通过。
- [ ] 用户可以查看、纠正、删除心理与认知风格字段，并阻止其继续参与推荐。
- [ ] 新会话或刷新页面后，证据、画像和路径状态仍一致，而不是仅存在前端内存。

### 7.6 页面可见性

- [ ] 卡片对话中的有效解释能实时写入右侧卡片。
- [ ] 页面无需刷新即可看到卡片、画像和路径状态变化。
- [ ] 不向用户展示内部 JSON。
- [ ] 当前任务、当前卡片和对话上下文保持一致。
- [ ] 生成过程有进度，失败时有可理解的原因和重试入口。

### 7.7 成熟仓库可信度

- [ ] 页面显示覆盖周期、学习事件数量、证据时间线和原始记录入口。
- [ ] 每个长期状态都能回到历史会话、评估或卡片证据。
- [ ] 至少展示一次画像判断被后续证据修正，而不只展示单向增长。
- [ ] 至少展示一次隔日或跨会话复测；没有真实复测时明确标为待验证。
- [ ] 不用卡片数量、节点数量代替学习效果。
- [ ] 旁白不主动解释素材属性，也不夸大成未经验证的真实长期提升。

## 9. 最终判断

这条视频不需要证明 AXIOM Space 已经理解一个人的全部心理，也不能证明一次干预具有长期教育效果。

它需要严谨证明的是：

```text
系统没有把“跟不上”草率归因为能力差；
它从对话中提出了一个可被反驳的学习机制假设；
这个假设实质性改变了学习计划和讲解方式；
学生在统一标准下从失败变为能够解释、反例、验证和迁移；
系统把这次变化转化成了下一轮学习决策。
```

当这五件事都能由真实页面连续证明时，A3 的“个性化学习闭环”才算在这个黄金案例中成立。
